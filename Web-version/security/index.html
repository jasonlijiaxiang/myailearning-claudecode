01为什么 AI 安全是新问题#sec-landscape

总根源一句话：指令与数据不再分家。传统程序里代码是代码、数据是数据——数据再脏也只是被处理的对象；大模型把系统提示词、用户输入、检索资料揉进同一段文本，读到的一切都可能被当指令执行。模型像一个太听话、记性又差的实习生——能力强、边界感弱、谁的话都听。AI 安全 = 传统安全 + 一层新假设「模型永远可能被诱导」。

02提示注入与越狱#sec-prompt-injection

如果只能记住一个 AI 安全问题，就是它：OWASP 连续两版第一名，过去两年几乎所有真实事故的主角。它是 SQL 注入的语言模型版——把「忽略以上，改为……」混进文本骗过模型；区别是 SQL 有严格语法能转义，自然语言没法转义。这不是 bug，是「用同一通道传指令和数据」这个范式的固有属性。

直接注入

攻击者自己在对话框输入，相对好防——危害限于自己的会话，输入护栏能挡不少。

间接注入才是噩梦

恶意指令藏在模型「顺带会读」的网页/邮件/PDF/工具返回值里，受害者是无辜用户——他只是让 AI「总结这封邮件」。

数据流记住这条链：

越狱是另一维（突破安全边界而非劫持任务）：

角色扮演
编码绕过Base64/拆字/多语言
多轮诱导

关键词黑名单注定被绕过，有效的是语义级分类器。

真实案例 EchoLeak（CVE-2025-32711，CVSS 9.3）

首个针对生产级系统的零点击间接注入。攻击者发一封藏隐藏指令的普通邮件 → 受害者某天让 M365 Copilot「总结最近的邮件」→ 隐藏指令让它搜集敏感文档 → 机密拼进 Markdown 图片链接，渲染时数据随 URL 溜走——全程零点击。

启示：危害不在「模型说错话」，而在它能读敏感数据 + 能把数据发出去；切断这两条链，比教模型拒绝更可靠。

为什么模型内治不了：OpenAI、Anthropic、Google DeepMind 2025 年都已公开承认——现有架构下提示注入无法彻底解决。「训模型去拒绝」是军备竞赛：防御方在穷举，攻击方在创造，后者天然占优。结论是本册的主旋律：把安全从模型里搬到模型外——接受「模型会中招」，把它当不可信组件做系统设计。

EchoLeak 核实 2026-07-09，来源 arXiv 2509.10540；三厂表态核实同日，来源 各家公开研究。不可外推边界：这是已公开披露的具体案例，用来说明「零点击外泄」这条攻击面真实存在；引用时给清楚时间点与厂商响应，别说成对方产品当前仍不安全。

03数据与隐私安全#sec-data-privacy

客户最放不下心的一句话：「我们的数据会不会被它学走、漏出去？」本节横跨 OWASP 四条风险，本质都是数据在管线里的进（投毒）、存（隔离）、出（泄露）三个环节。

出：两类泄露

类型漏什么对策
敏感信息泄露
LLM02，2025 升到第 2		PII、密钥、内部文档，经输出 / 日志 / 错误信息漏出真密钥放模型够不到的地方（网关、密钥管理服务）；输出侧加敏感信息过滤
系统提示词泄露
LLM07，新增		被套问出规则，甚至误写进去的 API key系统提示词按「一定会泄露」来写

模型像嘴不严的管家——问题不在钥匙，在「谁问都答」。

「模型会记住训练数据吗」要分场景答

用公开模型做推理通常不会；拿客户 PII 微调再对外提供才有记忆化与成员推断风险——脱敏、去重、隔离、高敏用差分隐私。所以 RAG 往往比微调更安全：数据留在库里、带权限、可溯源、可删除，不烧进权重。

进：投毒三时机，门槛递降、越贴近应用方越可控——

预训练期规模大难追溯，模型厂担
微调期触发词后门，谁供数据谁担
检索期往知识库灌恶意文档——不改模型只改它「读到什么」，门槛最低也最该防

RAG 侧三道闸当场能落：数据准入、来源可信、写入鉴权。存：向量库是新入口（LLM08）：检索投毒、跨租户泄露（A 的问题命中 B 的文档）、嵌入反演——向量化 ≠ 匿名化。对策：带 ACL 的权限过滤检索 + 多租户隔离 + 入库审校。 RAG 侧工程细节在 RAG 网页版。

04供应链与模型来源#sec-supply-chain

AI 供应链四环节：模型、数据、依赖、插件——比传统软件多前两环，都来自外部、都难审查。像自家种的菜 vs 进口食材：不查产地、不看有没有被掺东西，出事只是时间问题。入口关是最便宜的防线：在引入那一刻拦，远比运行时补救省。

模型即代码

很多权重用 pickle 格式存储，加载时会执行其中的代码——一个被做过手脚的模型文件，load 的瞬间就能跑任意命令；后门模型「平时乖、遇暗号发作」，常规测试完全看不出。

对策：优先 safetensors 等不执行代码的格式、沙箱加载、模型扫描。

「能跑」不等于「可信」——可信靠证据：

模型签名哈希校验没被篡改
Model Card来源/数据/许可/局限，三无模型直接扣分
AI-BOM物料清单——哪天某模型爆雷，一秒查到「我用没用到、用在哪」
许可合规商用场景法务过原文

准入门禁查两样：AI-BOM 是清单，四层工件是证据链——

只签容器不签模型和数据，等于安检只查行李不查人；组合发布把全部组件一起固定，任一撤回都能定位受影响部署。MCP 工具生态的供应链面（工具描述投毒）在下一节与 MCP 网页版。

05Agent 与工具安全#sec-agentic

Agent 是风险放大器：自主 × 工具 × 记忆三个乘数，注入后果从「说错话」升级到「做错事」。第 1 节那个太听话的实习生，现在手里多了公司的钥匙、邮箱和支付权限。 Agent 安全的核心是「控权」不是「防话」——既然拦不住它被骗，就限制它被骗之后能干什么。

三种「过度」，任一种都放大后果

过多功能

给了用不到的工具。

过多权限

工具权限超所需。

过多自主

高危动作不经审批。

爆炸半径 = 一次被攻破最多波及多大范围：只读 Agent 半径小，能删库 / 转账 / 群发的半径巨大。对策落地三件套：scoped 短期身份 + 每动作授权 + 不可逆动作人在环上（HITL）。

两个新攻击面

攻击面怎么打怎么防
工具描述投毒Agent 靠工具的自然语言描述决定怎么用——2025–2026 已有把恶意指令藏进 MCP 工具描述、接管编码 Agent 的真实案例。还有混淆代理：Agent 拿高权限身份替低权限攻击者办事工具描述当不可信输入审；身份不下传
记忆投毒诱导写进长期记忆，之后每个会话都被污染，已有跨会话存活案例不可信内容进记忆前过滤、任务间清临时上下文

身份体系要三条链分开

链答什么长什么样
身份链你是谁每跳都能回答现在到底是谁
授权链你能干什么授权书只写查询余额，不含转账
凭据链凭什么信短期签发、轮换撤销、审计关联

最常见翻车：「用户同意过」被当成无限委托，Agent 拿宽 scope 长效 token 到处用。正解是每个资源服务器独立核验，下游凭据绝不回传给模型。

OWASP Agentic 威胁地图（2026）按能力分五类：目标劫持 / 记忆 / 规划与自主 / 工具使用 / 部署运营——缓解主线就是身份隔离 + 每动作授权 + HITL 提权 + 强可观测。编排侧视角在 Agent 网页版。

06防护工程：护栏 · 模式 · 红队#sec-defense

「模型外兜底」具体化成三样：

护栏输入/输出/动作检查层
架构级设计模式用系统结构从根上限制危害
红队主动攻击自己持续验证

纵深防御四道闸：输入闸（注入检测/PII 过滤）→ 动作闸（最小权限/参数校验/HITL——Agent 时代最关键）→ 输出闸（敏感过滤/输出当不可信数据，别直进 SQL/HTML/shell）+ 监控闸全程兜底。单点护栏必被绕过。

护栏工具（2026 格局）形态特点 / 适用
Llama Guard开源输入/输出安全分类器，常作第一道语义闸
NeMo Guardrails开源（NVIDIA）Colang 定义对话/动作策略，<100ms，复杂对话边界
Guardrails AI开源50+ validator，强在结构化输出校验
Lakera Guard商业实时 AI 防火墙，单 API 查注入/越狱/PII，接入快
Bedrock GuardrailsAWS 托管6 类策略：内容/拒绝主题/敏感信息 + 上下文接地
Azure AI Content SafetyAzure 托管内容审核 + Prompt Shields 防注入/越狱

怎么选：多数生产系统组合 2–3 个——语义注入分类器 + 敏感信息过滤 + 按需的结构化校验，别指望一个工具包打天下。架构级防注入六类模式（研究综述归纳）：Action-Selector （只能从预定义动作里选）、Plan-Then-Execute（执行中读到的不可信内容不能改计划）、 Dual LLM（分权）、Code-Then-Execute（生成受限程序再安全执行）、Context-Minimization、 Map-Reduce（拆块隔离）——共同思路：不训模型拒绝，用结构限制「不可信内容能影响什么」。

双 LLM / CaMeL 深潜（目前评估最充分的防御）

在 AgentDojo 上以可证明的安全解决约 77% 任务——这是「有效但不够」的定量证据，所以还要纵深。

红队把防线常态化验证

PyRIT（微软开源）自动生成攻击、多轮诱导、给攻破率报告，按 ATLAS 选题；接进 CI，每次改 prompt / 换模型都跑，打穿的用例修复后固化成回归——攻破率指标汇入 Evaluation 网页版的安全维度验收。

CaMeL 77% 核实 2026-07-09，来源 arXiv 2503.18813；六类模式来源 arXiv 2506.08837。不可外推边界：77% 是论文基准环境下、对已知攻击集合的防护成功率，不是生产承诺，对未见手法没有保证——所以它在本册被放在纵深防御的一层，不是终点。

07治理与合规框架#sec-governance

技术防护再好，大客户还会问：「你们符合哪些框架？出了事怎么向监管交代？」治理拼图四块，位置摆清就很专业：NIST AI RMF 给方法论、EU AI Act 给法律底线、ISO/IEC 42001 给可认证的体系、MITRE ATLAS 给威胁情报。

三套框架各管一段

框架是什么用来干什么
NIST AI RMF四大功能：Govern（建治理机制）/ Map（识别风险）/ Measure（量化评估）/ Manage（处置监控）。GenAI Profile（AI 600-1，2024）另列 12 类风险和 200+ 条可操作建议自愿性框架但被当事实标准——像 AI 版的质量管理循环，给自评骨架
EU AI Act法律底线，见下方时间线涉欧盟业务必答
ISO/IEC 42001第一个 AI 管理体系国际标准（类似 ISO 27001 之于信息安全）对外招标的通行证

EU AI Act 硬时间点

企业落地三步：盘清 AI 资产（不知道公司里有多少模型和 Agent 就无从谈治理）→ 定策略设门禁（模型准入/数据分级/上线红队/高危审批做成 CI 门禁）→ AI-SPM 持续监控（Defender for Cloud / Security Command Center / GuardDuty 组合）。治理是持续过程不是一纸报告——文档是结果，不是目的。

EU AI Act 时间线核实 2026-07-09，来源 欧盟官方。不可外推边界：时间线是官方节点，适用与否取决于业务是否触达欧盟以及风险分级；日期本身不构成对某个具体产品的合规判断，以法务核定为准。

08中国监管合规#sec-china

在中国落地 AI，合规是上线前的门票，不是上线后的补课：备案全流程 5–8 个月、内容标识 2025-09 起是法定义务、安全国标 2025-11 已实施——三件事都已进入执法期。截至 2026-06-30 累计 988 款服务完成备案、598 款应用完成登记——说明这是常态动作，不是拦路虎。

分诊三问定路线

最常见误区是把「登记」当「备案」自己吓自己：

你是哪种走哪条路要多久
自研 / 微调模型向公众提供服务服务备案算法备案约 2 个月 + 大模型备案 3–6 个月，全流程 5–8 个月（北京 8–10 个月），无加急通道——排进产品计划
调用已备案模型做应用应用 / 功能登记地方网信办，周级
纯内部自用暂行办法不适用数据安全与个信义务照常

备案材料三件硬货（按 GB/T 45654 口径）：语料标注规则、拦截关键词库（覆盖 17 类风险，建议 ≥1 万词）、评估测试题集。罚则最高 10 万 + 暂停更新——真正的代价是「不能上线」。

两个已生效的硬约束

内容标识

《AI 生成合成内容标识办法》2025-09-01 施行 + 强制国标 GB 45438-2025。显式标识给人看（生成入口就近打标）、隐式标识给机器看（元数据 / 水印，导出链路别洗掉）。已有 App 因违规被约谈下架——代价不是罚款数字，是产品下线。

安全国标 GB/T 45654-2025

2025-11-01 实施：训练数据 / 模型 / 安全措施三方面、五大类 31 小类风险，备案安全评估按它出。

法规栈

暂行办法2023-08，总纲
算法推荐规定2022-03，算法备案依据

风向（2026-07 口径，易过期）：AI 综合性立法已列入国务院 2026 立法计划。

数据出境先查豁免再谈通道：《促进和规范数据跨境流动规定》六类豁免 + 自贸区负面清单；豁免不了走三通道——安全评估（重要数据/大规模个信必走）、标准合同（中等规模常用）、保护认证（集团内跨境）。客户问「数据能不能出境」先问三件事：是不是重要数据、多少人的个信、在不在自贸区清单内。云平台（火山引擎备案材料支持、阿里云百炼护栏默认挂载与备案指引）帮到材料与护栏，法定「服务提供者」义务不转移——审核、标识、投诉、备案主体一样不能少。

安全验收：攻击可复现、控制可阻断、事件可止损

安全方案不看「上了几个产品」，看四段能不能闭环：

段验什么证据
边界 + 预防资产 / 身份 / 数据与能力边界清楚，高风险路径有多层控制威胁模型 + 责任人；策略、最小权限、沙箱
检测异常与攻击可观测、可按切片看trace、审计、告警（接 AI-Ops）
响应能力可快速撤销token / 工具 / 网络出口三个急停开关都演练过
恢复 + 治理版本可回退、数据可修复；剩余风险有签批回滚 + 删除 + 回归样本；例外有期限与复核

终极判断：从一条攻击路径能一路追到控制、检测、止损、恢复、责任人与剩余风险——「模型拒答了」不等于纵深防御成立。

10现场高频追问#qa

只收会改变某个决定的问题——会改变技术方案、风险判断、验收方法、对客沟通或下一步行动。

每题标出它加入本题库的日期：2026-07-09 来自讲义首版对练页， 2026-07-17 来自中国合规增章。

大模型不就是个聊天框，能有多大安全风险？2026-07-09

只要它能读外部内容、或能触发动作，风险就是真的。纯聊天最多被套话；一旦接了检索、工具、数据库，一段精心构造的文字就可能让它泄密或误操作。

「模型安全 ≠ 应用安全」：测试保证的是模型对齐，注入、泄露、过度自主大多发生在「模型+提示词+工具+数据」拼起来的应用层。

下一步　用自查器过一遍能力清单，圈出爆炸半径最大的那条链路，它就是第一个上护栏的地方。

依据　三层攻击面

提示注入到底能不能彻底解决？2026-07-09

坦率说，现有架构下不能——OpenAI、Anthropic、谷歌都公开承认。但能把风险压到可接受：模型外护栏、最小权限、动作审批，让注入成功也偷不走数据、动不了要害。

「训模型拒绝」是防御方穷举、攻击方创造的军备竞赛——防线要建在结构上不建在模型上，这是全册的主旋律。

下一步　EchoLeak 的启示当口径：切断「能读敏感数据+能外发」两条链，比教模型拒绝更可靠。

依据　三大厂共识

我们的数据会不会被模型学走、被别人问出来？2026-07-09

用 RAG 不会——数据留在你的库里、带权限、可随时删，不进模型参数。只有拿敏感数据去微调再对外提供才有这个风险，那种场景用脱敏、隔离甚至差分隐私来兜。

「RAG 优于微调」是数据安全侧最有力的一张牌；另一半提醒是向量化 ≠ 匿名化——嵌入能被部分反推，该脱敏还得脱敏。

下一步　多租户场景补一句带 ACL 的检索和租户隔离——否则 A 的问题可能命中 B 的文档。

依据　数据与隐私

从开源社区下模型很方便，有什么风险？2026-07-09

两个新风险：模型文件本身可能带恶意代码（pickle 加载即执行），或被植入后门（测试期看不出、遇暗号发作）。要过来源验证和扫描这道关，别直接拉到生产。

「能跑」不等于「可信」——可信靠签名/哈希、model card、安全格式三样证据，不靠试用。

下一步　把准入 checklist（来源/格式/文档/登记 AI-BOM）做成评审门禁——入口关是最便宜的防线。

依据　供应链

给 Agent 放权自动干活，出了事谁负责、怎么控？2026-07-09

责任在部署方，所以控制权要抓在系统手里：最小权限、短期身份、不可逆动作强制人工审批——即使模型被骗，能造成的最大损失也被提前框死。

Agent 安全的重点不是防它被骗，而是限制被骗后能做什么——爆炸半径是设计出来的，不是祈祷出来的。

下一步　给每个工具做「工具—权限—是否需审批」表，一句话说清这个 Agent 的爆炸半径有多大。

依据　控权而非防话

买一个 AI 护栏产品，是不是就安全了？2026-07-09

护栏是必要的一道闸但不是全部：输入+动作+输出+监控四层叠加，再加架构级最小权限。单靠一个护栏，换个绕法就被穿了。

防护有效性要量化举证：红队按 ATLAS 打一遍给攻破率，修复后固化回归——「被持续验证」比「买了产品」有说服力得多。

下一步　按爆炸半径分级投防护：低风险只读应用护栏+最小权限就够；能碰敏感数据又能动作的链路才上双 LLM 级结构。

依据　纵深防御

我们要过 EU AI Act，具体要做什么、什么时候？2026-07-09

先看业务是否触达欧盟。触达的话：GPAI 义务 2025-08 已生效，执法和罚款 2026-08 启动（最高全球营收 7%），存量模型 2027-08 前整改完——现在就该倒排。

治理框架组合用：NIST RMF 给自评骨架、ISO 42001 给可认证体系、 EU AI Act 是法律底线——对外招标常要求前两个。

下一步　治理落地从盘清 AI 资产开始——不知道有多少模型和 Agent，就无从谈合规。

依据　EU AI Act 时间线

我们私有化部署、只给内部员工用，要备案吗？2026-07-17

暂行办法管「向境内公众提供服务」——纯内部使用不适用备案，但数据安全与个信保护义务照常；一旦对外（哪怕小程序），就要重新分诊。

分诊三问是中国合规的入口：自研对公众走服务备案（5–8 个月）、调已备案模型走登记（周级）、纯内部不适用——最常见误区是把登记当备案吓自己。

下一步　对公众的产品把备案排进产品计划——没有官方加急通道，988 款已备案说明这是常态动作。

依据　分诊三问

不打 AI 标识会怎样？2026-07-17

2025-09 起是法定义务，配套强制国标；已有 App 因标识违规被约谈、下架——代价是产品下线与渠道信任，不只是罚款数字。

显式标识给人看（生成入口就近打标）、隐式标识给机器看（元数据/ 水印，导出链路别洗掉）——两层都要落。

下一步　把标识做成生成服务的默认输出（服务端注入），别靠前端各业务自觉补——审计时查得出、改得动。
