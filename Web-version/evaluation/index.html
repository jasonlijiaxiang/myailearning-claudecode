01为什么评估这么难#eval-why-hard

大模型输出是概率采样：同一问题两次措辞不同，且多数任务没有唯一「正确答案」—— assert 式单元测试直接失效。LLM 评估天生是「改作文」不是「对答案」：量的不是对不对，而是好不好 + 稳不稳。

「感觉不错」不可规模化：手感只能覆盖个位数场景，真实问题分布是长尾的——「demo 惊艳、上线客诉」是行业常态，因为 demo 挑的是顺手问题。最常见的事故是隐性回归：改一版 prompt，盯着的问题答好了，原本正常的三个场景悄悄答坏——没有回归评估根本发现不了。有没有可重复、可比较、可回归的评估，是玩具与产品的分水岭。

三层分工：高考、面试、试用期

基准测试量模型通用能力（高考），应用评估量你的场景加你的数据（面试），生产监控量真实流量下的持续质量（试用期）——三层各管一段、互相不能替代。某客户按榜单选了第一名模型，POC 一跑发现它在自家行业术语上不如第二名——榜单是高考分，客户场景是岗位面试。口径：榜单筛入围、应用评估定选型、生产监控保长期。

评估集是真正的护城河

模型会换、prompt 会改、框架会过时；沉淀下来的是「你的场景里什么算好」加一套能验证它的题——评估集是复用率最高的工程资产。换模型时，有评估集的团队一天内跑完回归给出数据化结论；没有的团队人肉重试拖两周还说不清。边界声明：本册管「质量」，延迟/吞吐归 LLM-Inference 网页版——快不快找推理，好不好找评估。

02基准测试全景#eval-benchmarks

基准＝统考：固定题集 + 统一判分，价值在横向可比，局限也在此——题是公开的、通用的，既可能被押题，也不代表你的场景。「榜单筛入围，实测定输赢。」

GPQA Diamond博士级理科推理头部 81~94%，渐近饱和但仍有区分度

2026 年中格局一句话：Claude Opus 4.6 领跑 SWE-bench/Terminal-Bench/OSWorld， Gemini 3.1 Pro 领跑 ARC-AGI-2 等，GPT-5.2 领跑 τ²-bench——没有全能冠军，按任务选对口基准。能力比拼已从「答题」转向「干活」。

03判分四法#eval-methods

四种判分方式各管一段，设计顺序是铁律：能用代码判的绝不上判官，能用判官的绝不堆人工——人工定标准、校判官、兜底线，用 5% 的人工判断撬动 95% 的自动化。

判法成本速度可靠性适用
人工评估最高最慢金标准定标准、校准判官、高风险兜底
代码判分最低最快客观题上最可靠格式 / 精确答案 / 代码 / 结构化输出
参考指标（BLEU 等）低快LLM 时代失灵仅作辅助信号
LLM 判官中（推理费）快校准后可用，有偏差主观质量的规模化打分

BLEU/ROUGE 靠词面重叠打分，来自机器翻译时代——LLM 的答案常与参考措辞完全不同却语义正确，词面重叠会冤枉好答案、放过坏答案。客户的评估方案若还拿 BLEU 当主指标，这本身就是一个咨询切入点。设计评估先想「能不能把题出成可代码判的形式」（选择题化、结构化输出）——大部分企业任务能用「结构化输出 + 代码判」覆盖七成判分需求，评估成本立刻降一个量级。

判分方案设计器

按第 3 章设计顺序（先代码判 → 再判官 → 人工定标兜底）给一个任务配判法组合。判据全部来自本页各节；一个评估集里三种判法混用是常态，不是设计失误。

04LLM 判官深潜#eval-judge

判官把「作文题阅卷」自动化了，是应用评估能规模化的关键一环——也是最容易被用错的一个：裁判自己就是 LLM，有立场、有口味、会被顺序和长度带偏。

三种判法与三大偏差

LLM 更擅长判别式任务：成对比较（两个输出选更好，最稳）和评分细则（把「好」拆成可核对条款逐条判，最可控）远比裸的单点打分稳定——问「这碗面打几分」难答稳，问「这两碗哪碗好吃」谁都答得准。选题型：回归监控用细则、版本对比用成对比较、绝对分数只作趋势参考。

偏差表现实验证据缓解手段
位置偏差偏向特定顺序的答案换序可致判决漂移 >10%换序双跑（A-B 与 B-A），不一致判平局或送人工
冗长偏差偏爱更长的答案无新信息的加长版被偏好 >90%细则加简洁性条款，要求引用证据
自我偏好偏爱像自己写的输出与生成困惑度显著相关判官与被评模型错开厂家；多判官交叉
口径漂移绝对分数随批次漂移单点打分重测不稳定改成对/细则判；锚定样本定期复测

校准四步与使用边界

校准流程：人工标一批金标准判决（几十到几百条）→ 最强模型当判官跑同一批 → 算一致率，达标（常见门槛 85%~90%，视任务）才上量 → 换便宜判官降本后重测。像新阅卷老师先试批 50 份卷子和阅卷组长对口径。2026-03 开源的 Judge Reliability Harness 压测结论：没有判官全面可靠——可靠性是逐任务的属性，不是模型的属性。校准时优先改细则措辞、不急着换模型：多数不一致源于标准含糊。

边界：判官增强而非替代人工——高风险场景人工必须在环；实务分层是「判官全量打分 → 低分与边界样本送人工复核 → 人工判决回流为新校准数据」。评审任何判官方案用判官三问： ①和人工对齐过吗？②换序复核了吗？③高风险有人工兜底吗？

偏差证据来源 arXiv 2306.05685 / 2410.21819 / 2410.02736，校准与可靠性 Judge Reliability Harness（2026-03），核实 2026-07-09。不可外推边界：一致率门槛是常见做法不是通用标准，随任务而变；压测的结论也正是「可靠性是逐任务的属性」——换任务必须重新校准，不能沿用上一个场景的判官结论。

05自建评估#eval-build

你自己的评估集从哪来？核心方法只有一个——错误分析：评估项不是从论文里抄的，是从你系统的真实错误里长出来的。黄金集起步 20~50 题足矣，贵在真实，不在数量。

黄金集与错误分析

构成配比参考：60% 高频典型题 + 25% 边界与刁钻题 + 15% 必须完美的高价值题（涉及价格、合规话术等），比例按业务风险调。错误分析的做法：先逐条看几十条真实输出，每条错误用短语记下来，归并成类别，每个高频类别对应一个评估项——像给 AI 建错题本。实例：某企业问答系统看 50 条日志归出三类高频错误（引用过期文档 38%、过度拒答 24%、术语用错 18%），评估集立刻有了三个专项维度，比拍脑袋写的「准确性/流畅性」有用得多。

冷启动三条路与指标设计

没有标注数据的三条起步路：专家手写（半天 30 题，质量最高）、日志采样（分布最真）、合成造题（量最大，但模型爱出「文档原句就能答」的软柿子题——造题 prompt 里要强制出对比题、多跳题、无答案题，且必须人工抽审后才能入集）。三路并进，两周从 0 到 100 题。

指标设计两条：先定成功标准再倒推指标（「95% 事实正确、100% 不泄露内部价格」好过「回答质量高」）；能二分类就别打分——每题「通过/不通过」判得稳、算得清、和验收线好挂钩，五分制里 3.8 和 4.1 的差别没人说得清。指标不超过 5 个，每个都要能翻译成一句业务话（事实错误率↔客诉风险；拒答率↔转人工成本）。

进 CI、持续生长

把评估当测试跑：prompt 改动、模型升级、检索配置变更都触发评估集回归，通过率低于门槛就挡住上线——「只改了一个词」和「只改了一行代码」一样都是变更。评估集要持续生长：线上坏例回流（同一个坑不踩第二次），像免疫系统越打越强。两个维护点：防泄漏（评估题别混进微调训练数据）、防过拟合（留 10–20% 保留集不参与日常调优，定期补新题）。

06场景验收#eval-scenarios

三个场景共用一个骨架：黄金集 + 分层判分 + 回归门禁——只是指标各有侧重。这一节收编了全库的评估线。

场景核心指标判法最大的坑
RAG忠实度 / 上下文精确率 / 召回率检索代码判 + 生成判官判检索生成混着评，定不了责
Agent任务通过率 / 工具正确率 / pass^k分层：代码判 + 判官 + 统计只评 happy path，漏异常处理
微调目标提升 + 通用回归 + 行为稳定冻结回归集 + A/B只看 loss，漏灾难性遗忘

RAG：检索与生成分开评

Ragas 三角

开卷考试三件事：翻到对的页、别抄不相干段落、答案只依据书上写的。

指标量什么低了说明去修哪
忠实度答案是否只基于检索材料生成端在编造改 prompt / 换模型
上下文精确率检回来的相关吗检回来一堆噪声重排序 / 过滤
上下文召回率该检的检全了吗检索端没找到修 chunking / embedding / 混合检索

评估顺序：先把检索评到位（recall@k、MRR，代码判、便宜），再评端到端（判官判忠实度，贵）。经验规律：RAG 生产事故的大头出在检索端——先修检索是性价比最高的路径。管线机制见 RAG 网页版。

Agent：结果与轨迹双轨，五层展开

结果评估管验收（办成没有），轨迹评估管定位（该调的工具调了吗、有没有绕路死循环）——像考驾照：只看到没到终点，危险驾驶也算通过。再往下拆五层：Outcome → Trajectory → Tools → Safety → Efficiency。「危险的成功」是五层存在的理由：任务完成了但中途越权改了库——只评结果会把它放行。多次运行看 pass^k（k 次全过才算稳）；专门造「工具报错、返回为空、用户中途改主意」的扰动用例，别只评顺风局。Agent 侧的护栏与验收线见 Agent 网页版。

微调与安全维度

微调验收别只看 loss：目标能力提升了吗、通用能力掉了吗（灾难性遗忘）、格式与安全行为稳吗、线上 A/B 赢了吗——训前冻结一套与训练数据零重叠的通用回归集，训后先过回归再上线。安全维度进验收清单：有害内容率、注入抵抗、泄露、幻觉率（无依据陈述比例）——同样可以出题、判分、算通过率。护栏与评估的分工：护栏是在线安检门挡个案，评估是离线体检看水位——互补不互替，护栏拦得准不准也要靠评估度量。

07工具链与闭环#eval-tooling

开发期用开源框架（快、进 CI），生产期用平台（观测、追溯）——两套并行，没有单一工具通吃两端。先框架后平台：评估习惯没建立就买平台，买回来也是闲置。

在线评估

生产流量 1~10% 采样跑判官、指标进看板、超阈值告警；新 prompt 先灰度 5% A/B 一周再全量——离线管变更，在线管日常。

评估资产要可迁移

OpenAI 平台内置 Evals 已官宣 2026-10-31 起只读、11-30 关停——教训：评估资产（题集、细则、历史结果）的寿命远长于任何平台。落地三件事：黄金集进 git 管版本、判官 prompt 与细则当代码管、结果定期导出归档。口诀：「题在手，平台任你换；题在云，平台绑你走。」三大云均有托管评估服务（Bedrock 评估作业 / AI Foundry 评估器 / Vertex 评估服务）——客户在哪朵云就从哪家起步，黄金集保持平台无关。

四道发布门与质量飞轮

把评测结果变成发布决策的四道门：契约门（schema/工具/权限：零关键失败，挂了阻断构建）→ 质量门(数据集+评分器：基线与分片达标) → 风险门（安全/红队/隐私：高风险清零或例外签批）→ 生产门（金丝雀 SLI/业务结果：预算内持续达标）。两条铁规矩：不能用下游平均分覆盖上游硬失败；数据集与评分器的版本是门禁的一部分——换了判官必须重新校准、重建可比基线。

质量飞轮：线上坏例 → 错误分析归类 → 进评估集 → 针对性修 → 过回归门禁 → 上线 → 监控采样 → 新坏例……每转一圈评估集厚一分。给客户的运营建议：每周半天「评估例会」一起看坏例——整个体系里回报率最高的投入。上线不是项目结束，而是飞轮起点。

工具与云服务格局核实 2026-07-09，来源 2026 对比评测与 OpenAI 官方文档。不可外推边界：平台清单是当期快照，能力与存续都会变（本节的关停案例就是），它不构成对任何一家的能力背书；可迁移的是你的题集与细则，不是平台。

客户常说一句话纠正
「榜单第一，选它准没错」榜单是统考，您的场景是面试——筛入围可以，定选型要实测
「评估就是跑个分」跑分只是判分；建题集、定标准、进流程，才是评估体系
「先上线，有问题再说」没有评估的上线，出了问题连「变差了」都证明不了
「让 AI 自己评自己就行」判官要人工校准、要防偏差，「自家评自家」更是大忌
「必须 100% 正确才验收」LLM 到不了 100%——定验收线 + 高风险人工兜底
「评估做一次就够了」模型在变、数据在漂：评估集是要一直养的活资产

现场高频追问

只收会改变某个决定的问题——会改变技术方案、风险判断、验收方法、对客沟通或下一步行动。

每题标出它加入本题库的日期：2026-07-09 来自讲义首版对练页， 2026-07-12 来自五层栈与发布门深潜增补。

你们模型在榜单上分数很高，是不是拿来就好用？2026-07-09

榜单是通用统考，证明基础能力；你的场景是岗位面试，得拿你的数据实测。先建 20-50 题黄金集跑 POC，用数据说话——这也是各大模型厂官方指南推荐的第一步。

三层分工（基准/应用评估/生产监控）是全册的骨架——这题答稳了，后面所有「分数 vs 实测」的争论都有了框架。

下一步　把客户场景对应的对口基准挑出来当入围参考，同时启动黄金集共建。

依据　三层分工 · 应答三步

让 AI 评 AI，靠谱吗？2026-07-09

校准过的判官在明确细则下与人工一致率能到九成上下，足够做规模化初筛；没校准的判官确实不可信。关键不是「用不用判官」，而是「有没有按流程校准」。

可靠性是逐任务属性不是模型属性（Judge Reliability Harness 结论）——「判官准不准」没有通用答案，只有「在这个任务上校准过没有」。

下一步　用判官三问评审现有方案：对齐过吗、换序复核了吗、高风险有人工兜底吗。

依据　判官深潜 · src-jrh

判官会不会偏向你们自家模型？2026-07-09

会，这就是自我偏好偏差，有实验证据（与生成困惑度显著相关）。正式评估要么用第三方模型当判官，要么多判官交叉，要么以人工金标准定期复核。

「裁判和运动员同门」是客户方案里最常见的坑——用哪家模型当判官，本身就是评估设计决策。

下一步　检查现有判官与被评模型是否同厂家，是就换或加交叉判官，并重测一致率。

依据　三大偏差

我们没数据、没标注团队，评估怎么起步？2026-07-09

三条路并行：业务专家半天写 30 题、试运行日志采样、大模型从文档合成题目再人工抽审。两周从 0 到 100 题很现实。

合成题有坑：模型爱出「原句就能答」的软柿子题——要强制出对比题、多跳题、无答案题（考拒答），且必须人工抽审。

下一步　黄金集按 60/25/15 配比（典型/边界/高价值）起步， POC 阶段陪客户一起建。

依据　自建评估

RAG 答错了，怎么知道是检索还是生成的锅？2026-07-09

分开评：先看检索指标（该命中的文档进没进前 k 条），命中了再看忠实度（答案是否只基于材料）。经验上大头是检索的锅。

「检索生成混着评，定不了责」是 RAG 验收最大的坑；三角指标把「谁错了」拆出来，修的优先级自然就有了。

下一步　验收集里放 2 道「文档里没答案」的题考拒答——最能暴露编造问题，演示也最震撼。

依据　RAG 三角 · RAG 网页版

Agent 这种多步任务怎么验收？2026-07-12

双轨加五层：结果通过率管「办成没有」，轨迹检查管「过程对不对」，再往下拆 Tools/Safety/Efficiency；多次运行看 pass^k。先共同定义 20 个代表性任务当验收集。

「危险的成功」（任务完成但中途越权）只评结果会放行——五层缺一，都可能误判；这也是和「只跑分」供应商拉开差距的地方。

下一步　验收集里专造扰动用例（工具报错/返回为空/用户改主意）——Agent 的价值恰恰体现在异常处理上。

依据　Agent 双轨五层 · Agent 网页版

微调完的模型，怎么证明没有「变笨」？2026-07-09

训前冻结一套与训练数据无重叠的通用回归集，训后必须先过回归再上线；交付报告里「新能力」和「老底子」两组数字都有。

只看 loss 会把「偏科」当「进步」——灾难性遗忘是微调最隐蔽的风险，回归门禁是唯一系统性的防线。

下一步　把「训完给回归报告」写进微调服务承诺——既专业，又给自己留质量凭据。

依据　微调验收

万一评估平台以后停服了怎么办？2026-07-09

好问题——OpenAI 的 Evals 平台今年就宣布退役了（10-31 只读、11-30 关停）。评估资产（题集/细则/结果）全部以开放格式自持，平台只是跑分器，随时可换。

评估资产的寿命远长于任何平台——「题在手，平台任你换；题在云，平台绑你走」。这也是写进客户方案的信任卖点。

下一步　黄金集进 git、细则当代码管、结果定期导出——三件事今天就能做。

依据　资产可迁移 · src-openai-evals

评估体系建起来要多久、多少钱？2026-07-09

最小可用两周：50 题黄金集 + 判官校准 + CI 接入，一名工程师加业务专家兼职即可。工具开源免费，主要成本是判官推理费和少量标注——量级远低于一次返工。

评估的投入产出常被高估门槛、低估回报：换模型从两周人肉试变一天回归，这一项就回本。

下一步　从第 5 节错误分析开始：看 20 条真实日志、归类、长出前三个评估项——今天就能动手。

依据　自建评估 · 工具链

有了护栏是不是就不用评估了？2026-07-09

护栏是在线安检门，挡个案；评估是离线体检，看整体水位和趋势。而且护栏本身拦得准不准，也要靠评估来度量——配合关系，不是替代。

这对分工在验收清单上的落点：安全维度（有害内容率/注入抵抗/幻觉率）照样出题、判分、算通过率——多数客户想不到，提出来就是专业加分。

下一步　验收模板固定留一节「安全指标」，用 promptfoo 红队插件批量生成注入用例。

依据　安全维度 · 四道门
