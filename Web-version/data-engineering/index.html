01数据就绪度是第一风险#de-what-why

库里五处证据都指向同一个坑：POC 三要素的数据坑、企业搜索「连接器是报价大头」、内容生成「产品库是唯一事实源」、坏答案「大半源于脏数据」、「解析质量决定切分质量」——五句话都是果，因在同一处：没人把数据搞就绪。本页就是那个「接住」的动作。对客口径：数据乱不是障碍，「假装不乱」才是。

数据就绪度四问（第一次客户拜访就要问完——答案直接决定排期与报价结构）： ①数据在哪（几个系统、能不能连）②什么格式（PDF 扫描件占比多少）③谁有权（ACL 能不能映射过来）④多久变一次（增量同步节奏）。管线总图四段各对应一个可收费工程件：解析管线（按文档类型分档）→ 清洗同步（连接器按源系统个数）→ 入库（向量库建设）→ 质量运营（运营包按月）。「为什么你们报价里有一块数据工程」的答法：别家不是没有，是藏在里面或做砸了才补——列成明细，每一分钱买了什么看得见，既是收入项也是防烂尾的保险。

数据就绪度评估器

四问的交互版：填客户情况，得到排期量级、报价结构与技术路线建议。判据全部来自本页各节，不新增事实。

02文档解析管线#de-parsing

解析是管线第一段：PDF/扫描件先变成带结构的文本，下游切分才有的放矢——上游错，下游救不回；解析错的表格进了库，检索永远找不回来。选型分水岭是表格保真与阅读顺序：纯文字 PDF 谁来解都差不多，表格、双栏、图纸、红头文件才拉开数量级差距。

方案画像（2026-07）适用
LlamaParseVLM 解析复杂版面、托管 API、agentic OCR<1000 页/天、非敏感文档、要省事
DoclingIBM Research 开源，AI 版面检测、保留语义层级自托管零按页成本，生态好
MinerUOpenDataLab 出品，CJK 复杂版面最强，出 MD+JSON中文文档 / 扫描件重的国内项目
Unstructured老牌解析管线，格式覆盖广多格式混杂的存量清洗
云文档智能Textract / Document Intelligence / Document AI企业已有云协议时的默认起点

基准口径（opendataloader-bench，200 份 PDF）：Docling 0.877 > marker 0.861 > MinerU 0.831——但这是英文基准，CJK 版面是 MinerU 的主场，中文场景自测是唯一可信口径。三个选型硬约束：敏感文档不出域 → 自托管（Docling/MinerU）；中文扫描件占比高 → MinerU 优先；按页计费 vs 自托管运维按月页量算交叉点，量大自托管划算。对客动作：先拿客户 20 份最难的文档（表格重/扫描件）跑对比测试——解析选型永远用客户自己的文档定，不用厂商 demo 定。解析产物交给 RAG 网页版按结构切分。

四强格局与基准核实 2026-07-11，来源 2026 解析横评。不可外推边界：解析基准多以英文版面为主，对中日韩版面的参考价值有限——本节的选型分水岭因此写成「拿客户自己最难的 20 份文档实测」，而不是照抄榜单。

03连接器与增量同步#de-pipeline

一个连接器 = 五件事：①认证（OAuth/服务账号）②拉取（API 限速与分页）③解析（格式转换）④ACL 映射（权限同步）⑤增量（同步模式）。报价口径：按「源系统 × 五件事」估工作量——SharePoint/Confluence 类成熟源有现成连接器（Airbyte 模式）天级接入；自研系统每个都是一个小项目，那才是大头。

增量同步三模式按新鲜度需求分级：轮询（实现最简单，新鲜度差——低频源够用）、webhook （准实时，要源系统支持回调 + 丢消息补偿）、CDC 变更日志（最可靠，只有数据库类源才有，工程最重）。真实问题是「哪些内容需要多新鲜」——制度文档天级足够，工单类才值得准实时，分级同步省一大笔。两件没人爱做的脏活决定客户第二个月还信不信系统：去重（同一文档的邮件附件/网盘/IM 转发五个副本——内容指纹保留权威源；不去重的后果是检索 Top-K 被同一文档霸屏；多数企业第一次实测重复率超过 20%）与失效（离职人员文档、废弃版本——源系统删除/归档联动下架；「AI 引用了三年前的报销标准」是信任事故）。同步链路按生产系统运维：断供、限速、权限失效都有告警与补偿重试，每个连接器有健康页——不是脚本跑一次就完。

04向量库选型#de-vectordb

选型第一问是「你已经有什么」：有 Postgres 用 pgvector（生产可用至约 50–100M 向量，ACID、免新系统）、有 ES 就地升级（原生混合检索最成熟）——新引一套系统的运维成本常超过它带来的性能差。五个锚点：pgvector 中小规模首选、Qdrant 过滤检索最强开源（p50 4ms）、Milvus/Zilliz 十亿级分布式、Weaviate/ES 原生混合、托管（Pinecone 等）零运维快速上线。

向量库是「按规模演进」的决策，不是「一步到位」的决策：起步（pgvector/ES 就地） → 增长（监控召回与延迟）→ 触顶（三个信号：HNSW 索引重建时间不可接受、P99 随库量线性恶化、过滤条件多导致召回骤降）→ 迁移（Qdrant/Milvus）。迁移纪律：嵌入模型不换则向量可搬，换模型必须全量重算（向量空间不兼容——所以嵌入选型要留余量、全量重建做成自动化管线）；双写灰度，新旧库并行跑评估集对比；别为「将来可能的十亿级」提前买复杂度。性能答法：各家都在毫秒级、量级内差距对多数应用无感——真正拉开差距的是过滤/混合检索能力和运维成本，选型看能力匹配不看跑分排名。国产化要求：开源自托管路线全可控（Milvus 是国内团队主导的开源项目）。多数客户永远到不了触顶那天——但方案书里有预案这件事本身，就是专业度。

选型锚点核实 2026-07-11，来源 2026 向量库对比。不可外推边界：对比表是通用场景下的能力排序，不含客户的过滤条件、并发与运维口径；文中的量级建议是常见生产区间，不是硬上限。

05数据质量与覆盖率#de-quality

把「垃圾进垃圾出」变成四个可度量指标：解析保真率（抽样人工比对，表格重点——异常回炉换解析路线）、重复率（内容指纹统计——异常修订去重规则）、新鲜度（各源最后同步时间 vs SLA——异常查连接器健康）、覆盖率（高频问题的「有答案率」——反向驱动文档补全）。全部自动化采集进月报：「数据很乱」变成「重复率 23%、覆盖率 71%」，改进就有了靶子。

坏答案回流：修数据，而不是改提示词。「AI 老答错是不是模型不行」的归因链：经验上八成问题在检索侧，检索问题的大半又在数据侧（解析错/文档缺/内容过期）——在线评估抓坏答案（AI-Ops 网页版）→ 归因（检索的锅、解析的锅还是缺文档）→ 修数据（补/换/下架）→ 评估集重跑验证。多数客户以为要「调模型」，归因数据会告诉他们该「修数据」。覆盖率仪表盘的售前故事：高频无答案问题清单 = 客户补文档的优先级表——把「AI 答不上」变成知识治理的入口；补写是客户团队的活，「补什么最值」由数据说话。运营节奏：日看新鲜度告警、周看坏答案归因、月出四指标报告——与 AI-Ops 运营包同节奏合并交付；停止运营的知识库半年就烂。

06标注与合成数据运营#de-labeling

边界先声明：合成数据配方（种子→扩展→judge 过滤）在 Fine-tuning 网页版——本节管运营与预算。 2026 共识：全人工标注已是少数场景，预算从「雇人标」转向「专家定标准 + 模型产量 + 人工守门」三笔钱：种子人工写（业务专家半天 30–50 条精品，质量标准也在这步定——最贵也最值）、扩量交模型（成本是人工的零头）、验收人工抽检（5–10%，不过线回炉调种子——守住「合成不等于放羊」）。

一份坏例，两条去向——分流先问「是不知道还是不听话」：「不知道」是知识缺 → 评估集线，修文档/RAG 更新，当天见效；「不听话」是行为偏 → 训练集线，攒够批次做微调。分流错了钱就白花——知识类坏例别硬塞进训练集。评估集线留 10–20% 保留集防应试。运营纪律：业务专家的时间是最稀缺资源，只用在三件事上——定标准（一次）、写种子（半天）、周审边界样本（两小时）——这是撬动整个数据飞轮的最小投入。合成数据「会不会越训越歪」：有纪律就可靠——真实种子打底、judge 过滤、抽检合格率是硬指标（低于七成说明种子或标准要回炉），纯合成才有坍缩风险。

07治理与权限衔接#de-governance

分工一句话：Security 说「要什么规则」，本页说「规则在管线哪一段执行」——安全评审问到数据层，答案全在三个执行点：①ACL 映射（连接器同步权限元数据、检索时实时过滤、无权限连存在性都不暴露）②采集点脱敏（PII 正则+NER 进库前处理、敏感字段分级——别等检索时再补救）③保留与删除（源系统删除联动下架、遗忘权可执行、审计留痕）。

决策大脑是「四类输入 + 一个策略决策」：主体 + 资源（客服 agent 替张三查——主体是「张三经由 agent」不是 agent 自己）、动作 + 上下文（同一份文档客服场景可检索、营销场景不可用于训练）→ 策略引擎给 allow/deny/附加义务（脱敏、行列过滤、水印、短保留、人工审批）——授权在资源服务端强制执行，模型、查询或客户端都不能自行裁决。验收：越权测试集是上线门禁（无权限账号 20 条敏感问题零泄露才放行，调岗/离职后重跑），加安全团队三个追问的标准答案——权限多久同步一次（随连接器增量，高危变更触发式同步、延迟窗口进 SLA）、向量能反推原文吗（向量化 ≠ 匿名化，要主动讲——该脱敏照脱敏）、删了真删了吗（向量库 + 缓存 + 衍生数据三处联动清——评估集里复制过的文档片段最容易被遗忘；「一键遗忘」是设计出来的能力，不是天然就有）。与 Security 方案不重复收费：那边定规则与防护体系，这边是规则在数据管线上的施工。

08现场高频追问#qa

只收会改变某个决定的问题——会改变技术方案、风险判断、验收方法、对客沟通或下一步行动。

每题标出它加入本题库的日期：2026-07-11 来自讲义首版对练页。

我们数据很乱，AI 项目是不是做不了？2026-07-11

能做，但顺序要对：先花两三周做数据就绪（解析+清洗+建库），再上模型。乱不是障碍，「假装不乱」才是。

数据坑是 AI 项目烂尾的第一原因——把数据工程列成显性报价明细，既是收入项也是防烂尾的保险。

下一步　用评估器把四问过一遍——答案直接定排期与报价结构；拍脑袋给的工期都是要延期的。

依据　四问与总图

我们全是扫描件和图纸，能处理吗？2026-07-11

能，这正是解析选型的分水岭场景：中文扫描件重默认 MinerU 路线（CJK 版面最强），高价值文档叠 VLM 解析兜底。

表格保真与阅读顺序拉开数量级差距；英文基准分数对 CJK 场景参考有限——自测是唯一可信口径。

下一步　拿客户 20 份最难的文档跑对比测试，用结果定方案——永远不用厂商 demo 定选型。

依据　选型分水岭

接一个系统要多久？数据多久能同步一次？2026-07-11

分两类：主流系统有成熟连接器，天级；自研系统按五件事（认证/拉取/解析/ACL/增量）评估，通常周级。同步按源系统能力分档：webhook 准实时、轮询按小时/天。

真实问题是「哪些内容需要多新鲜」——制度文档天级足够、工单类才值得准实时，分级同步省一大笔。

下一步　拿系统清单当场分类给排期；连接器报价按「源系统 × 五件事」列明细。

依据　五件事与三模式

要不要专门买一套向量数据库？2026-07-11

多半不用急：已有 Postgres 用 pgvector（生产可用到五千万至一亿向量），已有 ES 升级配置就支持混合检索。规模触顶或过滤要求极高时再迁 Qdrant/Milvus。

新引一套系统的运维成本常超性能差；量级内的毫秒差距对多数应用无感——选型看能力匹配度不看跑分。

下一步　方案书附「触顶预案」：监控哪三个指标、什么阈值启动迁移、双写灰度方案——预案本身就是专业度。

依据　按规模演进

AI 老答错，是不是模型不行？2026-07-11

先归因再下结论：经验上八成在检索侧，检索问题的大半又在数据侧——解析错、文档缺、内容过期。坏答案分析一路追到数据层，多数「模型问题」修完数据就好了。

「修数据而不是改提示词」是本册的中心论点——归因占比是对客最有力的论据。

下一步　挑 10 个坏答案做归因练习（检索/解析/缺文档三类占比），把「感觉」变成数据。

依据　坏答案回流

标注要雇多少人？合成数据可靠吗？2026-07-11

尽量不雇：专家半天写种子、模型扩量、人工只抽检 5–10%。合成有纪律就可靠：真实种子打底、judge 过滤、抽检合格率硬指标——纯合成才有坍缩风险。

预算换个花法效果更好；业务专家时间只花在定标准、写种子、审边界三件事上——每周两小时撑起整个标注运营。

下一步　坏例先分流：「不知道」修数据当天见效、「不听话」攒批次微调——分流错了钱白花。

依据　三笔钱与双线

数据向量化之后是不是就安全了？删除能真删干净吗？2026-07-11

向量化 ≠ 匿名化——嵌入在一定条件下能部分反推原文，敏感数据进库前照做脱敏、向量库照做权限隔离。删除要三处联动：向量库、缓存、衍生数据（评估集/训练集候选）一起清，留审计痕。

「一键遗忘」是设计出来的能力——最容易翻车在评估集：坏例回流时复制过的文档片段常被遗忘。

下一步　上线门禁跑越权测试集（20 问零泄露），权限变更后重跑；删除演练把评估集纳入清单。

依据　三个执行点
