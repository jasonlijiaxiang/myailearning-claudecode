01平台全景#aip-overview

裸机之痛四种浪费：卡在谁手里靠群里喊 Excel 记、贵卡半夜空转（无管理集群平均 利用率常只有 10–20%）、一个人的进程 OOM 把同机别人拖垮、训练跑到第 20 天崩了从头 再来。上了调度 + 配额 + 抢占后翻到 50%+ 很常见——卖 AI 平台的第一句话不是「我们 能管 GPU」，而是「我们能把您已经买的卡的利用率翻一倍」。

平台四大职责像机场塔台（塔台不开飞机，但没它机场就瘫痪）：调度（排跑道——最贵 的卡永远有活干）、隔离（分廊桥不打架）、容错(天气突变改航保安全）、 观测（雷达全程盯着）——客户的任何平台需求先归类到这四项，方案就有骨架。为什么 都长在 Kubernetes 上：像盖楼不自己烧砖——声明式 API、自愈、CRD 生态、云全托管，GPU 调度监控存储都是往框架上挂的插件；AI 平台 ≈ K8s + 一堆 GPU 感知的插件。Slurm 在 纯训练/HPC 仍主流且常与 K8s 并存——趋势是融合不是替代。分工一句话：Compute 讲铁、 Training/Inference 讲负载本身、本页讲「把负载高效跑在铁上」的中间层。

02K8s + GPU：从数卡到懂卡#aip-k8s-gpu

K8s 原生只认可数资源（CPU 毫核、内存字节），GPU 是「整块、有型号差异、能多种方式 共享」的特殊设备——需要「翻译层」讲给调度器听。第一代 device plugin 解决了 「看见卡」（nvidia.com/gpu: 2），但三个天花板：只能数数量（表达不了「要 80GB 显存 那种」「要 NVLink 直连的两张」）、共享方式僵硬、调度器不懂拓扑——像点菜只能说 「来 2 份」，不能说「要不辣的、靠窗的」。

第二代 DRA（K8s v1.34 于 2025-08 GA 并默认启用，GKE/AKS/OpenShift 已跟进； KubeCon EU 2026 上 NVIDIA 把 GPU 的 DRA 驱动捐给 CNCF——2026 年 GPU 调度的社区主线 就是它）：用「设备类 + 声明」结构化表达需求——型号/显存/是否 MIG、要不要拓扑 相邻、以什么方式共享，调度器第一次真正「读懂」GPU。从「来 2 份」升级成标准点餐单。 对客口径：新平台按 DRA 设计、老平台 device plugin 仍能跑、两者可共存过渡；机制是 K8s 上游标准，AMD 也有对应插件——标准化本身就是防锁定。GPU Operator 是一揽子 安装器：驱动、容器运行时、插件/DRA 驱动、DCGM 监控、MIG 管理器一个 CRD 装齐——扩容 100 节点自动就绪版本统一，是集群能弹性长大的前提。平台标准起手式：托管 K8s → GPU Operator → 选调度器 → 配切分 → 接监控。

DRA GA 与捐赠核实 2026-07-09，来源 kubernetes.io / KubeCon EU 2026 报道。 不可外推边界：GA 说的是接口稳定，不等于各发行版与云厂商都已跟进；落地前查目标集群的版本与厂商支持矩阵。

03作业调度#aip-scheduling

gang scheduling 是 AI 调度的分水岭：像划龙舟——20 个桨手必须同时上船，先上 12 个不但划不动还白占着船。分布式训练要么一次给齐、要么一张都不给；K8s 默认调度器 没有它，会造成「半个作业占着卡启动不了」的死锁——这不是优化项，是能不能正确跑 起来的问题。判断调度器能不能跑训练，第一问就是支不支持 gang。

碎片是贵卡空转的隐形元凶：像电影院还剩 20 个座但都是单座——总空闲卡够、却 拼不出「同机 8 卡/同域 64 卡」的形状，大作业照样排不进去，利用率虚高实低。解法是 bin-packing 装箱 + 拓扑感知：小作业塞满同节点、给大作业留整块——碎片治理直接等于 利用率。调度器不是二选一，是分层叠用（2026 共识）：准入/配额层用 Kueue（队列、 配额、作业准入），成组放置层用 Volcano（老牌批调度、稳）或 KAI Scheduler（NVIDIA 2025-04 开源 Apache 2.0，gang + 拓扑感知 + 装箱 + DRA 集成，与商业版 Run:ai 同一 调度核——Run:ai 的溢价买的是 UI/多集群/分片运行时/SLA）；Slurm 守 HPC 与托管服务。 小集群（<16 卡）单用 Kueue 或 KAI 够用。配额 + 抢占 + 回填三件套让贵卡既公平 又不空转（像手术室排班：科室保底额度、急诊插队、大手术备台空档插小手术）：生产训练高 优先级保按时、实验作业可抢占享受空闲卡——被抢占的作业先 checkpoint 再暂停、从断点续跑 （靠第 5 节容错），是双赢不是牺牲。

04GPU 切分与多租户#aip-sharing

整卡独占对小负载是奢侈（小模型推理、开发调试、notebook、CI——给它们各配一张 H100 等于用跑车送外卖）；切分是继调度之后提利用率的第二板斧——一张卡跑 5 个小推理 服务，利用率从独占的 15% 拉到 70%+。像大三居隔成独立单间分租。

切法显存隔离故障隔离灵活度 / 卡型典型场景
MIG（最多 7 实例）强（硬件）强（硬件）档位固定 / 特定卡生产多租户、对外服务
时间片无无高 / 任意卡内部开发、CI、低优批处理
MPS弱弱高 / 任意卡可信任务并发提吞吐
HAMi（CNCF）中（软件限额）中高 / 任意卡非 MIG 卡要显存配额

三把尺子选型：生产多租户用 MIG（硬件砌墙，一户失火不殃及邻居，性能可预测； 缺点是档位固定、改配置要重置）；内部可信榨利用率用时间片/MPS（共享办公室轮班 ——便宜灵活但没隔墙，一个任务掀桌大家遭殃，绝不用于互不信任的外部多租户）；非 MIG 卡要配额用 HAMi（给合租房间装智能水电表设上限——填补硬隔离与无隔离之间的空档）。 先问「这些租户彼此信任吗、是不是线上服务」再定切法和份数。2026 主线：DRA 把各种 切法统一到声明式接口下——不是取代是统一编排；切分要配合配额与公平共享，防「一租户 拖垮全卡」。

05训练容错工程#aip-faulttol

规模 × 时间 = 故障必然：同步训练所有卡每步对齐，任何一张卡出问题整个作业 停摆（一万人乐团连奏一个月，任一乐手断根弦整场停）。真实账本：Llama 3 在 1.6 万张 H100 上 54 天中断 466 次、约八成硬件相关——平均每天好几次，是万卡尺度的物理规律。 比的不是坏不坏，是坏了多快恢复。

checkpoint 两难与破局：存太勤浪费吞吐（每次暂停写 TB 级）、存太疏一崩重算 几小时——像写论文存档。破局 = 异步（先秒级快照到内存/本地盘、后台慢慢刷远端——解耦 「暂停时间」与「落盘时间」）+ 分级（高频存内存、低频存远端）。频率是「拿存储换算力」 的保险费，异步技术让保险变便宜；存储侧的洪峰设计在 姊妹册第 7 节。自愈循环 四步：检测（DCGM 指标 + 主动探测发现坏卡坏链路）→ 驱逐 → 替换（备用池补位）→ 续训（全体从最近 checkpoint 恢复）——像自动驾驶的赛车维修队，车手不用下车。有无自愈的 差距：人工救火每次几十分钟到几小时、自动化压到分钟级——乘以中断次数，goodput 差距可达 一成以上，纯利润。托管训练服务的容错是「该不该上云」的关键论据：HyperPod 类 自动健康检查、坏节点替换、断点续训开箱即带（2026 又加一键建集群、Slurm continuous provisioning）——自建这套自愈体系工程量巨大；「省下的救火人力 + 挽回的 goodput」比 功能清单更打动人。

HyperPod 更新核实 2026-07-09，来源 AWS 官方 文档。 不可外推边界：这是单一云厂商的产品能力，不能外推成「托管训练平台都这样」——各家在容错粒度与计费方式上差别很大。

06可观测性与利用率#aip-observability

「我们 GPU 利用率 90%」往往是幻觉。三层口径像出租车三本账：占用率（车被派 出去了——可能在空跑）≠ 活跃率（表在跳字——nvidia-smi 的 GPU-Util 只表示有 kernel 在执行，不管是不是有用功）≠ 有效率（真载着客人往目的地开——MFU/goodput）。三者能差 好几倍：同一训练 nvidia-smi 显示 95% 活跃，MFU 一算只有 38%。

承诺和验收都用 MFU（训练）和 goodput（生产），别用活跃率——前者是钱，后者是 错觉。MFU 40% 不是水平差，是行业顶尖（Llama 3 也在 40% 上下）——目标不是 100%， 是把黑洞一个个堵小。四大利用率黑洞按大小排查：数据加载（GPU 等存储喂数据—— 常是最大的洞）→ 通信等待（集合通信没和计算重叠）→ 碎片排队 → 故障重算——本册前几章 正好是四把铲子，观测的价值是告诉你「今天漏得最多的是哪个洞」。观测底座：DCGM （显存/算力/温度/功耗/NVLink 流量/XID 错误码全从它出）喂 Prometheus/Grafana。机制侧 是 showback/chargeback 成本分摊：像合租装分户电表——以前电费均摊没人关灯， 每户看到自己的账单浪费立刻自觉下降；没有分摊，利用率是「公地悲剧」。闭环：DCGM 采数 → 看板暴露黑洞 → MFU 定基线 → chargeback 让各团队为自己的利用率负责——技术 + 机制 双管，利用率才可持续。

07推理服务平台化#aip-serving

训练与推理对平台的要求几乎相反：训练是作业、批处理心态（包场办婚宴——gang 一次要 一大把卡、跑完释放、能排队）；推理是服务、7×24 常驻（开快餐店——延迟敏感、按流量 弹性、多副本高可用）。混部常态：推理占基线卡保 SLA、训练作为低优先级填谷， 流量一涨就被抢占让位——一套集群两种节奏。

引擎是货、平台是货架：vLLM/SGLang 负责把模型跑快（细节在 LLM-Inference 网页版），平台层负责部署好、 扩缩好、路由好——KServe（K8s 标准推理服务框架）、llm-d 与 NVIDIA Dynamo（1.0 于 2026-03 GTC GA：P/D 分离编排 + KV 感知路由 + NIXL 传输）。厨房再强，没有连锁运营也 开不成店——能跑 demo 和能扛百万请求，差的就是这一层。弹性扩缩三要点：按队列 长度/TTFT/并发/KV 占用而非 CPU（LLM 推理瓶颈在 GPU 显存，CPU 很闲——像餐厅加服务员看 排队长度不看后厨电表）；提前预测扩容（冷启动要加载几十上百 GB 权重、几十秒到分钟级， 不能等队列爆了才扩）；留热备副本。发布要可追溯：源与数据、运行与环境、制品与评测、 发布与运营四类对象用不可变 ID 连成图（食品溯源码：出问题一码到底、双向可查）——审批与 门禁长在图上，不在流程文档里。

08云上算力形态#aip-cloud

云上四种形态是一条「责任递交线」，像住的四档：①GPU 裸金属/虚机（毛坯房—— 调度/切分/容错/观测全自己搭）②托管 K8s+GPU（精装公寓——云管控制面和插件，策略自己 配）③托管训练/推理集群（酒店式公寓——连调度容错都包，HyperPod/Vertex 类）④托管 API（住酒店——连模型都给）。越往上越省心、越少定制、单位价越高——选形态本质是 选「你想自己扛多少平台层」。

本页四大能力对应「自建的隐藏工作量」清单：调度（裸金属自装 Kueue/Volcano/KAI）、切分 （自配 MIG/HAMi）、容错（自建自愈体系——最难，常是压垮自建决心的那根稻草）、观测 （自搭 DCGM + 看板）——客户说「我们自己买卡建」时，把四列活摆给他看。三问定档： 有没有专职基础设施团队（没有 → 往上选托管）、负载是啥（只调 API → 直接 API；训练微调 → 托管训练或托管 K8s；极致定制压成本 → 裸金属）、多快上线（急 → 托管）。常见落点： 初创只调模型 → API；有 AI 团队要微调 → 托管训练集群；大厂稳定大负载 → 自建 + 自研 平台；多数成熟客户是混合：基线自建、弹性和尝鲜上托管/API——混合方案恰恰最能 体现顾问价值。演进纪律：按责任递交线渐进（先托管验证、量上来再下沉自建），底层用 K8s/DRA 开源标准降低锁定风险。售前别一上来推最贵的——先诚实定档，信任感比单笔大单 更值钱。

09现场高频追问#qa

只收会改变某个决定的问题——会改变技术方案、风险判断、验收方法、对客沟通或下一步行动。

每题标出它加入本题库的日期：2026-07-09 来自讲义首版 对练页。

我们卡不多，直接 SSH 上去跑不行吗？2026-07-09

几张卡、一个团队时够用。但卡数或用卡的人一多，SSH 模式立刻失控：抢卡、空转、 互相踩踏。平台层的价值是把利用率从个位数拉到 50%+。

无管理集群常年 10–20% 利用率，上调度和配额后翻倍很常见——等于 「不花钱多买了一倍的卡」，是整个 AI 投资里 ROI 最高的一环。

下一步　四问体检任何 GPU 环境：谁在用哪张卡？利用率 多少？作业挂了会怎样？训练崩了能自动恢复吗？

依据　裸机之痛

K8s 自带调度器不能跑训练吗？2026-07-09

默认调度器没有 gang scheduling，会造成「半个作业占着卡启动不了」的死锁。跑 分布式训练必须上 Volcano/KAI 这类支持成组调度的——不是优化项，是能不能正确跑 起来的问题。

选型别问「哪个最好」，问「准入配额用什么、成组放置用什么」—— Kueue + Volcano/KAI 分层叠用是 2026 共识。

下一步　利用率上不去先查碎片和缺抢占——bin-packing + 配额 + 抢占三件套通常明显改善。

依据　gang 与分层

一张卡到底能给几个团队用？2026-07-09

看隔离要求：MIG 硬件切分最多 7 个隔离实例，适合互不信任的多租户；软件方式 （时间片/HAMi）可以更多但隔离弱，适合内部可信任务。

三把尺子（显存隔离/故障隔离/灵活度）定切法：生产用 MIG、内部榨 利用率用时间片、非 MIG 卡用 HAMi——先问租户彼此信任吗。

下一步　切分对「小而多」的推理场景立竿见影（15% → 70%+）；大训练反而要把多张卡合成一张用。

依据　四种切法

故障真有那么频繁吗？感觉有点夸张。2026-07-09

有公开数据：Llama 3 在 1.6 万卡上 54 天中断 466 次、约八成硬件问题——平均每天 好几次。这是万卡尺度的物理规律，不是某家运气差。

容错不是加分项，是有效算力的乘数：坏了 30 分钟恢复 vs 3 分钟 自动恢复，一个月下来 goodput 差一大截。

下一步　用客户自己的规模算账：中断次数 × 人工恢复时长 vs 自愈时长——客户往往低估频率、高估恢复速度。

依据　故障必然

nvidia-smi 利用率一直 90%+，还有优化空间吗？2026-07-09

大概率有：那只是活跃率（有 kernel 在跑），不代表有效。真正的指标是 MFU—— 大规模训练 40% 就算优秀；90% 活跃率背后往往藏着一半浪费。

三层口径（占用/活跃/有效）能差好几倍；四大黑洞按序排查：数据 管线 → 通信重叠 → 碎片 → 容错。

下一步　三层体检表 + 四黑洞清单做免费利用率诊断—— 先量出浪费，方案自然就有了；配 chargeback 让浪费有主。

依据　三层口径

有了 vLLM 这些引擎，为什么还要推理平台？2026-07-09

引擎把单个模型跑快，平台把它变成能扛生产的服务：多副本高可用、按流量弹性、灰度 发版、KV 感知路由。能跑 demo 和能扛百万请求，差的就是这一层。

推理扩缩看队列/TTFT/KV 而非 CPU；冷启动分钟级所以要提前预测、 留热备——这是和无状态 Web 最大的运维差异。

下一步　训练推理同集群混部：推理保基线 SLA、训练填谷做 低优——一套集群两种节奏，利用率最高。

依据　引擎与货架

我们到底该自建平台还是用托管？2026-07-09

三问定档：有没有专职基础设施团队、负载类型、上线速度。没团队或要快 → 托管集群/ API；有团队、负载稳定、压长期成本 → 自建。多数成熟客户是混合。

自建的隐藏成本 = 调度/切分/容错/观测四列活，尤其容错自愈最难—— 托管的溢价买的正是这些。

下一步　用定档器把三问过一遍； 演进按责任递交线渐进，底层用 DRA 等开源标准防锁定。
