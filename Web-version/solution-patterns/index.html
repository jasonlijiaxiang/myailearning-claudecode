01从技术轴到场景轴#sp-what-why

客户不会问「你们 RAG 用什么重排序」，他问「我的客服能不能少雇十个人」——场景轴负责把后者翻译成前者。工作循环五步：需求发现 → 参考架构 → POC → 报价 → 落地运营； POC 失败回到需求发现重挑场景是常态不是事故。角色像全科医生：先分诊、开检查单，深度问题转回技术模块（转专科）——价值在业务语言与技术语言之间的「翻译」。

同一块积木在不同场景里角色完全不同——RAG 在客服里是「知识兜底」（防幻觉第一道闸）、在企业搜索里是「绝对主角」（权限感知检索就是产品本身）、在营销里是「幕后配角」（产品参数的事实源）。对客永远从场景讲起、技术配角出场：先说「少雇十个人」的漏斗，再说漏斗里那格是 RAG——顺序反了就成了推销技术。「先做哪个场景」的通用答案：内部知识库或客服（数据现成、风险可控、ROI 好算），更负责的答法是按数据就绪度 × 容错成本排矩阵。

客户开口第一句 → 场景定位 + 积木清单 + 成本量级 + 口径提醒。分诊错了后面全错——宁可多问三个问题再分诊，也别顺着第一句话直接进方案。判据全部来自本页各节，不新增事实。

02方案共性方法#sp-method

通用参考架构五层：POC 常只建模型 + 数据两层，方案书必须画全五层——客户的安全团队和采购看的恰恰是后三层，这也是「POC 成功了为什么上生产还要收钱」的答案。

POC 三要素

像装修先砌样板墙——三样缺一样，POC 就会变成永远「再试试」：

真数据

拒绝演示数据——拿不到真数据本身就是项目风险信号。

签字的指标

提前书面定义「好」，指标要能自动测。

验收线

低于多少算失败、高于多少扩大。

前置一层是一页 intake（把诉求变成决策合同）：结果 / 用户 / 数据 / 风险四个字段，任何一个没证据就退出——「能停」是专业不是失败。

成本三本账

账怎么算为什么重要
token 账输入输出分开算缓存能砍 50–90%
并发账峰值定限流档位私有化下直接决定卡数
人力账提示词维护 / 知识运营 / 评估回归最常被漏报的一笔，漏了它报价第二年爆雷

报价公式：单位成本 × 用量 × 1.3 缓冲 + 人力运营包——先量级后精确。

口径鉴别术（性价比最高的专业度信号）：deflection 挡走率 ≠ resolution 真解决率（追问「被挡走的用户后来去哪了」）、演示数据 ≠ 真数据（「换我的数据现场跑一遍？」）、最佳延迟 ≠ P95、起步价 ≠ 满负荷 TCO（「按我的量算三年总账？」）。数据锚点：客服 AI 厂商自报解决率 51–80%，独立口径（Zendesk 企业全量）中位仅 41.2%——30–40 个百分点的差距是常态。五类高频坑没有一个是技术坑（需求/数据/期望/验收/安全）——AI 项目烂尾，八成烂在方法和预期管理上。

03智能客服#sp-customer-service

客服 AI 的目标不是替代人工，是漏斗前移：L1 自助扛量（FAQ/RAG 直答）→ L2 AI 深度办事（Agent 查单改地址退款，带审批）→ L3 人工处理复杂/情绪/高危。升级率是健康指标不是失败指标；升级时带摘要，别让用户再说一遍。对客话术：让人工从「答 FAQ」升级到「处理值钱对话」。

积木清单

RAG兜知识——答案必须有出处
Agent办业务——高危操作 AI 草稿 + 人工确认
语音链路有电话渠道才上：SIP 网关接呼叫中心、级联为主可审计、P50 <1.5s、$0.01–0.17/分钟——深潜在 Multimodal 网页版第 8 节
Gateway入口治理
Evaluation永远不是可选件

口径战场是本场景的灵魂

自报 51–80% vs 独立中位 41.2%——承诺测法不承诺数字：用客户真实工单建评估集测基线，验收线一起签。

成本对标着讲 ROI：文本 $0.01–0.05/会话、语音 $0.01–0.17/分钟 vs 人工 ¥3–8/通——但 ROI 只对「被真解决的那部分」成立，解决率口径直接决定 ROI 真假。

04企业知识库与 AI 搜索#sp-knowledge-search

先分形态再报价，两者差一个数量级：部门级知识库问答（圈定文档集、RAG 直答、低代码平台周级上线，工程量以知识运营为主）vs 全公司 AI 搜索（全司异构源：邮件/IM/ 文档/工单/CRM，连接器 + 权限是主工程，季度级）。客户说「我要个企业知识库」，第一件事是分清他要哪种。

权限是命门

ACL 从源系统继承

不另建一套。

查询时实时过滤

不是索引时过滤一次。

存在性也是机密

无权限时连「有这份文档」都不暴露。

POC 验收必含越权测试集：无权限账号问 20 条敏感问题一条都不能漏，安全团队签字才上线。

连接器工程最容易被低估：每个源系统都是一个小项目（认证、增量同步、格式解析、ACL 映射）——报价按源系统个数算。

市场锚点：Glean 模式

ARR $300M（2026-05，+89% YoY）、估值 $7.2B、Agent 年化 1 亿+ actions——卖的是「权限感知知识图谱」不是搜索框，「找得到 + 不泄密」被市场定过价；且搜索正在变成 Agent 的地基：先找得到，才谈得上会办事。

文档质量差怎么办：垃圾进垃圾出，但方案可带知识运营——覆盖率仪表盘找高频无答案问题，反向驱动文档补全。

Glean 数字核实 2026-07-10，来源 TechCrunch 2026-05-28 / Glean press。不可外推边界：这是全公司 AI 搜索这一形态的市场锚点，是公司层面的数字不是项目数字；部门级知识库问答不适用这个量级——两种形态的工程量差一个数量级。

06AI Coding 与研发提效#sp-ai-coding

「三选一」是伪命题——双层格局配组合：IDE 补全/编辑层管单人效率（Copilot/ Cursor，按人头月费）+ agentic 层管整任务委托（Claude Code 类终端 agent，端到端修 bug、按用量计费）。70% 工程师同时用 2–4 个工具；网关统一管预算，工具迭代不伤架构。

报价量级（2026-07）：$19–40 / 人 / 月的 IDE 层月费 + agentic 层的 token 用量——两层分开算，别只报一个「人头费」漏掉 agent 层的用量账。

08ChatBI 与数据分析#sp-chatbi

三层需求先分清（混着报价必翻车）：L1 查数（「上月华东区收入多少」，两三周能上）→ L2 归因（「为什么环比降了」，依赖语义层厚度）→ L3 报告（Agent 工程）。准确率的真相在语义层不在模型：裸 Text-to-SQL 随基准的真实程度断崖下滑——学术单轮基准 80–90%、较干净的企业化基准五六成、最贴近真实脏 schema 的企业基准（Spider 2.0 类）仅 10–21%，对客一定讲最低那档；绑定语义模型后 51%→90%+（Snowflake 内部 150 题），dbt 语义层基准上限 98–100%。机制底座见 RAG 结构化数据节。

把「你们准确率多少」转化为「语义层建到多厚」，口径治理问题就赢了。

三道闸控住「算错数」风险

算错一个数比答错一段话严重得多：

① 语义层

高频指标只走语义层确定性生成。

② 只读 + 行级过滤

SQL 只读执行 + 行级权限过滤。

③ 可复核

答案附生成的 SQL 与口径出处。

裸 SQL 只留给内部临时分析。

新变量：语义层「太贵」的理由正在失效

Snowflake 2026 推出 Semantic View Autopilot——用 ML 从元数据和 BI 使用模式自动发现指标定义，把建语义层从天级压到分钟级。异议处理时要知道这条：客户拿「语义层建设太贵」当反对理由时，它已经在被自动化削弱。

成本结构：大头是语义层建设与运营的人力，不是模型调用——起步 10–20 个高频指标定口径，两三周出 POC。

10现场高频追问#qa

只收会改变某个决定的问题——会改变技术方案、风险判断、验收方法、对客沟通或下一步行动。

每题标出它加入本题库的日期：2026-07-10 来自讲义首版对练页，2026-07-11 来自 ChatBI 与会议助手两章。

POC 成功了，为什么上生产还要再收一笔钱？2026-07-10

POC 只建了模型和数据两层样板墙；上生产要补入口、保障、运营三层——高可用、权限、审计、评估回归都是生产件，这才是工程大头。

「POC 建两层、方案书画五层」是本册最常用的一张图——客户的安全团队和采购看的恰恰是后三层。

下一步　四阶段交付门（价值/技术/NFR/运营）把风险逐步收敛——每阶段有独立退出条件，「能停」是专业。

依据　五层架构

你们报价比竞品贵 30%，贵在哪？2026-07-10

先对齐口径：我们的报价含人力运营包和评估验收，低价单常报裸 token。按满负荷 TCO 对比通常持平甚至更低——账可以当场一起算。

人力账（提示词维护/知识运营/评估回归）是最常被漏报的一笔——漏了它的报价第二年爆雷。

下一步　报价公式：单位成本 × 用量 × 1.3 缓冲 + 运营包；集成费单列别藏进单价。

依据　三本账

客服解决率能承诺多少？2026-07-10

不先拍数字——厂商自报 51–80%，独立实测中位只有 41.2%，差在口径（deflection ≠ resolution）。承诺测法：用真实工单建评估集测基线，验收线一起签。

30–40 个百分点的自报与实测差距是常态不是个例——会追问口径的售前，客户当场就能感到「这人懂行」。

下一步　拿 20 条真实工单手工标注「真解决」比例——这就是讲口径战场时最有力的一手证据。

依据　口径战场

员工会不会搜到不该看的东西？2026-07-10

这是方案的第一设计约束：权限从源系统继承、查询时实时过滤、无权限连存在性都不暴露。POC 验收带专门的越权测试集，安全团队签字才上线。

搜索越好、泄密越快——权限缓存过期、索引时过滤一次、另建权限都会漏；权限必须跟着「查询这一刻的身份」走。

下一步　权限不是「搜索的一个功能」，是「搜索能不能上线」的开关——安全评审主答卷提前准备。

依据　权限命门

生成的内容侵权了算谁的？2026-07-10

两层兜底：选商业安全定位的模型（训练数据可溯源、企业 IP 赔付条款），加发布前人审留痕。合同权责写清——这是方案区别于「随便接个 API」的地方。

对外内容的翻车成本是指数级的——商业安全不是加分项，是营销场景的入场券。

下一步　人审按风险分级：对外广告 100%、社媒抽检 10–30%、模板变体机器初筛——风险定价而非不信任。

依据　商业安全

AI Coding 该买 Copilot 还是 Cursor 还是 Claude Code？2026-07-10

不是三选一：主流是 IDE 层选一个 + agent 层选一个的组合，网关统一管预算。按团队画像配比——工具会迭代，接入架构不变。

双层价值不同：补全管单人效率（按人头），agentic 管整任务委托（按用量）——70% 工程师同时用 2–4 个工具。

下一步　2 团队 × 8 周试点出自己的数字（PR 周期/缺陷率/ DORA）；enablement 决定效果差 3 倍，写进方案。

依据　双层格局

数字人会不会有法律风险？2026-07-10

三条红线管住就可控：真人形象与声音书面授权（含撤回机制）、AI 生成显著标识、内容审核照常跑。合规是默认件不是选配。

三问（形象哪来的、标不标识、授权链全不全）不过项目直接停——答得好，合规就是竞争力。

下一步　先分清内容工厂还是交互服务——差一个量级的是架构和成本结构，不是价格策略。

依据　合规红线

ChatBI 准确率能到多少？算错数怎么办？2026-07-11

取决于语义层：裸 Text-to-SQL 真实企业库只有一两成，绑定语义模型后能到 90%+。先建 10–20 个高频指标口径，用验收集实测。算错数靠三道闸：高频走语义层确定性生成、SQL 只读+行级过滤、答案附 SQL 与口径出处。

把准确率问题变成治理问题是本场景的胜负手——真难点是各部门口径打架（组织问题），售前提前做预期管理。

下一步　选型第一问「数据在哪」：已集中就顺势用原生问数，多源分散用独立语义层——不是「哪家模型准」。

依据　语义层跃迁

会议助手和 Teams/Zoom 自带的 AI 比，强在哪？2026-07-11

原生助手管单会够用。差异在三处：跨平台统一的会议记忆库、与业务系统的行动闭环（纪要直接变任务/CRM）、私有化选项。只要单会纪要，直说用原生的就够。

「直说用原生的就够」本身是信任建设；独立方案的入场券是差异化需求，不是功能对标。

下一步　验收分三层设指标：转写 WER、摘要忠实度、行动项召回——漏了「老板随口交办」才是客诉源。

依据　两形态三层价值
