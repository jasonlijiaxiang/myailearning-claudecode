01是什么 / 为什么#a2a-what-why

真实企业里不是一个万能 Agent，而是一堆专精 Agent：订票、报销、风控……像公司里不同部门的员工，靠「流程」把活儿流转。缺了标准就是 M×N 的定制对接地狱——A2A 把它塌缩成 M+N：规定 Agent 如何发现对方、委派任务、同步进度、交换结果。做好一次就通用：一个「合规审查 Agent」发布后，任何遵循 A2A 的上游都能直接委派，不用一对一对接。

MCP（模型上下文协议）A2A（智能体间协议）
连接方向Agent ↔ 工具/数据（纵向）Agent ↔ Agent（横向）
类比给 Agent 装的 USB 接口Agent 之间的协作/外交规范
解决让 Agent「会用」外部能力让多个 Agent「会协作」
粒度一次工具调用（tools/call）一次任务委派（Task，可长时运行）
关系互补，一个系统里两个都要——A2A 编排的 Agent 内部往往正用 MCP 调工具

客户最爱问「选哪个」，标准答案是「都要」——同一次差旅审批里，两个协议管的是两个方向：

来历与治理：Google 2025-04 发布（50+ 合作伙伴）→ 2025-06 捐给 Linux Foundation 中立托管——与 MCP 同一条「先厂商提出、再基金会治理」的路，是回应「会不会被锁定」最有力的答案。

02协议解剖：五大对象#a2a-protocol

记忆法一行：Agent Card＝名片（发现）、Task＝工单（有状态）、Message/Part＝信与附件（内容）、Artifact＝成品（产出）。规范 v1.0 以 Protocol Buffers 为唯一权威定义——保证三种传输绑定语义一致、跨语言 SDK 不跑偏。

对象是什么像什么 / 要点
Agent Card一份 JSON 元数据：name、url（服务端点）、capabilities、skills、鉴权要求，默认发布在 well-known URI商务名片 + 服务菜单——「我是谁、能做什么、去哪找我、进门要什么证件」。本身是公开菜单不含密钥；敏感能力靠 securitySchemes 声明鉴权，v1.0 还支持签名防冒充（第 7 节）
Task工作基本单元：唯一 ID + 状态 + 生命周期像给外包发的工单——有进度、能查询、能补料、能取消。「有状态」是 A2A 与一次性 RPC 的关键差别：天生支持长耗时、多轮、需人工介入的真实协作
Message一轮通信，带 role由若干 Part 组成
Part最小内容块text / file / data 统一抽象
ArtifactTask 的产出，也由 Part 组成可增量流式返回——大文件传 URI 引用而非内联

Task 状态机九种状态、三条路：正常线 submitted → working → completed；中断线 working ↔ input-required / auth-required（补料补授权后回来继续）；异常线随时可进 failed / canceled / rejected（另有兜底 unspecified）。两条纪律：终态不可回跳——重试＝创建新 Task（可关联旧任务），绝不修改终态历史，审计与回放靠这条撑着；状态真相永远在服务端，客户端只是观察者。

TaskState 九态核实 2026-07-12，来源 a2a-protocol.org 规范。不可外推边界：九态是规范定义的状态集，不是各家实现的现状——对接前按对方 Agent Card 与所用 SDK 的版本核一遍，引用状态机时带规范版本号。

03发现与传输#a2a-transport

发现：Agent Card 约定发布在 https://<host>/.well-known/agent-card.json ——像 robots.txt 放固定位置，按约定路径去拿就有，零协商免对接；不想公开可用私有注册中心 + 鉴权。传输三种绑定语义由 protobuf 统一约束、可互换——A2A 不逼你换技术栈。

绑定适合场景特点
JSON-RPC 2.0通用默认、Web 友好HTTP POST、人类可读、生态最广，preferredTransport 常用它
gRPC高性能、内网服务间HTTP/2 + protobuf，低延迟高吞吐、强类型，适合大规模集群
HTTP+JSON/REST已有 REST 基建贴合传统习惯，易接入现有网关/鉴权/监控，改造成本最低

拿结果的四种姿势

先判运行条件再选交付：

姿势怎么拿代价用在哪
Blocking同步等连接占用、易超时短活
Polling轮询空轮询与状态陈旧——用退避 + 变更标记网络最简单的场景
StreamingSSE 单连接持续推增量，比 WebSocket 更轻更易穿网关要处理断线重连与背压要看进度
Push客户端离线时服务端回调 webhook，像发快递留手机号回调必须幂等 + URL 校验 + 签名长任务

口诀：短活阻塞、简单轮询、看进度流式、长任务推送。同一个 Task 可同时被轮询 / 订阅 / 推送观察——通道只是观察方式。三种绑定都跑在 HTTP(S) 上，复用现有 443 与网关，一般不用单独开防火墙。

04跑通一次协作#a2a-handson

接入心智模型：server 端「把已有能力包一层 A2A 外壳」——声明 Agent Card + 把现有逻辑塞进 executor + 发布，三步；client 端「拉 Card + 发 message/send」，几行代码。业务核心基本不动——这是给客户估接入量的口径。

一次调用的真实报文（JSON-RPC）：请求 method: "message/send"，params 里一条 role 为 user 的 Message、Part 装「审查这张发票是否合规」；响应返回 Task——id: "task-7f3a"、状态 completed、Artifact 里 Part 装结论。

有什么
官方 SDK五语言（Python / JavaScript / Java / Go / .NET）覆盖主流后端栈，Python 已实现 1.0（含 0.3 兼容模式）
不用 SDK 行不行行——A2A 是开放的 HTTP + JSON-RPC / gRPC 协议，可手写报文对接，规范以 protobuf 为准
配套工具Inspector（调试观察报文）与 TCK（协议一致性测试套件）

一个 Agent 可以既是 server 又是 client——链式协作里，中间 Agent 对上游是 server、对下游是 client，这正是多 Agent 编排的基础。

05多智能体协作#a2a-orchestration

opaque agents（不透明协作）：Agent 之间只通过 Card 暴露能力、通过 Task 交换输入输出，不共享内部提示词、工具、推理过程——像把活儿外包给另一家公司，你告诉它「要什么」、它交「成品」，不会把对方员工怎么干全看光。不透明 = 更好的封装与安全边界，协作靠接口契约而非看内部——对跨厂商、跨组织协作是必需的。

委派拓扑三种基本形状

与单机编排同构，区别只在跨进程 / 跨厂商：

形状怎么走例子
编排者—执行者主 Agent 拆活、分发、汇总差旅审批调度订票 / 预算 / 风控
链式流水线A 的产出作 B 的输入翻译 → 校对 → 排版
并行汇聚同时委派，等全部返回再合并尽调并行查财务 / 法务 / 舆情

三层叠起来看

成本提醒：跨 Agent 有网络与模型开销——能单 Agent 完成就别拆，真正需要专精 / 隔离 / 跨厂商时，协作的收益才划算。

06生产落地与上云#a2a-production

上线前六条验收线

每条都问「查什么 + 拿什么证据」，答不出证据就还没到生产：

发现与版本

Card 来源可信、接口选择可重复——证据：签名 / 域名校验记录、supportedInterfaces 兼容表。

任务语义

状态流转合规、终态纪律、重试建新 Task——证据：事件序列可回放、失败 / 取消用例。

身份与授权

逐操作、逐租户、audience 限定——证据：跨租户越权回归、scope 用例、审计日志。

异步交付

断线重连、事件顺序、推送至少一次送达——证据：快照对账、重复通知的幂等证明。

运营与恢复

SLO、容量、故障演练、可回滚——证据：压测报告、告警配置、runbook。

质量与安全

业务结果、恶意输入、工具副作用——证据：评测集、红队记录、人工介入控制。

终极判断：任意一次协作都能回放「谁委派、做了什么、为何结束、交付了什么、谁验收」。

生态数字核实 2026-07-12，来源 Linux Foundation 一周年通稿；版本号来源 规范仓库 releases。不可外推边界：基金会通稿的采纳规模是自报口径、未公开统计方法：它能支持的是「不是某一家的私有协议」，不能当作「已经在生产里大规模跑起来」的证据。

07安全#a2a-security

Agent 自动发现、自动委派，攻击面随之扩大：假冒 Agent、越权调用、恶意投毒、数据经 Artifact 外泄——从「人工审批每一单」变成「机器自动接单」，不设门禁冒牌供应商也能混进来。自动化的收益必须配套默认设防：验证身份、限定权限、留痕审计，安全前移到协议层。

信任基石：Signed Agent Cards（v1.0 头号新特性）

给名片加密码学签名，接收方能验证「这张名片确实由该域名所有者签发」——像盖了防伪钢印，不再是「谁都能印一张说自己是银行」。它把发现从「默认信任」升级为「可验证信任」，是跨组织自动协作的前提。

鉴权与授权三件：OAuth 2.0（Card 声明 securitySchemes，调用带 token 校验 scope）+ mTLS（服务间双向证书）+ 授权范围（按 skill / 租户限定谁能调什么，越权直接拒）。

五个信任边界对五组控制

边界威胁控制
发现Card 可伪造可信域 + 签名
身份凭据扩散带外授权 + audience 限定 + 短期凭证
任务越权读取逐操作鉴权 + 租户绑定
模型跨 Agent 提示注入指令数据隔离 + 工具最小权限
回调伪造重放URL 校验 + 签名 + 幂等 + 限流

与 MCP 一套安全心法：验身份、限权限、默认关自动批准、全链路留痕——A2A 多出的「跨组织委派链信任」用签名名片补上。安全主张要能被测试复现（签名校验日志 / 越权回归 / 注入语料 / 重放测试），不靠口头承诺。威胁全景与架构级防御在 Security 网页版。

08现场高频追问#qa

只收会改变某个决定的问题——会改变技术方案、风险判断、验收方法、对客沟通或下一步行动。

每题标出它加入本题库的日期：2026-07-09 来自讲义首版对练页，2026-07-12 来自状态机与交付方式深潜。

A2A 和 MCP 到底啥区别，是不是二选一？2026-07-09

不是。MCP 连工具/数据（纵向），A2A 连 Agent（横向），一个系统里常常两个都用—— A2A 编排的 Agent 内部往往正用 MCP 调工具。

这是本册最常被问的一题：粒度也不同——MCP 是一次工具调用，A2A 是一次可长时运行的任务委派。

下一步　用差旅审批一张图讲两层：A2A 决定谁接力这一单， MCP 决定每个 Agent 手里有什么工具。

依据　分工对照表

能不能只用一个 Agent，别搞这么复杂？2026-07-09

能就别上。只有当存在多个独立、专精、可能来自不同厂商的 Agent 要协作时 A2A 才有价值；单 Agent 场景直接 MCP 即可。

跨 Agent 有网络与模型开销——拆分本身有成本，专精/隔离/跨厂商的收益要大过它才划算。

下一步　拿选型器把手头场景过一遍，判断尺子就一条：是否存在多个异构 Agent 要接力一件事。

依据　选型尺子

这是不是 Google 的私有协议，用了会被锁定吗？2026-07-09

不会。2025-06 已捐给中立的 Linux Foundation；150+ 组织参与、五语言官方 SDK、三大云托管且互通，治理和演进公开。

「中立治理 + 广泛采用」是回应「昙花一现」质疑最有力的组合——和 MCP 走的同一条去厂商化路线。

下一步　30 秒成熟度话术用四个数字组织：150+ 组织 / v1.0 稳定版 / 5 个 SDK / 3 大云。

依据　采用度

接 A2A 要不要换我现在的 Agent 框架？2026-07-09

不用。A2A 是框架之上的互操作层：LangGraph、ADK 等都能对外暴露/调用 A2A，内部实现不动。server 端主要是「包壳」，client 端几行发消息。

「业务核心不动」是接入量评估的关键口径——新增的只有 Card 声明、 executor 包装、发布三块。

下一步　挑熟悉语言的官方 SDK 跑通 hello-world 级 server + client，再给客户列接入清单。

依据　接入心智模型

Task 有状态，服务端要一直存着？会不会很重？2026-07-09

Task 状态需要持久化（内存/DB 皆可），但这正是长任务、断点续传、推送回调的基础；短任务也可即时 completed，不强制长存。

「有状态」是 A2A 与一次性 RPC 的本质差别——真实协作是长耗时、多轮、要人工介入的，工单模式才接得住。

下一步　记住终态纪律：终态不可回跳、重试建新 Task——审计与回放靠它；状态真相永远在服务端。

依据　Task 状态机

长任务客户端断了怎么办？2026-07-12

用推送通知：注册 webhook，Task 完成或变更时服务端回调你——客户端无需挂连接，天然适配离线与批量。

四种交付各有代价：阻塞占连接、轮询会陈旧、流式要处理断线背压、推送要防重复投递与回调伪造（幂等 + URL 校验 + 签名）。

下一步　口诀选型：短活阻塞、简单轮询、看进度流式、长任务推送——先判运行条件再选交付。

依据　四种交付

怎么防止别人伪造一个 Agent 冒充我们？2026-07-09

用 v1.0 的 Signed Agent Card：名片带密码学签名，接收方验签确认确由你的域名签发——伪造的 Card 过不了验证。

自动协作把发现从人工审批变成机器接单，「可验证信任」是跨组织委派的前提——这正是 v1.0 头号新特性的动机。

下一步　给安全负责人讲「默认设防」全景：验签 + 最小权限 + 关自动批准 + 全链路审计，别只说「支持 OAuth」。

依据　Signed Agent Cards

跨多个 Agent，出错了怎么定位？2026-07-09

每个 Task 有 ID 和状态、可观测，链路上逐段查即可；生产部署给每个 Task 打 trace，配统一可观测。

验收的终极判断：任意一次协作能回放「谁委派、做了什么、为何结束、交付了什么、谁验收」——答不全就还没到生产。

下一步　按六条验收线（发现/任务语义/授权/异步交付/运营/ 安全）列上线 checklist。

依据　生产验收
