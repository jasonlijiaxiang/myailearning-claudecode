01是什么 / 为什么#gw-what-why

M 个应用 × N 家模型 = M×N 条私接线——格式、鉴权、参数、限流各不同，改模型要动几十处 代码。AI 网关坐在应用与各家模型之间：对上给统一接口，对下接所有供应商，接入、路由、 成本、安全、观测在这一层集中治理。不止一个模型，就该有一个网关。

区别于普通 API 网关的「LLM 专属六件套」：统一接入/协议转换、按 token 限流、成本归集、 语义缓存、护栏/内容安全、AI 可观测。客户说「我有 Kong/APISIX 了」——正确回答不是取代， 是在它上面加 LLM 能力，很多产品本就是同一网关的「AI 模式」。多一跳会不会慢？网关转发是 毫秒级，而语义缓存命中、就近路由、失败快速兜底整体反而常常更快——大头延迟在模型推理， 不在网关。

边界辨析：一张表分清四个概念

概念管的是什么和 AI 网关的关系
普通 API 网关通用 HTTP 流量：鉴权、限流、路由AI 网关是其「LLM 增强版」
MCP / A2A（协议层）Agent↔工具、Agent↔Agent 的互操作标准网关是「落地协议 + 统一治理」的位置（第 7 节）
推理服务（vLLM 等）单个模型怎么又快又省地跑网关在其之上，管多模型统一治理
编排 / Agent 框架一次任务里怎么串工具与步骤网关提供统一模型入口与观测，不替代编排

记忆锚点:MCP/A2A 是协议、推理服务是引擎、AI 网关是运营管控层——互补不互斥。

02统一接入#gw-unify

网关对外暴露一套 OpenAI 风格接口，对内翻译成各家原生格式——应用只学一种调用姿势， 换模型只改 model 一个字段。像万能转接头：带一个欧标插头走遍全球。

各家在 function calling、JSON 模式、多模态、上下文长度上并不一致，网关做三件事： 参数映射、能力探测、优雅降级（不支持的特性降级或报清晰错误，个别厂商独有参数支持 透传）。要点澄清：网关解决的是「接入」成本，不是「效果对齐」成本——不同模型脾气不同， 提示词和输出解析可能要微调，后者靠评估。

虚拟密钥：一把逻辑钥匙管多把真钥匙

真 key 锁在网关的保险箱里，发给应用的是「房卡」：每把虚拟密钥可单独设预算、限速、 可用模型范围、有效期，出事一键注销。原厂 key 一旦泄露，账单和数据全裸奔且难吊销—— 虚拟密钥是企业级管控的地基，也是「最小权限/爆炸半径」在网关的具体落地。

深潜：十步请求链与五层策略栈

一条请求在网关里走十步：客户端认证 → 格式检查 → PII/提示护栏 → 模型别名解析 → 配额 → 路由 → 协议转换 → 上游身份 → 流式转发 → 用量与追踪。顺序即设计：越早拒绝浪费越少 ——认证放第一、护栏在路由前、计量在最后（像机场动线：把安检放在登机口之后，危险品就已经 进了候机厅）。配套给整条链定一个端到端 deadline，别让每层各自耗尽超时叠出分钟级等待。 身份侧五层策略栈：调用身份 → 租户映射 → 准入策略 → 资源配额 → 上游身份——像公司报销链， 每层各有账本，审计能一层层对上；日志记身份映射与策略版本，绝不记录原始凭据。

03路由与容灾#gw-route

路由就是给请求分诊：医院不会让所有病人都找主任医师。简单问答用便宜模型、复杂推理用 强模型，再叠加「哪家现在更快/没限流」——别用杀鸡的牛刀烧钱。

策略怎么选典型场景
成本优先按每千 token 价选最便宜的达标模型预算敏感、批量任务
延迟/负载选当前最快、未限流的后端高并发、实时交互
质量优先关键请求固定走旗舰模型对错误零容忍的场景
复杂度路由对请求打分（长度/代码/推理特征）选档位通用降本，零 embedding 依赖
语义路由按意图用向量相似度选专用后端多专家模型、领域分流

实战常叠加：先按语义/复杂度选档位，再在档内按延迟均衡，最后配容灾兜底。降本有论文背书： RouteLLM（LMSYS/Berkeley）训练路由器判断走强还是弱模型，在接近旗舰质量下把大量请求分给 便宜模型；FrugalGPT（Stanford）用级联——先问便宜模型、不够好再逐级升级。路由是 可控降级：关键场景质量优先，只对可容忍流量降档，用评估持续盯降档流量的质量。

路由省钱计算器

把「路由 + 缓存为什么省钱」算成账。价格是你自己填的口径（默认值只是量级 示意，当天价格以官网为准）；节省比例取决于流量结构，先做成本画像再承诺。

容灾：把爆炸半径关进笼子

三件套：fallback（主挂了自动改走备用模型/供应商，对调用方无感）、指数退避重试 （上限 1~2 次封顶——fallback 是保命不是无限重试，别把一个坏请求在多家模型上重复烧钱）、 熔断（连续失败的后端隔离一段时间，某路短路空开只跳那一路）。多区域部署再叠一层 地域容灾；数据合规用「数据驻留」路由约束强制——某类数据只允许走特定区域的后端， 网关是执行这条策略的最佳位置。

RouteLLM / FrugalGPT 见本模块书单（arXiv 官方链接）； 路由与选型的市场侧（三层路由 1:8:1、五约束）见 Model-Landscape 网页版—— 双向互指：那页定「选哪些模型进路由池」，本页是工程落地件。

04成本治理#gw-cost

LLM 的成本和负载由 token 数决定：一条几万字的长请求抵几百条短请求——按「请求数」限流 管不住成本。按用电量收费，不是按开关次数：TPM（token/分）+ RPM（请求/分） 双维度限。

花费归集像公司给每个部门发预算卡：每把虚拟密钥挂 TPM/RPM 上限、日/月预算、可用模型范围， 网关按 key 记账聚合到团队/项目——月底能出「谁花了多少、花在哪个模型」的账。接进 FinOps 看板后，哪个应用突然涨了、哪个模型性价比低一目了然，再回头调路由策略——成本治理和路由 是一对搭档。预算撞线时是「拒绝」还是「降级到便宜模型」，是要和业务约定清楚的产品策略： 接近上限先告警、再降级、最后才拒绝。

语义缓存：把相同的问题只回答一次

精确缓存一字不差才命中，换个说法就落空；语义缓存把请求算成向量、按相似度命中—— 「怎么退货」和「退货流程是啥」是同一个问题，命中即毫秒级返回、零模型计费。阈值是 关键旋钮：太松会把「北京天气」和「上海天气」当一个问题答错，太紧几乎不命中。落地 清单四条：①按场景分级设阈值；②个性化/实时请求绕过缓存；③缓存键带上租户/权限， 避免跨用户串答（既是成本设计也是安全设计）；④监控命中率与疑似错答持续调参。

云上口径核实 2026-07-09：Azure APIM 是目前唯一原生内置「按 token 限流 + LLM 专属策略」的云厂商网关（网关侧预估 prompt token 提前拦截 + llm-semantic-cache）， 来源 learn.microsoft.com。缓存输入约 1 折、省约 90% 为三大厂 2026-06 口径（登记于 Prompt-Engineering 模块事实表）。

05安全与护栏#gw-guardrail

所有对模型的请求都经过网关，所以鉴权、脱敏、审计、护栏在这一层配一次就对全公司生效 ——一处挂、处处生效，不必指望每个开发者都记得做安全。网关是 LLM 流量的安检口。

护栏挂成 pre/post 钩子：请求前（进模型前搜身）做 PII 脱敏、注入检测、敏感话题、 越权工具拦截；响应后（出模型后过检）做有害内容过滤、机密信息回收、格式与合规校验 ——命中就拦截、改写或告警。审计是网关的天然职能：「谁、什么时候、拿什么 prompt、调了哪个 模型、花了多少、命中什么护栏」只有网关能一处收全——这是 EU AI Act、NIST RMF 要的 「可追溯、可问责」证据链。

诚实的边界要主动讲：护栏不能 100% 挡住提示注入——业界共识是注入无法在模型内根治， 网关护栏是纵深防御的一道闸（拦掉大部分已知手法），要配合最小权限、人审关键动作一起用。 脱敏会有误报，规则要可配置、可审计、能回溯调优。网关自身也要按高价值资产防护：最小暴露面、 密钥库加密、定期红队——安检口本身也要有安保。与 Security 模块的分工：那边出 「威胁与对策清单」，网关做「能集中执行的那部分」；红队结论回来调网关护栏阈值。

06可观测#gw-observe

流量必经之路，就是最好的仪表盘：一处埋点就能看见「调了谁、多少 token、多少钱、多快、 命中缓存/护栏没有、走了主还是备」——不必在每个应用里各埋一套。

支柱通用含义AI 网关特有维度
日志 Logs每次请求的明细prompt/响应（可脱敏）、命中的护栏、路由决策
指标 Metrics聚合的量化趋势TPM/RPM、成本、缓存命中率、各模型延迟与错误率
追踪 Traces一次请求的全链路时序网关→路由→模型→（工具/MCP）的端到端 span

排查实例：「某应用今天变慢又变贵」——看网关面板发现语义缓存命中率掉了、fallback 频繁 触发（某家在限流），几分钟定位，不用翻十个服务的日志。标准正在收敛：OpenTelemetry GenAI 语义约定 / OpenInference 统一了模型名、token 数、成本等字段——网关按标准吐 数据就能接任意 APM（Grafana/Datadog/Langfuse），不锁定。两个提醒：留存日志要脱敏、分级、 限权（能看见 ≠ 明文全留）；可观测答「快不快贵不贵」，评估答「好不好」——网关把质量信号 纳入观测，就把两者连起来了（消费与行动层在 AI-Ops，trace id 两层贯通）。

07MCP 网关#gw-mcp

Agent 时代流量结构变了：一次任务里除了调模型，还要调大量工具（MCP Server）、甚至委派给 别的 Agent——这些流量同样需要统一接入、鉴权、限流、审计。Higress、Envoy、APISIX、 Azure APIM 都在把两者做进同一产品：「AI 网关 + MCP 网关」二合一是 2026 的分水岭。

MCP 网关＝把对「模型」的那套治理平移到「工具」上：统一代理、发现与注册（MCP 市场）、 鉴权与授权（最小权限）、限流与配额（防 Agent 失控狂调）、审计与观测（串进同一条 trace）、 安全护栏（工具描述投毒检测、危险动作拦截）。存量改造有捷径：openapi-to-mcp 读 OpenAPI 描述批量把现成 REST API 暴露成 MCP 工具——像给旧电器配智能插座，企业几百个 API 几乎零改造就能被 Agent 安全调用，汇成自建 MCP Marketplace。

MCP 授权：窄票替代万能钥匙

难点在「代理身份」：Agent 替用户调工具时，工具那头要知道真正的用户是谁。规范口径 （2025-11-25 修订版，OAuth 2.1 底座）：MCP Server 就是一台 OAuth 2.1 资源服务器， audience 校验（RFC 8707）与资源元数据（RFC 9728）是 MUST，PKCE 强制，token 透传 明令禁止。网关做 RFC 8693 token 交换：把用户的宽 token 现场换成「audience 限定、 最小权限、短时效」的窄票——像演唱会实名制门票，检票口只认本场本人。

授权反模式症状网关怎么破
Token 透传把用户的宽权限 token 原样转发给上游一律走 RFC 8693 换窄票，上游永远见不到原始 token
混淆代理工具只认网关/Agent 的高权服务身份，谁来调都同一个账号请求级身份传递 + act 委派链，权限判定以最终用户为准
万能钥匙给 Agent 发长期、全 scope 的静态凭证短时效 + 最小权限 + 虚拟密钥可吊销，爆炸半径钉死在「单工具、几分钟」

三合一终态：网关从「模型的统一入口」长成模型 + 工具（MCP）+ Agent（A2A）的中枢 控制面。协议机制在 MCP 网页版，Agent 侧的多模型/ 多工具调用在 Agent 网页版——本页讲的是把这些流量 统一治理的运营层。

授权规范核实 2026-07-10（来源 modelcontextprotocol.io spec / Red Hat Developer）；下一版规范预计 2026-07-28 发布，本节届时复核。 Envoy 的 MCP 授权、OIDC token 交换仍在路线图——是当前最受关注的能力缺口，也是售前要盯的 演进点。

08选型与问答#gw-cheatsheet

自托管还是托管

选自托管：要多云/信创、数据敏感要留在自己边界内、想深度定制、有平台团队扛可用性 （代表：LiteLLM、Higress、Envoy AI Gateway、APISIX、Kong）。选托管：想零运维快速 起步、已重度用某朵云、可用性交给云 SLA（代表：Azure APIM、Bedrock、Apigee、Cloudflare AI Gateway）。常见路径：托管起步验证价值 → 规模/合规上来后按需转自托管或混合。 选型三条筛：支持你要的模型与部署形态吗、原生做 token 治理与护栏吗、跟上 MCP 与 OTel 标准吗。

产品一句话定位亮点 / 注意（2026-07 口径）
LiteLLM最广泛采用的开源 LLM 代理100+ provider、虚拟密钥、7 维复杂度路由；Python 是性能天花板
Portkey开源+托管，主打护栏与缓存护栏、语义缓存、提示词管理；2026-03 完整开源
Kong AI GW老牌 API 网关的 LLM 插件已用 Kong 的团队天然加装；AI 高级插件需企业订阅
Envoy AI GWCNCF Envoy 血统，标准派v1.0（2026-06-23）；OTel 观测强；provider 覆盖仍在补
Higress阿里 AI-Native 网关国内生态全、MCP 能力最激进、含注入检测
APISIX云原生 API+AI 网关3.17.0（2026-06-15）支持 MCP 与 Agent 流量治理

三云对照一句话：Azure APIM 目前 LLM 网关能力最完整（原生 token 限流 + 语义缓存 + Unified Model API）；AWS 强在 Bedrock 生态（Multi-Provider GW 仅官方 Guidance，自部署）； 多云/边缘看 Cloudflare。国内/云原生看 Higress，标准派看 Envoy，最快上手 LiteLLM。 「开源 ≠ 免费全功能」——Kong 企业订阅这类报价点提前讲。

我已经有 API 网关了，为什么还要 AI 网关？2026-07-09

不是取代，是增强：多了按 token 限流与计费、多模型语义路由、语义缓存、注入/PII 护栏 ——普通网关没有的 LLM 六件套。很多产品（Kong、APISIX、Higress）就是同一网关的 「AI 模式」，平滑升级即可。

这题答成「换掉重建」就把客户推走了——正确姿势是在存量投资上加层， 迁移成本近零。

依据　路由与容灾 · 成本治理

自动路由到便宜模型，质量掉了怎么办？2026-07-09

路由是可控降级：关键场景设质量优先固定走旗舰，只对可容忍流量降档；再用评估持续监控 降档流量的质量，掉了就调阈值。省钱和质量之间有旋钮，不是赌博。

「降档流量谁来盯质量」是网关方案最常被挑战的点——答案是接评估： 网关采样、判官打分、超阈值告警。

下一步　给降档流量配 1~10% 采样评估（方法在 Evaluation 网页版），把质量线写进验收。

依据　可控降级 · Evaluation 网页版

语义缓存会不会给不同用户串答、泄露数据？2026-07-09

会，如果缓存键没带身份。正确做法：缓存键里带上租户/权限维度，个性化和带用户数据的 请求绕过缓存。这条既是成本设计也是安全设计。

阈值同理是双刃剑：太松答错（北京天气≈上海天气）、太紧白配—— 「宁可少命中，别答错」。

下一步　按落地清单四条过一遍缓存配置：分级阈值、绕过规则、 键带身份、命中率监控。

依据　语义缓存

护栏能 100% 挡住提示注入吗？2026-07-09

不能，也没有产品能这么承诺——业界共识是注入无法在模型内彻底根治。网关护栏是纵深 防御的关键一层，要配合最小权限、人审关键动作、模型外确定性策略一起用。

诚实讲边界反而更专业——「100% 防注入」的承诺本身就是方案不可信的 信号。

下一步　把「网关拦已知手法 + 权限控爆炸半径 + 人审兜底」 画成纵深防御图，逐层标注证据。

依据　安全与护栏

合规审计具体能提供什么证据？2026-07-09

网关能出完整调用链日志：身份、时间、prompt/响应（可脱敏留存）、模型、成本、命中的 护栏——正是 EU AI Act、NIST RMF 要的「可追溯、可问责」证据链。

审计日志只有网关能一处收全——这是「安全策略落在网关层」最硬的 理由，也是合规评审的入场券。

下一步　给合规团队演示一条完整 trace：谁调的、为何允许、 路由到哪、花了多少、质量怎样。

依据　审计 · 可观测

我们有几百个内部 API，要让 Agent 用，是不是要一个个改造？2026-07-10

不用。openapi-to-mcp 读 OpenAPI 描述批量把存量 API 转成 MCP 工具，再由网关统一托管 鉴权限流——存量资产几乎零改造就能被 Agent 安全调用。

「存量资产瞬间对 Agent 可用」是 MCP 网关最实的卖点：不是重写业务， 是给旧电器配智能插座。

下一步　挑一个内部 API 做 openapi-to-mcp 试点，挂上鉴权限流， 跑通「Agent 经网关安全调用」的完整链路。

依据　MCP 网关

Agent 替用户调工具，权限会不会被放大、被滥用？2026-07-10

这正是 MCP 授权要解决的：网关做 token 交换（RFC 8693），把用户真实身份换成 「audience 限定、最小权限、短时效」的窄票——Agent 手里永远没有万能钥匙。

三大反模式（透传/混淆代理/万能钥匙）都有规范级红线：token 透传是 MUST NOT——网关做身份传递有合规依据，不是自选动作。

下一步　对照反模式表审一遍现有 Agent 集成；规范下版 2026-07-28 发布后复核条款。

依据　MCP 授权 · src-mcp-auth

我们该自建还是买托管？2026-07-12

小团队/快验证先用云托管零运维起步；多云、信创、数据敏感或要深度定制，上自托管 （LiteLLM/Higress/Envoy）。绝大多数客户是「托管起步 → 按需转自托管/混合」， 不用一步到位。

自托管网关本身会成为单点：要多副本、健康检查、多区域——没有平台 团队就别扛，这是路线选择的硬约束。

下一步　按三条筛（模型覆盖/token 治理/标准跟进）列两条路线的 对照表，标注客户现状的匹配项。

依据　选型

一句话，AI 网关到底给我带来什么？2026-07-09

三件事：省钱（路由+缓存+预算）、稳（多模型容灾）、可控（护栏+审计+观测）。 它把散落在每个应用里的接入与治理收成一层——换模型像改配置、管成本像看电表、 上安全像过安检。

第四个字是「长」：网关正从模型入口长成模型+MCP+Agent 的三合一中枢 ——现在选型要把这条演进线算进去。

下一步　给客户画「应用 → 网关 → 三类后端（模型/工具/Agent）」 的一页架构图——这张图是网关售前的开场白。

依据　定位 · 三合一
