## 建议一：SKILL.md 第 4 步补上「禁止静默覆盖」——**不是补新规则，是补主流程的漏引**

### 先纠正一个判断

自省最初写的是「`module-template.md` 没规定同日重名要加后缀，建议补规则」。**这是错的。**
复核发现该规则**早已存在**，就在 `references/module-template.md`「raw-data/history/ 成品历史
版本留痕」节：

> - **重名保护（2026-07-13 修订）**：写入存档前**必须先检查目标文件名是否已存在**，存在就顺延
>   后缀，**禁止用会静默覆盖的复制命令直接写入**（脚本里应"存在即报错"）。教训：2026-07-12
>   曾因 cp 静默覆盖丢失一份同日中间版存档。

也就是说：**2026-07-12 已经因为同一个原因丢过一次档，规则已经写好了，我今天又原样犯了一遍。**
所以这不是规则空白，是规则没被读到。

### 真正的缺陷在哪

`SKILL.md` 第 4 步（第 129 行附近）现文：

```
把旧版原样复制到该模块 `raw-data/history/`（命名 `YYYY-MM-DD-<原文件名>`，规范见
`references/module-template.md`）。
```

问题：**它复述了细则里的命名格式，却漏了细则里最关键的安全条款。**执行者（我）看到主流程
已经给出命名格式，就认为信息足够、不必再去加载 `module-template.md`，于是直接 `cp` 写入。
主流程「给一半」比「一点不给」更危险——给一半会让人以为拿全了。

### 改法（二选一，建议选 B）

- **A（最小改动）**：删掉主流程里的命名格式，只留「规范见 `references/module-template.md`」，
  强制执行者去读细则。
- **B（推荐）**：把安全条款一并写进主流程，让主流程自身可安全执行：

```
把旧版原样复制到该模块 `raw-data/history/`（命名 `YYYY-MM-DD-<原文件名>`；**同日多次改写
必须先查重名、顺延 b/c 后缀，禁止用 cp 等会静默覆盖的方式写入**——2026-07-12、2026-07-20
两次丢档均由此而来；完整规范见 `references/module-template.md`）。
```

推荐 B 的理由：留痕是**每次 A/B 类任务都会走**的动作，且失败后果不可逆（丢的是回滚能力）。
这种动作的安全条款应该在主流程就是自足的，不能依赖"执行者会去加载细则"。

### 配套（可选但建议）

`module-template.md` 那条规则已经说了"脚本里应存在即报错"，但库内没有对应工具，每次都是临场
手写 `cp`。建议在 `scripts/` 加一个十几行的 `kb_archive.py`：给定模块与文件，自动查重名、
顺延后缀、拒绝覆盖、打印落档路径。**规则靠自觉执行了两次都失败，第三次应该交给工具。**

## 建议二：把「绝不整包重存」的铁律扩面到「绝不用 XML 库解析回写 OOXML」+ 新增 audit 检查项 18

### 这条经复核成立，是真空白

`references/ppt-design-system.md`「改成品仍優先 zip 级最小手术」条现文只针对 **python-pptx
整包 round-trip**，论证的是"会重排 slide 文件名与 rId、放大 diff 面"，并且明确写了 round-trip
**「不会修复也不会引入」**顺序违规。整条规则**完全没有覆盖**「用 XML 库解析单张 slide 再回写」
这个变体，而它的失效模式和后果都不一样。

### 失效模式（本次实测）

用 `xml.etree.ElementTree` 解析 slide XML、改完 `ET.tostring` 写回，ET 会把 OOXML 的命名空间
前缀**重写**：

```
改前：<p:sld xmlns:a="..." xmlns:p="...">   ...   <a:t>文字</a:t>
改后：<ns0:sld xmlns:ns0="..." xmlns:ns1="..."> ... <ns1:t>文字</ns1:t>
```

XML 依然**良构**，命名空间语义也**等价**，所以：
- **检查项 17（XML 良构性）照样放行**——它只 `ET.fromstring` 试解析；
- 但 audit 其余检查全是**正则式**、按 `a:`/`p:` 前缀匹配，于是 **8 册全部 FAIL**。

这与 2026-07-17 章眉统一事故（正则双闭合 → 非良构、audit 照过、渲染丢页）是**同族但不同因**
的第二种"字节级手术把成品改坏"路径：那次是良构性失守，这次是**前缀约定**失守。

### 改法一：`ppt-design-system.md` 改写该条铁律

在「改成品仍優先 zip 级最小手术」条内补一段（或独立成条）：

> **纯文本改写一律字节级，绝不用 XML 库解析回写。** 改字这类只动文本内容的手术，必须在
> slide XML 的**字节/字符串层**做 `str.replace`，并满足两条：①替换串在该部件内**命中数
> 恰好为 1**（多于 1 说明定位不唯一，中止而非猜）；②改完只 `ET.fromstring` **解析校验良构，
> 绝不 `ET.tostring` 回写**。原因：ElementTree 等 XML 库回写会把 `a:`/`p:` 命名空间前缀
> 重写成 `ns0:`/`ns1:`——XML 良构、语义等价、检查项 17 放行，但违反库内 OOXML 前缀约定，
> audit 其余正则检查全线 FAIL（2026-07-20 实测 8 册全挂）。**「绝不整包重存」要读成
> 「绝不让任何 XML 库产出最终字节」**，不只是不用 python-pptx。

### 改法二：`scripts/audit_pptx.py` 新增检查项 18（FAIL 级）

紧跟检查项 17 之后（约第 202 行后），复用已取到的 `slides` 列表：

```python
# 检查项 18(v3.16, FAIL 级): slide XML 命名空间前缀约定——本审计其余检查全是按
# a:/p: 前缀的正则式, XML 库(如 ET.tostring)回写会把前缀重写成 ns0:/ns1:,
# XML 良构、语义等价、检查项 17 放行, 但正则检查全线失明(2026-07-20 实测 8 册全挂)。
for _pos, _name in enumerate(slides, start=1):
    _head = z.read(_name)[:400].decode("utf-8", "ignore")
    if "<p:sld" not in _head:
        errors.append(
            f"第 {_pos} 页根元素前缀非 p:（疑似被 XML 库回写成 ns0:/ns1:）"
            "——纯文本改写须走字节级 str.replace, 不得 ET.tostring 回写"
        )
```

放在检查项 17 之后是有意的：**17 管良构、18 管前缀约定**，两者都是"字节级手术的兜底门禁"，
挨着放便于将来一起维护。

## 建议三：反 AI 腔节补「先做征兆密度体检，再定范围」（措辞级，并入 v3.15 修订）

### 这条成立，但不是补漏洞，是补顺序

`references/ppt-rules.md`「反 AI 腔写作纪律」的**存量口径**已经写对了：

> 既有成品不因本节自动回刷，本节对新增与今后的改写生效；若认为某册/某份材料特别值得单独
> 立项打磨，列清单交用户拍板，不自行动手。

缺的不是"要不要拍板"，而是**拍板所依据的信息怎么来**。本次用户一度选了"19 册全面立项打磨"，
若直接照办，就是对 **6109 个成句段落、约 35.7 万字**盲刷一遍——而实测全库征兆只有 **47 处 /
1732 页**，其中 29 处「不是…而是」经逐条读上下文判定**全是实打实的区分**（"量化不是『开不开』
而是『开到几档』"、"推理模型不是『更好的模型』而是『更贵的档位』"），属于全篇最锋利的句子，
改掉纯属损失；行政腔命中也多为正当技术用语（落实 / 推进中 / 确保调用）。真正该改的只有
11 处「抓手」这类**踩用户语感锚点**的词。

### 建议补写（接在「存量口径」那段之后）

> **打磨类任务动手前必做「征兆密度体检」。** 用户点名要对存量做文案打磨（无论范围多大）时，
> **先量后改**：①按判定口径扫出全库/目标册的征兆命中，给出**密度**（命中数 / 总页数或总字数）；
> ②抽一册**通读**，把命中逐条放回上下文判真伪——判定口径里的"制造对立"指的是**虚假二分**，
> 不含承载真实区分的对照句式，后者是讲义最有效的表达之一，**误伤即质量倒退**；③拿这两份数据
> 向用户复述范围建议，再动手。**密度低就如实说"不值得整体重写"并给定点清单，即使用户已经
> 选了大范围**——用户选范围时手里没有密度数据，给出数据、说明理由才是本技能该做的事，
> 照单盲刷不是。

### 附带的一条判断先例（可写进 CHANGELOG，不必进规则正文）

v3.14「四类不展开」遇到**存量已展开**的专名（本次实例：`NCCL，NVIDIA Collective
Communications Library`、`CLIP，Contrastive Language-Image Pre-training`）时，**不回改**。
理由：v3.14 原文是"**不强制**展开"，不是"禁止展开"；已经展开的对读者只有增益，为合规而删
信息是负收益。判据：**收紧类规则只约束今后动作，不追溯删除存量里无害的多余信息。**
