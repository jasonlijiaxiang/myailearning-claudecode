# 网页版可读性审查与整改 · 设计（2026-07-22）

> 状态：**已批准**（2026-07-22）。触发：用户翻 AI-Infra-Platform 的容错章与云上形态章，指出
> 「都是文字罗列，没有交互、没有图片、没有 flowchart……这是一个知识库网页，
> 一定要发挥网页版的优势」。

## 一、实测：三个问题，一个根因

### P1 · 源码换行落在中文之间，渲染出可见空格（全库 1716 处）

用户看到的「重算 几小时」「自愈的 差距」「goodput 差距可达 一成以上」不是错别字，
是 HTML 把源码里的换行 + 缩进折叠成**一个空格**，而这个空格夹在两个汉字中间会被真的画出来。

浏览器实测：正常两个汉字占 29px，跨源码换行处占 **33px**——每处多出约 4px。

**16 册一张图、一个卡片都没有。** 而且退化是**按批次单调的**：样板册 5 图 17 卡 →
第二三批 1 图 4~11 卡 → 第四批以后 0 图 0 卡。

拿用户截图那段做样本，一个 `<p>` 里塞了**四个不同性质的知识对象**：

| 段内成分 | 性质 | 它本来该长什么样 |
| --- | --- | --- |
| checkpoint 存太勤 vs 存太疏，破局＝异步＋分级 | 取舍 + 解法 | 对比表，或「暂停时间/落盘时间解耦」示意图 |
| 自愈循环四步：检测→驱逐→替换→续训 | 有序过程 | 流程图 |
| 人工救火几十分钟~几小时 vs 自动化分钟级 | 量化对比 | 两行小表 |
| 托管容错是「该不该上云」的关键论据 | 销售论点 | 结论卡 |

四样东西挤成一段 516 字的散文，读者得自己在脑子里把它拆开——**这正是网页该替他做的事**。

### P3 · 现有 7 张 SVG 写死浅色，深色模式下是亮块

MCP 的图都以 `<rect width="720" height="250" fill="#FFFFFF"/>` 开头、文字写死 `#0F172A`。
它们建于深色模式之前。我在批三给 `svg` 加的 `filter: brightness(.92)` 只是把纯白压成
`#EBEBEB`——在 `#0F1520` 的页面上仍是一块亮斑。**这是批三我漏检的一处**：当时目检了
表格、卡片、页头，没翻到有图的那一屏。

### 根因：收敛判据只量内容，从不量呈现

八批扩建的封版标准是「事实级 0 + 缺口级 0」——那是**内容缺口**的度量。呈现规格
（图/表/卡/段长）**从来没有被量过**，于是 19 册全部报「已封版」，而规格从 5 图 17 卡
一路退到 0 图 0 卡，八道门禁一道都查不出。

这与之前两次是同族教训：**页数账会漂、语义会过期、呈现规格也会退——没被量的东西就会漂。**

## 二、方案：止血 → 定规 → 回刷

### A. 排版止血（机械，可脚本化，全库一次做完）

写一个格式化器：把落在中文之间的源码换行**移到安全位置**（标点之后、中英文边界处），
而不是把段落拉成一整行——源码要保持可读、diff 要保持可看。
处理不了的位置（整段无标点可断）改用 `<!--\n-->` 吞掉空白。

配门禁 `check_cjk_spacing.py`：命中即 FAIL，防回潮。

### B. 呈现体检：**报告型，不设配额**（用户 2026-07-22 裁决）

原稿把三条写成「下限」，用户当场纠正：**「非散文承载」的数量要按具体需求来，
不能是一个硬性数字。** 这条实质是把本库已有的**「不设配额」铁律**（`web-rules` §七
现在只写在问答轴上：「不逐章机械配题，不同模块不追求相同题数」）**扩展到呈现轴**——
同一条原则，此前只管了问答，没管图表。

所以三条是**判据**，不是配额；脚本是**体检报告**，不是硬门禁：

1. **一章通篇散文 = 值得看一眼的信号**，不是错误。报告列出该章的承载构成
   （表/图/卡/交互件各几个），由执行者判断这章的内容性质要不要换承载。
   依据：一章全是散文，等于把讲义原样译成 HTML，换个介质也成立——`web-rules` §六·六
   把这类列为优先级最低。
2. **超长段落列清单**（默认阈值 260 字，只报不判）。超长通常不是话多，是**一段里塞了
   多个知识对象**（上面那张四对象拆解表就是证据）。拆不拆、拆成什么，看内容。
3. **每册的图数列出来**，不设下限。19 册里 16 册一张没有——这个事实本身值得摆在台面上，
   但「该有几张」由内容决定。

唯一硬 FAIL 的是 P1 排版——那是确定性错误，与内容无关。

### C. 按批回刷 16 册呈现（**只换承载，一字不改事实**——用户 2026-07-22 裁决）

写作尺度定死：把长段里的知识对象抽出来放进表/图/卡，**文字原样搬运**，只做必要的断句与
去重；不新增、不改写任何事实与判断。这样每一处改动都能逐条对照原文核验，巡检时一眼看得出
没夹带私货。

改的是**呈现形态**——web-rules §六·六 明确把它列在「不回流 PPT」那一侧，
所以这一轮只动网页面，讲义不动。四种承载按内容性质对号入座：

| 内容性质 | 换成 | 本库现成的例子 |
| --- | --- | --- |
| 有序过程 | 流程图 | 自愈四步、RAG 四段管线、会议五段链路 |
| 取舍 / 对比 | 对比表或双栏卡 | checkpoint 两难、自建 vs 托管、四种云上形态 |
| 并列要点 | 卡组 | 弹性扩缩三要点、数据就绪四问、运营包五件套 |
| 有公式可算 | 接进已有交互件 | 显存账、成本账 |

同批把 **SVG 家法改成令牌驱动**（`currentColor` + CSS 变量替代写死色值），
顺手修好 P3 的 7 张老图——以后新画的图自动跟随深浅色。

**分批**：按「超长段数 × 缺图」排序，16 册分 4 批，每批 4 册。
每批收尾走 web-rules §九·五 九步，并跑新门禁复量。

## 三、明确不做

| 不做 | 理由 |
| --- | --- |
| 为凑指标硬塞图 | 判据是「这段内容本来是什么性质」，不是「每章一张图」 |
| 改讲义 PPT | 呈现形态不回流，这是 §六·六 的挡板；讲义有自己的节奏 |
| 重写内容 / 补新事实 | 本轮只换承载。要补事实是另一件事，走 B 类 |
| 把长段简单切成两段 | 切开还是散文。该拆的是**知识对象**，不是字数 |
