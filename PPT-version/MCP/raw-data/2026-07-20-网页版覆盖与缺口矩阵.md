## 三、深度自查：八项覆盖与真实缺口

按 web-rules 第三节八项逐条对照讲义现状：

| 深度项 | 讲义覆盖 | 判定 |
| --- | --- | --- |
| 1 定义与责任 | p5、p16 | 够 |
| 2 输入与输出 | p25 报文 | 够 |
| 3 工作机制与因果 | p18、p19、p32 | 够 |
| 4 生命周期与关键组件 | p19/p20 时序、p17 组件 | 够 |
| 5 失败方式、适用边界与**替代方案** | 第 6 章讲失败充分；但"何时不该用 MCP、替代路径是什么"只在 p12 对练答一句 | **缺口 A** |
| 6 会改变架构/选型/风险/成本/验收的判断 | p31 选型、p54 上云、p73 验收；工作量有口径（几人天），**运行期成本没有** | **缺口 B** |
| 7 实际应用与客户价值 | p45、p54、全册对练 | 够 |
| 8 验证方法与证据 | p73、p74 | 够 |

**缺口 A · 边界与替代方案。** 讲义把"什么时候值得上 MCP"当判断题散在对练和动手做里，没有正面展开"不上 MCP 的三条替代路径分别适合什么场景"。网页面适合独立成节——这正是 web-rules 说的"会改变方案的问题才立题"。此节不需要新的联网事实，是对既有判据的重组与展开。

**缺口 B · 运行期成本。** 讲义给了开发工作量口径，但工具描述占用上下文、多 server 并存时的 token 开销、网关一跳的延迟，这些是售前现场真实会被问到、且讲义答不出的。**补它需要先联网核实**，按工作流第 2 步走；核不到可靠数字就不写，不自造。建议列为样板册定稿后的第一个增补项，不塞进本次试点。

## 四、本册交互件（web-rules 第九节）

**报文逐字段注释器**（落点 p25）。给出一次 `tools/call` 的真实往返，逐字段可点：点 `id` 说明它怎么配对一问一答、点 `method` 链到三大原语、点 `params.arguments` 说明工具描述如何决定模型传什么参、点 `isError` 说明错误语义。

增益判据自查（第九节）：它帮助**理解**（把 JSON 从"一坨"变成有结构的对象）和**追证**（每个字段链到讲它的那一节）。不是装饰性展开——讲义 p25 的"解读"一段本来就在做这件事，只是纸面上只能一次讲完，网页可以按读者的手指走。

无 JS 时的降级：字段注释以静态列表形式全部呈现在报文下方，阅读路径不断（design-system 可访问性底线）。

MCP 有一条核心事实在 **8 天后翻篇**——2026-07-28 新版规范正式发布（无状态核心、Tasks 转正式扩展、Roots/Sampling/Logging 进入弃用）。MANIFEST 已登记建议复查日 2026-07-28。

对试点的两个影响：

- 样板册定稿后一周内必须回刷，涉及 p19/p20 生命周期、p33 五版演进、p34 无状态核心、p70 版本口径。
- 反过来看，这是检验网页面"动态事实 + 复查日"机制的第一个真实用例：网页上这几处要带显式的"2026-07-28 待复查"标记，而不是把日期埋在正文里。做对了，07-28 那天该改哪几处一目了然。
