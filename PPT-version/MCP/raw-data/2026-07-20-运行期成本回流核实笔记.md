## 三个量级（均 2026-07-20 联网核实）

| 数字 | 说明 | 来源 | 条件与不可外推边界 |
| --- | --- | --- | --- |
| **17.6k tokens** | GitHub 官方 MCP server 暴露 94 个工具，仅工具定义占约 17,600 tokens | Atlassian 工程博客《MCP Compression: Preventing Tool Bloat in AI Agents》2026-03-29 | 对照表未公布模型与计数方法；单个大 server 量级，不能外推成"每个 server 都这么贵" |
| **~3.9k → ~0.5k** | 同样 94 工具，描述压缩后：低压缩约 3,900、最激进约 500 | 同上（同源同场景） | 压缩牺牲描述细节换上下文，而描述质量直接影响模型会不会误调，不是免费 |
| **1–2ms / 会话** | MCP 流量经网关一跳的额外延迟（加密迭代调至约 100 次后） | Envoy AI Gateway 官方基准《MCP in Envoy AI Gateway》2025-12-08 | MacBook Pro M1 八核、简单 echo 工具、按会话计非按调用；默认 10 万次 KDF 迭代下为数十毫秒；单机基准，不能当生产 SLA |

## 关键机制（放结论位）

工具定义**随每条消息进上下文，不是每会话一次**——所以成本按轮次乘。这是「贵在上下文、不在调用」的由来，也是三条判断的根：
- 别把工具堆进一个 server（工具数直接乘进每轮上下文）；
- 网关那一跳通常不是瓶颈（调优后 1–2ms 相对推理可忽略）；
- 选型先问"暴露多少工具"，比"支持哪些功能"更能预测成本。

客户端侧已有缓解（工具定义超过上下文一定比例时改按需检索加载），但各家实现与阈值不同，不写进讲义口径。
