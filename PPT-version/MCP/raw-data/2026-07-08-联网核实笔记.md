## 协议版本与路线图（2026-07-08 现状）

- **当前稳定版规范：2025-11-25**（https://modelcontextprotocol.io/specification/2025-11-25）
- **下一版 2026-07-28：RC 已于 2026-05-21 锁定，正式版 2026-07-28 发布（本月底！）**，十周验证窗口给 SDK 维护者。
  来源：https://blog.modelcontextprotocol.io/posts/2026-07-28-release-candidate/
  核心变化（讲 PPT 时必提，售前谈"协议演进方向"的最佳素材）：
  - **无状态协议核心（stateless core）**：删除 `initialize`/`initialized` 握手与 `Mcp-Session-Id`
    （SEP-2567、SEP-2575），协议版本与 client info 改为随每个请求的 `_meta` 携带
    → 任何请求可打到任何实例，普通轮询负载均衡即可横向扩展，无需 sticky session。
  - **Extensions 扩展框架**（SEP-2133）：反向 DNS 标识、capability map 协商、独立 `ext-*` 仓库独立版本化。
  - **Tasks 长任务**从实验特性转正为官方扩展（SEP-2663）：task handle + `tasks/get`/`tasks/update`/`tasks/cancel`；
    旧 `tasks/list` 移除（用了 2025-11-25 实验版 Tasks 的要迁移）。
  - **MCP Apps**（SEP-1865）：server 可下发交互式 HTML UI，沙箱 iframe 渲染，UI 与工具调用走同一 JSON-RPC。
  - **授权加固六连**：对齐 OAuth 2.0 / OIDC——client 必须校验 `iss`（SEP-2468）、注册时声明
    `application_type`（SEP-837）、凭据绑定签发方（SEP-2352）、refresh token 与 `.well-known` 发现澄清
    （SEP-2207/2351/2350）。
  - **正式弃用政策**（SEP-2577）：Active → Deprecated → Removed，两阶段之间至少 12 个月。
    首批弃用三件套：**Roots**（改用工具参数/资源 URI/服务器配置）、**Sampling**（改用直连 LLM API）、
    **Logging**（改用 stderr 或 OpenTelemetry）。
  - 其它：`Mcp-Method`/`Mcp-Name` 头做免解析路由（SEP-2243）、列表响应带 `ttlMs`/`cacheScope` 缓存
    （SEP-2549）、W3C Trace Context / OTel 传播（SEP-414）、工具 schema 全量支持 JSON Schema 2020-12
    （SEP-2106）、资源缺失错误码 -32002 → -32602（SEP-2164）。
- **历史版本脉络**（讲协议演进用）：2024-11-05 首发 → 2025-03-26（OAuth 授权、Streamable HTTP 取代
  HTTP+SSE）→ 2025-06-18（Elicitation、structured output、OAuth Resource Server 化）→ 2025-11-25（当前
  稳定版）→ 2026-07-28（无状态核心）。
- **治理**：2025-12 Anthropic 将 MCP 捐给 **AAIF（Agentic AI Foundation）**，Linux Foundation 旗下
  directed fund，Anthropic、Block、OpenAI 共同发起。变更走 **SEP**（Specification Enhancement Proposal）
  机制；Standards Track SEP 需在 conformance suite 有匹配场景才能到 Final。
  来源：https://aaif.io/blog/mcp-is-growing-up/

## SDK 现状（2026-07-08）

- 官方 SDK 分 **Tier 制**（按 conformance suite 打分）；Tier 1 SDK 须在十周验证窗口内跟上新规范。
- 面向 2026-07-28 规范的 **beta 已发**（来源：https://blog.modelcontextprotocol.io/posts/sdk-betas-2026-07-28/）：
  - **Python v2.0.0b1**（`mcp[cli]==2.0.0b1`）——装饰器 API 保留，`FastMCP` 更名 `MCPServer`，有官方迁移指南。
  - **TypeScript v2 beta**——单包拆分为多个聚焦包（如 `@modelcontextprotocol/server@beta`），
    官方 codemod：`npx @modelcontextprotocol/codemod@beta v1-to-v2 .`
  - **Go v1.7.0-pre.1、C# v2.0.0-preview.1**——不拆包、API 基本不变，迁移更平缓。
  - 所有 beta **同时支持新旧协议版本**（2026-07-28 与 2025-11-25）——讲"平滑迁移"的好素材。
- 讲义实操若求稳：Python SDK v1.x 稳定版（FastMCP API）；若讲前沿：v2 beta 的 MCPServer。

## 官方 Registry（2026-07-08 现状）

- **仍处 preview，未 GA**（官方页面明示可能有 breaking change / 数据重置）。
  来源：https://modelcontextprotocol.io/registry/about 、https://registry.modelcontextprotocol.io/
- 定位：官方**元数据仓库**（不托管代码，指向 npm/PyPI/Docker Hub 等包仓库）；
  命名反向 DNS（`io.github.user/server`），经 GitHub/DNS/HTTP 挑战验证属主。
- 设计上供**下游聚合器**（marketplace）消费，host 应用不直接消费；
  **不支持私有 server**——企业私有场景要自建 registry（可实现同一 OpenAPI 接口）→ 售前常见问题点。
- 安全扫描委托给底层包仓库与下游聚合器，Registry 本身只管命名认证 + 元数据。

## 安全事件与 CVE（售前必知，2026-07 现状）

- **体量**：2026 年 1–4 月披露 **40+ CVE**，横跨 Python/TS/Java/Rust SDK 与参考 server，
  受影响组件合计下载量约 1.5 亿。来源：https://dev.to/piiiico/mcp-security-vulnerabilities-in-2026-40-cves-and-counting-4pco
- **标志性案例**：
  - **CVE-2025-6514**（mcp-remote，43.7 万+ 下载）：恶意 MCP 端点经构造的 authorization URL 达成 RCE。
  - **MCPoison CVE-2025-54136 / CurXecute CVE-2025-54135**（Cursor 相关）：工具投毒/配置篡改类。
  - **mcp-server-git 三连**：CVE-2025-68145（路径校验绕过）+ CVE-2025-68143（不受限 git_init）+
    CVE-2025-68144（参数注入），可链式利用——官方参考实现也会出洞。
  - **工具投毒（Tool Poisoning）**：恶意指令藏在工具描述里，连上即注入（先于任何调用/审批）；
    MCPTox 基准（arXiv 2508.14925）实测自动批准场景成功率 **84.2%**。
  - Trend Micro：公网**裸奔 MCP server**（无鉴权）问题蔓延到云端。
- **权威指引**：NSA/CISA 等《MCP Security Design》PDF（2026-06）：
  https://media.defense.gov/2026/Jun/02/2003943289/-1/-1/0/CSI_MCP_SECURITY.PDF
- 防护要点（与 Agent 模块第 6 章共识一致）：最小权限、工具清单签名/固定（tool pinning）、
  人工审批高危操作、网关/代理集中管控（MCP gateway）、别把 server 裸暴露公网。
