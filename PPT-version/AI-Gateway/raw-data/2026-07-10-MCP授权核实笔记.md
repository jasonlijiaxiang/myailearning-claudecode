- **核实日期**：2026-07-10
- **用途**：第 7 章增补 3 页（授权时序图 / 规范现状 / 三大反模式）的时效性事实以此为准。

## 1. MCP 授权规范现行口径（2026-07-10 核实）

- **现行规范修订版为 2025-11-25**（modelcontextprotocol.io/specification/2025-11-25/basic/authorization）。授权模型自 **2025-06-18 修订**起以 **OAuth 2.1** 为基础，2025-11-25 版延续并强化。下一版规范预计 2026-07-28 发布（届时随 MCP 模块一起定点复查）。
- 核心规定（MUST 级）：
  - **MCP Server = OAuth 2.1 资源服务器（Resource Server）**：必须按 OAuth 2.1 §5.2 校验 access token，且必须校验 token 是"专门发给自己"的（audience 校验，依据 RFC 8707 §2）。
  - **RFC 9728 受保护资源元数据（Protected Resource Metadata）**：MCP Server 靠它告诉客户端"去哪家授权服务器拿票"；配 RFC 8414 授权服务器元数据发现。
  - **RFC 8707 资源指示器（Resource Indicators）**：MCP 客户端 MUST 带 `resource` 参数，把 token 绑定到目标 MCP Server，防一票多用。
  - **PKCE 对所有客户端强制**（S256）。
- **明令禁止的做法**：token 透传（passthrough）——MCP Server 不得把从客户端收到的 token 原样转发给上游 API；另禁 implicit flow、ROPC 密码授权、URI query 里带 bearer token、可用 S256 时用 plain PKCE。

## 2. 网关侧身份传递：RFC 8693 Token 交换（2026-07-10 核实）

- 主流落地模式：**身份网关**校验入站 token → 按细粒度策略判断"这次工具调用允不允许" → 通过 **RFC 8693 Token Exchange** 铸造下游委派凭证（audience 限定、scope 收窄、短时效）再发给 MCP Server。
- **act 委派链**：Agent A1 交接给 A2 时，A2 的 token 经 RFC 8693 以 A1 的 token 为 actor_token 换出，JWT 携带 act 链——MCP Server 能验证"用户 U → Agent A1 → Agent A2 → 本次调用"的完整链路，审计同时看到委派人与执行者。
- 生态实证：Red Hat（MCP Gateway 高级鉴权，2025-12）；Azure APIM 策略同时支持 RFC 8693 与 Microsoft Entra On-Behalf-Of；MuleSoft Flex Gateway 用 RFC 8693 做服务间认证；Maverics AI Identity Gateway 用 RFC 8693 委派 token + OPA 策略逐次评估 MCP 工具调用。
- 与既有事实衔接：Envoy AI Gateway 路线图中的"MCP 授权、OIDC token 交换"正是这套模式的标准化（见 2026-07-09 核实笔记）。
