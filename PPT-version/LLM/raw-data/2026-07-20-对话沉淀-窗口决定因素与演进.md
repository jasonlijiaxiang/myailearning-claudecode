## 对话要点（原始口径）

### 1. 窗口大小由什么决定——硬件排最后
- 同一批 GPU 既跑 128K 也跑 1M 的模型；窗口不是"插了多少显存"决定的。
- 三因素链：
  ①**训练配方**（能不能）：位置编码外推（RoPE + YaRN 等）只是入场券，瓶颈是长程依赖训练数据天然稀缺（整本书才 100–300K token），合成数据不像真实分布就学成应试技巧；长上下文训练阶段因平方级计算很贵，需序列并行。
  ②**架构**（贵不贵）：每 token 的 KV 成本差一到两个数量级——MHA → GQA/MQA → MLA → 滑动窗口/稀疏/线性混合（细节指向本册第 5 章，不重复展开）。
  ③**服务工程**（扛不扛得住）：HBM 容量/带宽、卡间互联、分层存储、chunked prefill（系统细节在 LLM-Inference 册）。
- 落点：几百 K 的模型不是显存不够，是没花那笔训练预算或质量不达标。

### 2. 2M/5M 会来吗——四堵墙与主战场转移
- 结论：会有，且已出现（本册 MANIFEST 既有核实：13 家托管模型 ≥1M；Gemini 3.1 Pro 2M；Llama 4 Scout 标称 10M）。
- 四堵墙：①训练（长程数据稀缺 + 平方级训练成本）②架构（O(n²)；稀疏/线性方案是有损压缩，省算力换长程精确回忆——本册既有 DSA/线性混合口径）③服务（KV 巨量、HBM 带宽墙决定 decode 速度、无缓存则超长 prefill 分钟级 → 超长交互离不开前缀缓存）④质量（**标称 ≠ 有效**：本册既有核实——RULER/MRCR v2/NoLiMa 显示多事实检索过 200K 普遍掉 30–60 分；NIAH 饱和但只测检索不测推理）。
- 最强拉长驱动：视频多模态（1 小时 ≈ 百万 token 量级——待核实项，见下）。
- 主战场转移：从"标称长度竞赛"→ 有效长度 + 单位 token 经济性 + 记忆分层（compaction/外置记忆/检索——对应 Agent 记忆章、RAG）。
- 对练题（补进第 4 章对练页）：
  Q1"为什么友商 1M、你们 200K？" A 比的不是硬件，是训练配方与注意力架构；反问有效长度与超长计费。
  Q2"以后有 5M，我等等再建 RAG？" A 成本曲线永远偏爱只装需要的东西；"够用窗口 + 缓存 + 检索/记忆分层"在 5M 时代依然成立（接本册"1M 窗口不是 RAG 终结者"既有串联线）。
