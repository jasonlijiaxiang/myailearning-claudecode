## 方法现状（PEFT / 全参）

- LoRA 仍是 PEFT 主导方法；QLoRA（4-bit NF4 量化底座 + 分页优化器）把可微调门槛降到单张消费级显卡。变体：DoRA（ICML 2024）、AdaLoRA/DyLoRA 等动态秩一族。
- 显存账参考：7B 全参微调约需 100–120 GB 显存（多卡 H100 量级）；同模型 QLoRA 单张 RTX 4090（约 $1,500）可跑。PEFT 省显存 10–20×，保留约 90–95% 质量。
- 2026 年风向：RFT（Reinforcement Fine-Tuning，可验证奖励 + 可编程 grader）成为推理/数学/代码类微调的新主流路径，从"模仿参考答案"转向"奖励可验证的正确结果"。
- 来源：<https://www.meta-intelligence.tech/en/insight-lora-finetuning>、<https://effloow.com/articles/llm-fine-tuning-lora-qlora-guide-2026>、<https://aidevdayindia.org/blogs/fine-tuning-llms-lora-qlora/fine-tuning-llms-lora-qlora.html>

## 开源训练框架格局（2026-07，GitHub stars 为当时数）

- 功能已趋同：四大框架都支持 LoRA/QLoRA/全参/GRPO/DPO/视觉模型，差异在工作流。
- LLaMA-Factory ~68.4K★：YAML CLI + Gradio Web UI（LlamaBoard），上手门槛最低，Amazon/NVIDIA/阿里云生产采用。
- Unsloth ~53.9K★：单卡速度王，自研 Triton kernel，宣称比 HF+FA2 快 2×、省 70% 显存；3GB 显存可玩，Colab/Kaggle 免费笔记本生态强；已出 Unsloth Studio Web UI。
- TRL ~17.6K★：Hugging Face 官方库（SFT/DPO/RLHF），"机构派"选择，可控性最强。
- Axolotl ~11.4K★：YAML 配置驱动、开箱即用 FSDP/DeepSpeed，多卡生产训练的稳重之选。
- 经验法则：单卡用 Unsloth，多卡生产用 Axolotl，要底层控制用 TRL，要 Web UI 用 LLaMA-Factory。
- 来源：<https://theaiengineer.substack.com/p/unsloth-vs-axolotl-vs-llama-factory>、<https://dev.to/ultraduneai/eval-003-fine-tuning-in-2026-axolotl-vs-unsloth-vs-trl-vs-llama-factory-2ohg>、<https://www.spheron.network/blog/axolotl-vs-unsloth-vs-torchtune/>、<https://github.com/unslothai/unsloth>

## 托管微调服务格局（2026-07）

- OpenAI 微调 API：三种方法并存——SFT、DPO、RFT。GPT-4.1 / GPT-4.1-mini 支持 SFT+DPO；o4-mini 支持 RFT（可编程 grader）。
- Anthropic：不通过自家公开 API 提供微调；唯一托管路径是 Amazon Bedrock 上的 Claude 3 Haiku SFT（us-west-2）。官方建议 50–10,000 条样本，窄分类任务 200–500 条见效。
- Google Vertex AI：三大云里微调最灵活（全参/LoRA/adapter tuning，Gemini 家族 + 第三方模型），微调-存储-鉴权-部署一体。
- Azure AI Foundry：OpenAI 系模型微调最完整；模型蒸馏功能 2026 GA（用 GPT-4o 教师产出微调 Phi-4 学生）。
- AWS：Bedrock 托管微调（含 Claude 3 Haiku、开源模型定制）价格激进但体验偏重；权重级深度定制走 SageMaker（JumpStart 支持 Llama/Mistral/Falcon 等）。
- 来源：<https://developers.openai.com/api/docs/guides/reinforcement-fine-tuning>、<https://developers.openai.com/api/docs/guides/supervised-fine-tuning>、<https://aws.amazon.com/blogs/aws/fine-tuning-for-anthropics-claude-3-haiku-model-in-amazon-bedrock-is-now-generally-available/>、<https://platform.claude.com/cookbook/finetuning-finetuning-on-bedrock>、<https://internative.net/insights/blog/enterprise-ai-platform-comparison-vertex-bedrock-foundry-2026>、<https://www.bitslovers.com/bedrock-vs-azure-ai-foundry-vs-vertex-ai/>

## 数据与趋势（2026-07）

- 合成数据 + 蒸馏流水线成为主流微调数据来源：种子样本 → Self-Instruct 式扩展（frontier 教师模型）→ LLM judge 过滤 → 产出 SFT 语料与 DPO 偏好对。三条路径：直接提示 frontier 模型+质量过滤、开源流水线框架、云厂商原生蒸馏 API。
- 小模型蒸馏浪潮：DeepSeek-R1 蒸馏系、Qwen 蒸馏系等"小而强"推理模型；(输入, 推理轨迹, 输出) 三元组蒸馏让小模型逼近大模型任务表现。
- 微调 vs RAG 共识口径：RAG 管"知识"（常更新、要溯源、按用户隔离），微调管"行为"（语气、格式合规、领域推理、规模化降本/降延迟）；多数生产应用两者组合——"微调管风格与格式，RAG 管事实"；企业场景默认 RAG 起步，行为改不动再上微调。
- 来源：<https://futureagi.com/blog/synthetic-data-fine-tuning-llms/>、<https://www.digitalapplied.com/blog/synthetic-data-generation-llm-training-decision-guide-2026>、<https://labelyourdata.com/articles/machine-learning/model-distillation>、<https://www.groovyweb.co/glossary/fine-tuning-vs-rag>、<https://unicoconnect.com/blogs/rag-vs-fine-tuning>
