## 对话中给出的解释要点(新章素材)

### 关系
- sub-agent 不是另一个物种:就是一个普通 Agent,只不过"雇主"是另一个 Agent 而非人。
- Agent 本质 = 模型 + 工具 + 循环;主 Agent 工具箱里有一个派遣工具(Claude Code 的
  Task/Agent 工具),调用它 = 现场启动一个全新的 agent 循环。运行时层级关系,同一机制递归一层。

### 三个关键性质
1. 独立上下文——看不到主对话历史,只看到那份"任务书"(prompt);
2. 窄带通信——只交回一段最终文字,过程废料随它销毁;
3. 一般不可再派——sub-agent 通常拿不到派遣工具,防无限递归。

### 三大动机(重要性递减)
1. 省主对话上下文(最核心):搜代码库产生几万 token 过程废料,主对话只为结论付钱;
2. 并行:互不依赖的子任务同时跑(fan-out / fan-in);
3. 专业化 + 权限收窄:单独配系统提示/模型/工具白名单(只读搜索员、便宜模型跑脏活、审查员人设)。

### 具体用法(Claude Code 为例)
- 什么都不做:主 Agent 自己判断派不派、派几个;
- 点名:"开一个 subagent 去调研 X";
- 自定义角色:项目 `.claude/agents/` 或用户级 `~/.claude/agents/` 放 markdown,
  frontmatter 写 name/description/tools/model,正文是系统提示词;description 是写给
  主 Agent 看的路由依据;
- 产品化编排:如 /code-review ultra 派一组 reviewer 并行审。

### 派与不派的决策账
- 成本:冷启动(不带主对话记忆)、窄带通信(任务书写不清就跑偏、中途没法追问)、token 总消耗更高;
- 不派:两三步的活 / 步骤强顺序依赖 / 依赖主对话已积累的理解;
- 派一个:过程废料多、结论短(大范围搜索、调研);
- 派 N 个并行:能切成互不依赖的块;
- 口诀:"我要的是结果,还是过程?"只要结果 → 适合外包;过程影响后续决策 → 留主对话。
- 类比:主 Agent 是项目负责人,sub-agent 是临时外包——没参加过之前的会(任务书必须自包含),
  交一份报告就走人(过程不占你的脑子),可以同时雇好几个(并行)。
