# 全库 PPT 视觉设计系统（v3.3 · "深蓝青橙"明暗节奏）

制作或改写任何模块讲义时都必须使用本文件。核心原则不变：**统一设计语言，不统一内容模板**——
尺寸、字体、字号、色板、边距、页脚和来源保持一致；页面构图由本页要表达的概念、流程、证据或
决策决定。本版设计语言以早期库沉淀的"深蓝青橙 + 明暗节奏"风格为基线定版。

## 1. 固定设计令牌

| 项目 | 统一值 |
| --- | --- |
| 页面 | 16:9，13.333 × 7.5 英寸（或 1280 × 720 等比例画布） |
| 标题字体 | Cambria（封面主标题、章节页标题、内容页大标题） |
| 正文字体 | Calibri（中英文正文统一声明 Calibri，交付前确认未被替换） |
| 代码字体 | Courier New，仅用于可运行代码、命令和协议报文；除此之外不得混用字体 |
| 内容页背景 | `#FFFFFF` |
| 深色页背景 | `#0B2540`（深海军蓝：封面、章节过渡页、动手做页、总收束页） |
| 主文字（浅底） | `#1A2733` |
| 次级文字（浅底） | `#51606E` |
| 主强调青 | `#128199`（重点、链接、章节标记；深色底上用亮青 `#1CA3B8`） |
| 深青 | `#0E4D64`（深色块、表头、案例标签） |
| 点缀橙 | `#F2A65A`（关键提醒、对比中的"另一面"、暖色标签） |
| 成功绿 | `#2F855A`（小结、通过、推荐语义） |
| 卡片浅底 | `#F3F8FA`（首选）/ `#EAF3F5`（备选） |
| 浅色块 / 分隔 | `#D6E1E8`（色块）、`#9FB3C0`（分隔线、弱文字） |

除数据图表确实需要增加系列色外，不新增品牌主色。图表新增色必须低饱和、与主色可区分，并在同一讲义
中保持相同语义；红色只表示风险或失败，不用于普通装饰。**内容页不加四边描边框、边条等页面级装饰**；
视觉记忆点靠明暗节奏和色板，不靠边框。

## 2. 明暗节奏（本设计语言的骨架）

全书用"深色骨架页 + 浅色内容页"形成翻页节奏：

- **深色页（`#0B2540` 底、白字/亮青字）**：封面、章节过渡页、动手做页、总收束页。
- **浅色页（白底）**：导览、学习目标、全部概念/深度/对比/案例页、对练页、小结、串联、验收清单、来源页。
- 深色页文字用白色与 `#1CA3B8`/`#F2A65A` 强调；浅色页按上表令牌。
- 章节过渡页只承担"章号 + 章标题 + 本章核心问题一句话"，留白要足。

## 3. 字号与页面骨架（档位由 KB-CONFIG「阅读场景」决定）

字号分两档，**全库只用一档**，由 KB-CONFIG 的「阅读场景」字段决定；默认**自读档**
（库主人在自己屏幕上阅读，字小页密看着舒服）；投屏授课/汇报的库用**投屏档**。

| 文本角色 | 自读档（默认） | 投屏档 | 规则 |
| --- | ---: | ---: | --- |
| 封面主标题 | 40–48 pt | 54–64 pt | 最多两行；优先缩短文字 |
| 章节过渡标题 | 33–40 pt | 44–52 pt | 一页只承担章节转场和核心问题 |
| 常规页标题 | 26–30 pt | 35–38 pt | 一行优先；必须是结论式标题（见 ppt-rules） |
| 副标题 / 大号结论 | 18–22 pt | 24–28 pt | 一句话结论、步骤名或图表结论 |
| 正文 | 12–15 pt | 18–22 pt | 信息较多时先拆页；自读档最低 11 pt、投屏档最低 16 pt |
| 表格 / 问答答案 | 11–13 pt | 16–18 pt | 低于下限必须拆表或拆页 |
| 图注 / 辅助说明 | 10–12 pt | 13–15 pt | 密集对照表、图下说明、导览条目；每页限量使用 |
| 来源 / 页脚 | 8.5–10 pt | 9–11 pt | 只放来源、模块名、章节 ID 和页码 |
| 代码 / 报文 | 11–13 pt | 15–18 pt | 超一屏优先拆页或只展示关键片段 |

- 左右安全边距至少 0.55 英寸，顶部标题区约 0.45–1.15 英寸，底部页脚区约 0.28 英寸。
- 正文、图表和图示不得侵入标题区或页脚区。
- 一页只表达一个主要主张；自读档允许更高信息密度，但仍不要密到失去层级。
- 标题、结论、图示和来源建立清晰层级；不要把整页做成密集的卡片仪表盘。
- 同一个库两档不得混用；改档位＝全库重刷事项，走 B 类并记入 MANIFEST。

## 4. 封面契约（每个字段都是强制的）

封面自上而下依次为，相邻元素**垂直间距 ≥ 0.25 英寸**：

1. 眉题：`<知识库显示名> · 讲义式 PPT`（**12–14 pt**，细字重或字距拉开）。显示名取
   KB-CONFIG「知识库显示名」字段（缺省用"AI 知识库"）——**全库所有讲义眉题必须逐字相同**；
   2026-07-14 审查曾发现三种眉题变体并存，根因就是显示名从未定义；
2. 主标题：`<模块名> · <一句话价值主张>`（54–64 pt，Cambria，最多两行）；
3. 英文副题或一句话定位（20–24 pt，可斜体、用主强调青）；
4. 章节概览（可选，列了就要列全，**禁止只列前几章**）按章数分档：
   - **≤6 章**：单行章节条列出全部章；
   - **7–10 章**：两行胶囊网格（如 2×4、2×5）或单行窄胶囊（7 章可行）——实库多册 8 章
     2×4 网格已验证效果良好；
   - **>10 章**：按篇分组条（如"原理篇 · 第 1–5 章"）或省略。
   分组条 **≥5 组时每格空间有限**：组描述压缩到 10 字以内，或改为两行布局，不得让文字
   被迫换行挤压；胶囊之间不得重叠或相触（RAG 曾因胶囊宽度累计超行宽而粘连）；
5. 读者定位一行 + 核实日期（右下或底部，10–12 pt）。
   **B 类增章后，本节 1–5 全部属于回刷对象**：章节概览的章数、核实日期都不得停在旧状态——
   `audit_pptx.py` 检查项 10 会按「封面最大编号（数字胶囊，或分组条『第 X–Y 章』取范围最大值）/『X 章一条主线』口径 vs 正文最大章号」硬校验。

**明确禁止**：模块代号/模块名以 >20 pt 的大字出现在主标题正上方（与主标题内容重复且挤压布局）；
任何两个文本框边界相触或重叠；核实日期与上方文字间距为零。封面属于逐页目检的点名对象。

## 5. 按内容选择版式

先判断本页的沟通任务，再选构图；不要循环套用同一版式。

| 内容任务 | 推荐构图 | 避免 |
| --- | --- | --- |
| 定义一个概念 | 大标题 + 一句定义 + 一个关键图或因果链 | 四宫格定义卡片 |
| 讲类比与案例 | 概念/类比/案例/小结可用标签行分块，但同构布局连续不超过 3 页 | 全章每页同一套四行卡片 |
| 解释机制 / 流程 | 左到右或上到下的 3–7 步流程；箭头先画、节点后画 | 交叉连线、没有起终点 |
| 做技术对比 | 结论先行的表格或二维矩阵；突出选择条件 | 罗列产品功能而没有取舍 |
| 展示架构 | 分层或泳道，边界、输入输出、控制面清楚 | 把所有组件堆成云朵图 |
| 展示代码 / 协议 | 关键片段 + 行级注释 + 运行结果或报文流向 | 超过一屏的代码墙 |
| 展示定量证据 | 大数字醒目呈现（32–44 pt）+ 上下文一句 + 来源短注 | 数字埋在段落里 |
| 做决策 | 约束 → 候选 → 证据 → 推荐；明确不适用条件 | 没有前提的"最佳方案" |
| 客户问答 | 每页 2–3 组问答；一章 5–8 题拆成多页 | 为塞满一页把答案缩到 16 pt 以下 |
| 章节小结 | 3 个可复述结论 + 下一章为何需要继续 | 重复本章目录 |
| 来源 | 分组列出来源、访问日期和对应章节 | 裸 URL 堆满页面 |

可使用原生 PowerPoint 形状表达简单流程、分层和表格；复杂关系图优先 Graphviz；只有确实能提升理解时
才使用图片搜索或图像生成。视觉资产服务于当前主张，不为装饰而添加。

## 6. 固定页面约定（结构以 ppt-rules 为唯一出处，本节只管长相）

页面顺序与每章固定元素——导览页、学习目标、动手做、对练页（题数与拆页档位）、本章小结、
速查章、收尾四件套——**定义、数量、内容要求全部以 `ppt-rules.md` 为准**；本文件不重复
写一遍（两处各写曾导致规则漂移与门禁打架）。这些页面的视觉归属：

- 第 1 页封面按第 4 节契约执行；第 2 页标题固定为"导览"（audit 检查项 7 硬校验），
  章多按篇分组，可参考三栏 + 圆形编号的分组版式。
- 深浅归属见第 2 节明暗节奏；章节过渡页只承担"章号 + 章标题 + 本章核心问题一句话"，留白要足。
- 对练页版式服从字号下限：达到档位下限就拆页，不许压字号凑一页。

## 7. 页脚与来源

- 常规页左下角放 `<模块名> · <章节 ID>`，右下角放页码；两者使用 9–10 pt 次级文字。
- 封面可不放页脚；导览、章节页和来源页仍保留页码。
- **插页后页码必须重排（2026-07-13 修订）**：页脚带页码的讲义，在中部插入或删除页后，其后
  所有页的页脚页码要按新页位重写（工具中立：按「旧页位 → 新页位」映射重写页脚末尾的数字
  run 即可），并在渲染目检时抽查一页确认。页脚不带页码的讲义不受影响；B 类增补频繁的模块，
  新建时可考虑页脚只放章节名不放页码，降低维护成本。
- **插/删页后必须同步 docProps/app.xml（2026-07-13 修订三）**：app.xml 登记着页数
  （`<Slides>`）与每页标题清单（`HeadingPairs` + `TitlesOfParts` 向量）。**只有 PowerPoint
  校验这份登记表**——LibreOffice、渲染目检、常规脚本全部无视它，登记与实际不符时 PowerPoint
  会弹「需要修复」。改动页数后按真实页序重建这三处（或整体重写为自洽的 app.xml）；
  `scripts/audit_pptx.py` 已将 app.xml 一致性纳入门禁（检查项 9），收尾自检必跑。
  教训：2026-07-13 插页批次漏了它，全库 PowerPoint 弹修复、用户被迫逐册手修。
  另注：app.xml `<Slides>` 只是**册内**页数登记；册**外**的派生页数账（模块/顶层 README、
  `_prep/` 一页纸、MANIFEST 讲义页数）在插/删页后同样要回刷，四面清单见 `illustration-rules.md` 第八节（v3.9）。
- **OOXML 顺序契约（FAIL 级门禁，audit 检查项 12）**：生成或修改讲义必须满足两条元素顺序
  规则——①`presentation.xml` 的 `<p:notesMasterIdLst>` 紧跟 `<p:sldMasterIdLst>` 之后、
  必须在 `<p:sldIdLst>` 之前；②任何 `<a:p>` 段落里 `<a:pPr>` 是**第一个子元素且只能有一个**
  （生成脚本合并段落时最易把第二段的 pPr 留在 runs 中间）。违规文件在 Mac PowerPoint 的
  「不受信文件」严格校验通道会弹修复，而 LibreOffice、渲染目检、宽松通道全部发现不了
  ——"门禁全绿仍弹修复"多半就是它（quarantine 机理与取证过程见库 `_maintenance/` 事故复盘
  及 CHANGELOG v3.4 修订二）。修复工具 `fix_schema_order.py`（零语义元素重定位，随库
  `_maintenance/`）。**排查方法论**（"只有 PowerPoint 拒收"类问题直接用）：拿一份
  PowerPoint 亲写文件（弹窗时手点 Repair 后保存的产物）做基线，ECMA-376 XSD + xmllint
  对我方文件全部件校验，只在我方出现、基线没有的违规类才是病。
- **改成品仍優先 zip 级最小手术**：小修（改字、调形状、补页脚）用 zip 层替换目标部件即可，
  不必整包交给 python-pptx 重存——round-trip 会重排 slide 文件名与 rId、放大 diff 面，
  且**不会修复也不会引入**上述顺序违规（2026-07-14 曾误判 round-trip 为弹修复根因，
  实为生成器违规被原样保留；归因已更正）。副本目录（对照库快照、整理导出）与主库
  **同病同修**。**给讲义配信息图（插新页）**也走 zip 级嫁接（`scripts/kb_insert.py`，新页
  独立生成、原部件字节不动、自动同步 app.xml），配哪些页 / 数量按需 / 绘图规范 / 布局踩坑
  见 `references/ppt/illustration-rules.md`。
  **纯文本改写一律字节级，绝不用 XML 库解析回写（v3.16）。** 改字这类只动文本内容的手术，
  必须在 slide XML 的**字节/字符串层**做 `str.replace`，并满足两条：①替换串在该部件内
  **命中数恰好为 1**（多于 1 说明定位不唯一，中止而非猜）；②改完只 `ET.fromstring`
  **解析校验良构，绝不 `ET.tostring` 回写**。原因：ElementTree 等 XML 库回写会把
  `a:`/`p:` 命名空间前缀重写成 `ns0:`/`ns1:`——XML 良构、语义等价、检查项 17 放行，
  但违反库内 OOXML 前缀约定，audit 其余正则检查全线 FAIL（2026-07-20 实测 8 册全挂，
  检查项 18 已程序化拦截）。**「绝不整包重存」要读成「绝不让任何 XML 库产出最终字节」**，
  不只是不用 python-pptx。
- **[Content_Types].xml 不得有孤儿 Override**（指向包内不存在部件的条目，生成器遗留脏数据；
  `audit_pptx.py` 检查项 11 已纳入门禁）。
- **theme / notesSlide 不得跨对象共享（FAIL 级门禁，audit 检查项 14；2026-07-14 定案）**：
  python-pptx 生成的讲义普遍有两处只有 Office 严格校验才拒收的共享——①`notesMaster` 与
  `slideMaster` 共用同一个 `theme`（每个 master 须独立 theme）；②多个 `slide` 引用同一个
  `notesSlide`（每 slide 须独立 notesSlide）。与检查项 12 同理：LibreOffice、audit 旧版、
  渲染目检**全部放行**，只有 Mac PowerPoint「不受信文件」通道弹修复——这是"生成即弹、
  门禁全绿仍弹"的**另一个**根因（区别于检查项 12 的元素顺序违规，不是同一回事）。修复工具
  `fix_theme_sharing.py`（随库 `_maintenance/`，零语义：复制出独立 theme、拆分共享 notesSlide
  副本，全程 zip 级不重存整包）。定案排查法同检查项 12：PowerPoint 亲写 Repair 版做基线、
  逐部件对比即现形（theme 1→2、notesSlide 补齐），全库 19 册一次性纯脚本修净。
- 时效性事实、产品能力、版本、安全事件和定量数据必须在**事实所在页**底部给出短来源
  （机构 + 文档名 + 日期），完整 URL 收录在末尾来源页和模块 raw-data 核实笔记。
- **来源反盖章**：来源短注只出现在含有对应事实的页面；类比页、动手做页、问答页若无新事实，
  不放来源行。同一来源在一章内最多出现在 2 页——每页都盖同一个章，等于每页都没来源。
- 同一页来源超过 3 条时，拆页或把事实聚合成表，不把来源压到不可读。

## 8. 既有讲义的一致性审计

新增模块前先抽查现有讲义；全库收尾时复查全部讲义。出现任一情况就视为不统一，先把旧版存入
`raw-data/history/`，再按本设计系统重制：

- 页面不是 16:9；
- 标题字体不是 Cambria、非代码正文声明字体不是 Calibri，或代码/命令/报文声明字体不是 Courier New；
- 标题或正文低于本库档位的下限（自读档：标题 26 / 正文 11；投屏档：标题 35 / 正文 16），
  或标题本应一行却发生换行；两档混用；
- 使用与统一色板冲突的品牌主色，且不是图表系列色或风险语义；
- 第 2 页不是"导览"，或封面违反第 4 节契约（含"章节条停在旧章数"的漂移——增章后
  封面/导览的章数口径、核实日期未随正文回刷；2026-07-14 审查一次抓出 4 册，已由
  `audit_pptx.py` 检查项 10 程序化兜底）；
- 页脚缺模块名 / 章节 ID / 页码，或时效性事实没有来源、来源被盖章式重复；
- 大量页面机械重复同一种卡片布局，或内容被模板迫使得过密/过空；
- 全库多数讲义页数相同（模板化信号）；
- 存在溢出、遮挡、裁切、断裂连接线、空占位符或无法解释的视觉元素。

## 9. 生成与验收门禁（工具中立：标准恒定，引擎按环境选）

本技能是**通用型技能**，必须能在任何 agent 环境（Claude Code、Codex、或其它）运行。因此本节
只定义**四种能力**和它们的验收标准；每种能力用什么工具实现，按"环境里有什么就用什么"解决，
**缺哪个工具就用等价物补，不得跳过验收**：

| 能力 | 验收标准 | 环境映射示例 |
| --- | --- | --- |
| ① 生成 PPTX | 产出合法 .pptx，遵守本文件全部令牌 | Claude Code：`pptx` 技能（pptxgenjs）；Codex：Presentations 技能（`@oai/artifact-tool`）；无现成技能的环境：直接写 python-pptx 或 pptxgenjs 脚本 |
| ② 文件校验 | 文件无损坏、无画布溢出 | Claude Code：`validate.py`；Codex：`slides_test.py`；兜底：能被 LibreOffice 无错打开并完整转 PDF |
| ③ 逐页渲染 | 全部页面渲染成图片供目检 | 任何环境通用：`soffice --headless --convert-to pdf` + `pdftoppm`（或 ImageMagick）；Codex 可用 `render_slides.py` |
| ④ 契约自检 | 16:9/字体/字号/色板/隐藏 placeholder 全过 | 任何环境通用：技能包 `scripts/audit_pptx.py`（零依赖 Python）；库内另有 `_maintenance` 门禁则一并跑 |

**禁止在任何产出物或规则里写死单一环境的专属工具名**；引用工具时一律按上表"能力 → 环境映射"
表述。python-pptx、pptxgenjs、`@oai/artifact-tool` 都是合法引擎，只要产出满足验收标准。

门禁步骤（顺序执行，全部通过才算完成）：

1. 每张常规内容页在最底层放一个覆盖完整 16:9 画布的显式背景形状。以 LibreOffice 渲染时，
   不用可能被优化掉的纯白无边框矩形；使用视觉上等同白色的 `#FEFEFE` 填充和同色 1 px 细线，
   作为稳定的 render anchor（深色页用 `#0B2540` 实底矩形，天然充当 anchor）。
2. 内容页优先使用无母版布局的普通 slide，并用安全边距内的普通文本框绘制可见标题。禁止隐藏、
   1 px、负坐标或画布外的 title placeholder；这类对象会让 LibreOffice 对同页可见文字做不稳定裁切。
3. 生成后先跑文件校验（按上表选工具），必须无画布溢出、无损坏。
4. 跑 `scripts/audit_pptx.py <讲义.pptx> --profile reading|projection`（按 KB-CONFIG 阅读场景选，
   默认 reading）：16:9、字体白名单、对应档位的字号下限、色板、负坐标、标题唯一性、
   app.xml 一致性、**章数一致性（封面章节条/「X 章一条主线」口径 vs 正文最大章号）**全部通过。
   脚本页码一律按放映顺序报告；小字预算不计表格单元格；非令牌色分「系统性漂移 / 零星」两级，
   系统性漂移（同色 ≥10 页且 ≥25% 页面）不强制整册重着色，但该册下次 B 类更新时应随手回正。
5. 渲染全部页面到**新的输出目录**；联系表只检查全局节奏，每一页仍要以全尺寸图片单独检查，
   不能复用旧目录里的陈旧图片。
6. 逐页目检**点名对象**：封面（第 4 节契约）、导览页、每章第一个深度页、稀疏的概念页/小结页/
   来源页（eyebrow、标题、statement、左下页脚不得缺开头或丢字）。
7. 检查标题唯一性与结论式标题、标题和正文字号、字体声明、色板、页脚、页码、来源（含反盖章）、
   代码可读性和表格断行。
8. 检查每章包含学习目标、动手做、5–8 个问答及本章小结；第 2 页必须是"导览"；讲义末尾
   收尾四件套齐全。
9. 修复后重新生成并重跑全部门禁；不得用"渲染成功"替代逐页视觉检查。
10. 在采用本知识库目录结构的项目里，全库收尾还要运行库内 `_maintenance` 门禁（如
    `validate_kb.rb`、`audit_artifacts.rb`）；自动门禁通过后仍需保留第 5–8 步的逐页人工检查。

全库一致性的判定依据是本文件，不以某一份历史 PPT 的偶然样式为准。

## 10. 五层防线（PPT 面的具体层；模型与使用纪律见 core-rules.md §三）

分层验收模型（"上层全绿、下层拒收"的事故结构、跳层留门、新事故先问哪层缺失）的唯一
出处在 `core-rules.md` §三；本节只列 PPT 面的五层与手段。历史取证：v3.4 OOXML 顺序违规、
v3.6 跨对象共享、检查项 17 非良构 XML、v3.12 画布裁切——缺任何一层都出过真实事故：

| 层 | 手段 | 只能抓什么 | 抓不住什么（对应真实事故） |
| --- | --- | --- | --- |
| 1 结构门禁 | `audit_pptx.py`（正则/zip 级） | 已编码成检查项的规则（令牌/字号/账面/顺序契约/共享…） | 未编码的病类；非良构 XML 照样 PASS（检查项 17 的由来：1105 处标签错配全绿） |
| 2 XML 良构 | 真 XML 解析器逐页 parse（已并入 audit 检查项 17） | 字节级手术弄坏的标签结构 | 语义合法但视觉出界/裁切 |
| 3 渲染目检 | soffice 渲**目标册本体**逐页看（禁止只渲独立预览册） | 溢出/裁切/丢页/字形/衔接 | 只有 PowerPoint 校验的部件（app.xml、theme/notesSlide 共享、元素顺序） |
| 4 账实脚本 | `check_page_ledger` / `check_ebook_ledger` / `check_html_links` | 派生账与实物的漂移（账目全集见 library-rules「派生账登记表」） | 文件内部的一切问题 |
| 5 PowerPoint 亲测 | 请用户双击抽验（Mac「不受信文件」严格通道） | 只有 Office 严格校验才拒收的一切 | ——最后防线，无法自动化替代 |

PPT 面执行口径：收尾从第 1 层依次跑到第 4 层，第 5 层交用户双击抽验（Mac「不受信文件」
严格通道是最后防线，无法自动化替代）。
