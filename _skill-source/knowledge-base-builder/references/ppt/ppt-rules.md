# 讲义式 PPT 规则

流程第 3 步（出章节大纲进方案）和第 4 步（亲自写成品）用这份。**PPT 由你亲自写好，是主力成品**，
不是电子书的附属。产出 `<模块名>-讲义.pptx`。本技能是**通用型技能**：制作 .pptx 时用**当前
环境可用的 PPTX 生成能力**（有现成 PPT 技能就配合它用，如 Claude Code 的 `pptx` 技能、Codex 的
Presentations 技能；没有就直接写 python-pptx / pptxgenjs 脚本）——**引擎随环境换，规范和验收
标准永远不换**（能力与工具的映射表见 `ppt-design-system.md` 第 9 节）。**生成和验收（流程第 4、5 步）前完整读取
`ppt-design-system.md`**；第 3 步大纲阶段只读本文件——大纲定的是内容与结构，视觉令牌与门禁
到动手做成品时再加载，两份文件分开省的就是这段上下文。动笔写讲义文字前另读
`writing-rules.md`（对外文字通用纪律：反 AI 腔、术语、语序——两面共用的唯一出处）。

## 定位与风格

课程讲义 / 教科书风格，**尽量全面覆盖、尽量展开、不压缩内容**（宁可页多，也不要跳步）。读者
画像见 KB-CONFIG——不论哪种画像，目标都是能把概念讲给别人听明白，所以每个概念都要讲"活"，
而不是给一句抽象定义了事。

**倾向于更丰富、更展开**：一个概念能拆成多页就拆多页；类比、案例、图示、对比表能加就加；宁可
啰嗦讲透，也不要为了省页数而跳步。默认把内容做厚，而不是做薄。


## 内容决定版式，设计系统负责统一

全库统一的是页面尺寸、字体、字号层级、色板、边距、页脚、来源和质量门禁，**不是把每一页塞进同一套
卡片模板**。概念、流程、对比、案例、代码、决策矩阵、问答和来源页应使用不同的内容结构；每页先写清
唯一主张，再选择最适合表达该主张的版式。具体设计令牌与验收命令见 `ppt-design-system.md`，全部属于
强制规则。

**每章固定的是"元素"，不是"页数"**：学习目标、动手做、对练页、本章小结、来源标注这些元素必配；
但一章的**深度页（概念/机制/对比/案例展开页）在 2–6 页之间按内容复杂度浮动**——核心章（如 RAG 的
检索、Agent 的编排）就该比周边章厚。**进阶/复杂专章的深度页取区间上限（4–6 页）**：GraphRAG、
多模态、Computer Use 这类本身就是「厚内容」的专章，并入、迁移或按新规范重制时**不得自动落到
下限配置**——章数并进来了、深度却缩水，等于白并。工作场景挂钩（如「上云怎么落地」）**每章
检查一次**，进阶专章尤其不得缺席（KB-CONFIG 配置为「无」除外）。**反模板化硬指标**：如果一次
全库重刷后超过半数讲义总页数相同，或所有章节页数完全一致，判定为模板化失败，说明内容在被骨架
削足适履，返工重排。

## 定量锚点（硬数字进正文，本版新增的重点）

讲义正文必须承载**定量证据**，不能全是定性的形容词和方向词。第 2 步联网核实到的百分比、倍数、
延迟、成本、规模等硬数字，凡是支撑本章论点的，**都要落到幻灯片正文里**（配来源短注），而不是
只存进 MANIFEST 的时效性事实表。要求：

- **每章至少 2 处带来源的定量锚点**（如"朴素 RAG 检索环节失败率约四成""GPU decode 阶段利用率
  常不足 1%"），核心章应更多。
- **每章至少 1 页含可手算复现的数值算例或可运行片段**（如 RRF 的 1/(k+r) 实算、成本三本账、
  真实报文），不许全章都是定性表格。
- 数字必须可溯源：页底来源短注 + 完整 URL 收录进末尾来源页与 raw-data 核实笔记。
- 没核实到硬数字的论点，宁可明说"以实测为准"，不要编数。
- **换算陷阱自检（v3.13，真实事故）**：折扣/倍率/百分比互转是**方向性错误**高发区——
  "省 90%"≠"打 9 折"（是打 1 折）。凡把核实口径换算成另一种说法，必须**双口径并写**
  （如 `0.1×（约 1 折，省约 90%）`），让读者和后续核对都能交叉验证方向；MANIFEST 时效表
  登记**原始口径**、不登记换算结果。数字弹药页/速查页引用的数字必须溯源到 MANIFEST 时效表
  或事实所在页的来源短注，**不从其它正文页转抄**——一处换算错会被高频取用页复制放大
  （2026-07-20 实例：一条"省 90%"写成"打 9 折"，经数字弹药页扩散到 2 册 7 处）。

## 导览页与页标题（全库统一）

- **第 2 页固定是"导览"页**，页标题就叫"导览"：全书章节一览（章多就两栏或按篇分组）+ 一句
  "本讲义怎么读"。
- **同章多页不共用同一页标题**，且**可见大标题必须是"结论式"的**——用一句话说出本页的主张或
  问题，如"RRF：不比分数，只比名次"、"先做 RAG / 非 RAG 决策，再画技术架构"；结构角色词
  （CORE CONCEPT / DEEP DIVE / 概念 / 深入 之类）只能出现在眉题，**不能充当大标题**。
  原因：讲义近百页，导航靠标题；"第 3 章 · 深入解析"这种标题既不能导航也不能复述。
- 若 PowerPoint title placeholder 在 LibreOffice / headless 渲染中造成裁切，优先使用正常坐标的
  普通可见标题文本框；不要为了大纲视图保留隐藏、微小或负坐标的 placeholder。导航可依靠唯一标题、
  缩略图和模块内导览，渲染完整性优先。

## 语言策略 · PPT 面落点（通用规则见 writing-rules.md）

术语中英对照、缩写展开英文全称、边界判定与语序铁律的**唯一出处在
`references/shared/writing-rules.md`**（连同反 AI 腔写作纪律），动笔写讲义文字前先过一遍；
本节只写 PPT 面特有的两条：

**开篇"重点术语对照页"口径**：讲义开篇设术语对照页的，条目一行一个术语，写法
`缩写，英文全称`（生僻术语补第三段中文译名）；只列"缩写↔中文"对照不合格。

**落点纪律**：缩写型术语的英文全称展开落在
最早可排版的正文出现处；封面、导览、章节扉页、眉题、页脚与纯英文章扉题不作落点；该处排
不下就顺延到下一处出现（此时清单/MANIFEST 记录首现页与落点页两个页码）。

## 结合工作场景落地（挂钩方式以 KB-CONFIG 为准，有真实关联才结合，不强行）

挂钩方式由 KB-CONFIG 的"工作场景挂钩"定义：默认是"上云落地"（下述）；也可能是"自家产品
对照"等变体（规则同理），配置为"无"则整节跳过、别硬塞。默认口径：读者在云/AI 平台工作，
最终要把技术讲成"在我们云上怎么落地"。所以**只要某个概念能自然对应到云服务，就点出来**：
这一步在云上通常用什么服务扛？给客户讲时能落到具体产品。例如 RAG——离线建库
（文档解析、切分、算 Embedding、灌进向量库）用什么云服务，在线检索/重排/生成又分别用什么云服务。

做法：在相关概念页或每章的一个"上云怎么落地"小节里，用一句话/一张小表把"技术环节 → 可选云服务"对上。
- **给主流云的对照，而非绑死一家**：同一环节列几个代表性选项（如向量库可用托管向量检索服务 /
  自建在对象存储+检索引擎上等），必要时联网核实服务名与现状。
- **是"有就点、没有别编"**：纯理论、和云无明显关联的概念（如某个纯算法推导），不必硬塞云服务，
  塞了反而离题。判断标准：这个环节在真实工程里是不是通常跑在某类云服务上？是才写。

## 讲每个概念的固定节奏：概念 → 通俗类比 → 具体案例 → 小结

这是本技能的灵魂。为什么这样排：售前的价值在于"把复杂讲简单又不失真"。类比负责让人瞬间抓住直觉，
案例负责把类比落到真实场景、防止只停在比喻。所以：

- 每个概念都配一个**生动类比** + 一个**具体案例或真实场景**，不要只给抽象定义。
- **复杂概念一页只讲一个点**，讲透了再翻页；不要一页塞多个概念。
- 涉及代码或协议的地方，给**可运行的具体例子**（真实能跑的片段/真实报文），不要伪代码带过。
- 节奏是内容节奏不是版式模板：同一章里概念页、类比页、案例页可以合可以分，按信息量定，
  不要求每个概念机械产出四页或挤成固定四行卡片。

## 每章结构

- **开头**：一页"学习目标"，列清这章学完能讲清楚什么。
- **主体**：按上面的节奏逐个概念展开（深度页 2–6 页弹性）；多用**图示、流程图、对比表格**帮助理解。
- **动手做**：每个学习阶段配一页"动手做"练习页（给任务 + 提示，让读者上手）；建议附一行
  "成功证据"——做到什么程度算过关。
- **对练页**：每章结尾附一页"<提问者>会怎么问"（提问者按 KB-CONFIG，默认客户；见下，必配）。
- **结尾**：一页"本章小结"，收拢要点。

## 「对练」页（每章必配，本技能的重点；提问者按 KB-CONFIG）

读者学这些不只是为了懂，更是为了**被问到时能当场答上来**。所以每章结尾放一页对练页——把
"最可能考你的那个人"（KB-CONFIG 的"对练页提问者"：默认客户，也可以是面试官/评审/同事）
现场最可能抛出的问题，当成面试题来准备。它一举两得：既是应对现场的弹药库，又是读者自己的
知识沉淀（问答形式最利于记忆和自查）。页标题随提问者定，如"客户会怎么问（面试题）"。

做法：列 **5–8 个高频问题**，每个配一个**简洁、能直接说出口的答案（2–4 句）**：先给结论，
再给边界/证据（能带数字或类比最好），口吻贴合场景，能落到价值/取舍/边界，而不是背定义。
**版式服从内容**：为保证字号不破档位下限（投屏档 ≥16pt、自读档 ≥11pt），投屏档每页
2–3 题、自读档每页 2–5 题（4–5 题仅限答案 ≤2 句的短问答），一章拆成 2–3 页都可以，
**但不许为了塞进一页把答案砍成一句话，也不许把答案压到档位下限以下**。优先覆盖：常见误解、"你们和 X 有什么区别"、成本/性能/
安全的尖锐追问、"什么时候不该用"这类反向问题。示例格式：

```
Q：RAG 和直接微调模型，我到底该选哪个？
A：要更新的是"知识"就用 RAG（改数据不改模型、可溯源、便宜）；要改的是"说话风格/能力"才考虑微调。
   多数企业知识问答场景，RAG 是性价比更高的起点，二者也常组合用。
```

## 模块收尾四件套（讲义末尾，全库统一）

讲义最后固定四类页面，一个都不能省：

1. **串联 / 知识地图页**：本模块与库内其它模块的真实关联（与 MANIFEST 串联出边一致）。
2. **总收束页**："一句话记住每一章"——每章一行、可背诵的口诀式回顾（章多可两栏或拆两页）。
   这是全书最高频复用的一页，认真写。
3. **生产验收清单页**（挂钩方式为"上云落地"等工程场景时必配）：把本模块内容折成一张
  "验收面 → 必须能出示的证据"清单，供方案评审/交付对照。
4. **来源页**：分组列出全部来源（机构 + 文档名 + 完整 URL + 访问日期），并指回 raw-data 核实笔记；
   多于一页就拆页。不以空泛"谢谢"页结束。

**速查章（对练类画像的默认末章，v3.5 起转正）**：KB-CONFIG 画像是售前/客户对练类时，
正文最后固定加一章"速查"——高频问题电梯版 / 启用条件决策树 / 上手排错 / 速查表 / 串联地图，
按模块性质取舍二三样。它是全库惯例，不再是可选建议；画像非对练类的库可省略。速查章章题
必须含「速查」二字（眉题如"第 9 章 · 售前速查"）——它不适用"每章固定元素"，
`audit_pptx.py` 检查项 13 靠章题里的「速查」识别并豁免。

## 跨章 / 跨模块串联（由技能主动建议并补上）

知识点之间常有联系（如 RAG 的检索质量 ↔ Evaluation 模块的评估指标；MCP 的工具调用 ↔ Agent 模块）。
这类联系是**动态**的：当前库里可能还没有可串联的点，那就不硬凑。但**只要存在联系，技能就应主动
发现并建议**，而不是等用户开口。

具体做法：产出方案时（流程第 3 步），先扫一遍顶层索引里已有的其它模块，判断本模块与它们、以及本
模块各章之间有没有真实关联；**在方案里单列一条"建议串联"**给用户看。用户认可后：

- 有真实关联：在相关章节加一页"承上启下 / 与 XX 的关系"（用一句话点明联系 + 一张小图或对比），
  并在模块 README 记一笔；最终 MANIFEST 必须写成可解析的 `目录#稳定章节ID`，不能只写模块名、
  "候选"或尚未建立的主题。
- 暂无可串联的点：明确说明"本次暂无强关联，留待后续模块补充"，不要生编。


**对练页新增题要标日期**：往对练页加新题时，题目后缀「新增 YYYY-MM-DD」。
只标题目，章与节不标；存量不补。理由与写法见 `shared/library-rules.md`「新增标注」。

## 增补一致性回刷清单（B 类更新后必做）

往已有讲义增补/修改章节后，**成品必须整体自洽**，以下五处逐一回刷，缺一处都算没做完：

1. **封面**：章节条/篇章分组要反映最新章节全集（列不下就按篇分组或去掉章节条，不能停在旧章数）；
2. **导览页**：章节一览与阅读路径同步到最新；
3. **章节编号与页标题**：新章插入导致的顺延要全书改到；
4. **总收束页**：新章的"一句话"补上；生产验收清单和来源页同步；
5. **页脚页码与 MANIFEST**：页码连续，MANIFEST 的章节清单、讲义页数、产出 skill 版本同步更新。

历史教训：曾出现讲义已扩到 12 章而封面章节条仍停在 5 章的漂移——增补只改正文不回刷框架页，
积累几次后成品就自相矛盾。

本清单是**增章/增补任务的切片**；全库派生账的账目全集与真源定义见 `library-rules.md`
「派生账登记表」（v3.13）——页数账四面回刷的操作细则在 `illustration-rules.md` 第八节。

## 章节相互独立

每章自成一体，便于**单独更新或加新章**。B 类更新时只改指定章节的那几页，别牵动其它章
（但上面的回刷清单属于框架页，必须动）。新加章沿用同样的每章结构，风格与已有章节保持一致。

## 大纲阶段（流程第 3 步）交付什么

先只交**章节大纲**给用户确认，包含：每一章标题、该章要讲的概念点、计划配的类比/案例/动手做、
**计划放哪些定量锚点（已核实的硬数字）**、每章对练页打算覆盖的问题方向、**哪些环节打算结合
工作场景落地**、每章预估深度页数（体现粗细差异），以及**单列一条"建议串联"**（本模块与已有
其它模块/各章之间的关联建议，没有就说明暂无）。用户拍板后再进第 4 步把整份 PPT 写出来。
