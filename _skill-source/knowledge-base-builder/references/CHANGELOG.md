# 版本历史（完整版，按需读——日常任务不用加载本文件）

技能当前版本记在 `SKILL.md` frontmatter 的 `metadata.version`；每个模块的 `MANIFEST.md`
记录产出它的技能版本。技能大改后据此列出"哪些模块还是老规范产出的"，决定要不要重刷
（重刷走 B 类，旧版自动进 history）。事故类条目只保留结论与规则指针，完整取证过程见
各库 `_maintenance/` 的复盘文档。

- **v4.3（2026-07-20）**：**电子书契约翻转——由「能免费下载的下进库」改为「一律只列正规渠道
  链接」**（用户判断：下载到本地这件事鸡肋，保留链接即可）。动机三条：本地文件是第三方版权材料，
  既不能随库分享给同事、也不能进版本库（同日同步 GitHub 时正是靠 .gitignore 排除掉的 302M），
  还要随版本更新重下；而链接三样都行。馆藏唯一优势是离线可用，实际用不上。
  **执行顺序是这次的关键**：先补出处、再删文件——逐份核查 71 份的出处可追溯性（README.txt
  `来源:` 行 / arXiv 编号反推 / 书单行内 URL 三者取并集），查出 **9 份无出处**，联网逐份补齐
  （两份 zip 交互式教程最难，一个 PDF 链接替代不了）后才动删除，并把「文件名 → 出处」快照存进
  库的 `_maintenance/<日期>-原电子书馆藏出处存档.md`。**顺序反了就是不可逆的信息丢失**，
  已写进 init-rules 迁移步骤。落地面：19 份书单 72 处 ✅ 行改链接行、19 份模块 README 与两层
  索引的馆藏描述、表头「获取方式 / 文件位置」→「获取方式」、KB-CONFIG 删「电子书馆藏」字段。
  `check_ebook_ledger.py` 判据整体换掉（文件存在性 → 每条有无可用链接 + 有无馆藏残留 + 链接
  形态合法），**新判据一上线就抓出 2 条历史坏账**：Data-Engineering 条目 6 只写"官方文档（三家）"
  从来没给链接、Model-Landscape 条目 10 的域名没写 `https://` 前缀——都不是本次改坏的，是旧判据
  查文件存在性时**结构性看不见**的盲区。同样按 v4.2 立的规矩注入错值证伪一次（三类坏账全报出）。
  **代价要认并已挂兜底**：链接会烂且烂得不均匀——arXiv/NIST/OWASP 稳，厂商白皮书（OpenAI/
  Anthropic/AWS/Google）最易被悄悄挪走，偏偏又是对客最常引用的；脚本不探活（404 与限流难分、
  巡检要秒级），改由 patrol-rules 的建议复查日轴 + 可选抽查兜底。
- **v4.2（2026-07-20）**：**页数账补第二根轴「全库总页数」**（起因：用户把知识库同步 GitHub，
  电子书与讲义历史快照排除在版本库外，给库根补了 README.md 作为在线门面——它写着全库总页数，
  于是问「挂进页数账」）。查下去发现真问题不在这一处：原四面账**全是单模块页数**，而全库总数
  另挂在四个文件五处（库根 README.html / README.md、`_prep/全库一页纸.html` 标题与页脚、
  `_prep/MANIFEST.md`），**这根轴此前零脚本覆盖**——19 册单模块账可以面面一致，总数却整体
  停在旧值而无人察觉。故不是把 README.md 塞进单模块检查（口径也对不上：一个是册页数、一个是
  求和），而是给 `check_page_ledger.py` 增设总数轴：各册实测求和 vs 全部总数声明，
  三种措辞（`共 N 页`／`实测 N 页`／`N 页讲义`）分别匹配——不用宽泛的 `(\d+)\s*页`，
  那会把一页纸里每个模块的 `<span class="pg">` 全部误吞。库根 README.md 标 required=False
  （非布局约定文件，别的库未必有）；总数轴仅在全部模块都读得出实测值时才判定，缺一册即跳过，
  否则会拿残缺的和去误判一堆正确声明。收尾用注入错值验证门禁真会 FAIL（改两处总数 → 两处
  都报出且退出码 1），**新门禁必须证伪一次才算数**。同步改 `illustration-rules.md` 第八节
  （总数从第 4 点的尾巴提为独立「总数轴」小节，列全五处落点与措辞）与 `patrol-rules.md` 页数账实核对。
- **v4.1（2026-07-20）**：**索引按「面中立 vs 面特有」拆两层**（用户提议：现在的 README 通篇是
  PPT 面的东西，它算 PPT 面的总览；根目录该单独出一个总纲）。动机：v4.0 建了 `Web-version/`
  之后，库根 README 仍是 19 册讲义的清单——一面的索引冒充全库门面，Web 面一起来必然打架。
  判据与 v4.0 把共享规则抽进 `shared/` 同源：**一块内容换到另一面就不成立，它就是面特有的**。
  四件：
  ①**根 `README.html` ＝ 总纲**（面中立）：知识地图（模块 × 层，两面共享的真源结构）、每日取用
  （`_prep/` 三件套）、两面入口卡（含各面状态，Web 面未建时走虚线待建样式）、库目录约定；
  ②**`PPT-version/README.html` ＝ PPT 面总览**：模块列表表格（页数/书单/更新日期）、预留扩充位、
  本面目录约定；表格模块链接去掉一级，加返回根总纲的 `../README.html`。模块 README 的返回链
  **不动**（仍 `../../README.html` 指根总纲——用户要回的是库的家，不是某一面的清单页）；
  ③**`check_page_ledger.py` 两处修**：模块表账改从 `PPT-version/README.html` 取（找不到回落
  库根，兼容 v4.0 及更早的库）；**并把三处「匹配不到就静默跳过」改成报 issue**。后者当场抓出
  一个已经发生的漏检——v4.0 号称"账实脚本双布局兼容"，但只给顶层 README 的行匹配加了
  `(?:PPT-version/)?` 前缀，**漏了 `_prep/` 全库一页纸的模板**；本库迁移把 `_prep` 链接改成
  `../PPT-version/<模块>/` 之后，一页纸这一账从此失配、被静默跳过，**"19 个模块四账一致"的
  绿灯实际只查了三账**。已补前缀并验证四账真读到（RAG/MCP/Security/AI-Ops 逐账打印比对）；
  ④`library-rules` 新增「索引两层」节（对照表 + 三条纪律：模块表账住面总览且移动索引必须同步
  脚本、知识地图只维护一份在总纲、模块返回链指根总纲），骨架章拆成根总纲/面总览两份并注明
  「讲义 PPT」列必须写成 `✅（NN 页）`（脚本靠这个形状取账）；`init-rules` 新增「v4.1 索引拆分」
  六步（可独立给已在 v4.0 的库用）；SKILL.md 第 1/5 步的「顶层索引」口径随之精确化为「面总览」。
  **教训（本版最值钱的一条）**：**静默跳过是比误报危险得多的失效模式**——门禁匹配不到目标时
  若不作声，绿灯的含义会从"查过且通过"悄悄退化成"没查"，而外部看不出区别。凡"找不到就跳过"
  的分支，一律改成报出来。这与 v4.0 修订⑩①「匹配条件要按语义定义、不能按字面形状定义」是
  同一个根：**匹配这件事，失配必须是一个事件，不能是沉默**。
  回归：本库四门禁全绿（布局 0 / 坏链 0，24 个 HTML 257 条链接 / 页数账 19 册四账真一致，
  合计 1732 页 / 书单账 19 册一致），根总纲渲染目检通过。
