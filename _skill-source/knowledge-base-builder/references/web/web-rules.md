# 网页版规则（Web 面：内容与结构）

网页版任务用这份（视觉令牌与质量门禁在 `web-design-system.md`，生成与验收时读）。
本文件是 v3.16 融合外部网页版项目抽取稿（`_reference/2026-07-20-网页版skill抽取稿.md`）
的定版：凡 kb-builder 已有权威出处的规则只写指针不复写；网页特有的能力全文落地。
**分批扩建已收官**（2026-07-21～22，八批做完全库 19 册）：MCP 样板册定稿封版后，每批 1–3 册、
批内每册完整走覆盖矩阵与判停清单，不做一次性全库转换——这个节奏被验证是对的，新增模块沿用。
收尾清单见第九·五节。

## 一、定位：一源两面，账本共享

PPT 是基底，网页是它的延伸和加强（尤其交互）；两面**同一受众**（读者画像以 KB-CONFIG 为准）。
站点根 = KB-CONFIG「Web 面根目录」（默认 `Web-version/`），模块内容真源在「PPT 面根目录」
（默认 `PPT-version/`）——目录布局定版见 `library-rules.md`；**动手前先跑
`scripts/check_kb_layout.py` 确认布局就位**（退出码 2 = 旧布局，先走 init-rules
「v4.0 布局迁移」再回来）。

- **两面关系的定版表述**：网页与 PPT **共享事实、术语、边界和来源关系；不共享目录、
  页序、卡片数、字号比例和信息节奏**。
- **分工**：网页负责搜索、稳定深链接、章节导航、渐进阅读、机制解释、证据上下文、
  响应式和持续更新；PPT 负责有意识的线性叙事、视觉聚焦、讲解与个人顺序阅读。
- **三不**：不把 PPT 纵向拼成网页；不把网页截图直接做成 PPT；不声称不存在的自动同步能力。
- **账本唯一（裁决定版）**：模块 `MANIFEST.md` 是两面共同的唯一账本——章节结构、时效性
  事实、来源关系、更新记录都以它为准。网页构建需要的机器可读注册表（模块表/章节表/术语表/
  来源表）一律**从 MANIFEST 生成**，禁止手工双写第二份。

## 二、从目标开始，不从模板开始

- 先明确读者要在网页上做什么（检索、学习、判断、答客户问），再选结构、组件、图表和交互；
  先建知识模型，再定网页形态。
- 大库建设 = 完整框架 + **至少一个足够深入的代表模块**（即试点样板册），样板定稿再推广。
- **分批扩建的选册判据**（2026-07-21 第二批定版，按收益排）：①内容性质里有**天然交互件**的册
  优先（第九节的判据在这些册上兑现得最实——推理册的显存计算器、模型格局册的选型筛选器都是
  规则里点名过的例子）；②**时效性事实密度高**的册优先（保鲜标注与定点复查徽标在网页上的
  收益随密度涨）；③用户高频取用的册优先。每批 1–3 册，批内每册完整走覆盖矩阵与判停清单。
- 读者页面不展示读者画像、编辑说明和制作规则——与「内部工作用语不进成品」（SKILL.md
  风格底线）同一条纪律。

## 三、内容深度模型（网页面的节奏，区别于 PPT 面）

网页长文的核心判断链：**讲清概念 → 理解机制 → 识别失败与边界 → 连接真实应用 →
形成方案或判断 → 回答真实问题 → 用证据与实验验证**。
（PPT 面维持自己的「概念→类比→案例→小结」节奏——两面不共享信息节奏。）

按主题需要覆盖八项，作为深度自查单：

1. 定义与责任；2. 输入与输出；3. 工作机制与因果关系；4. 生命周期、动作循环或关键组件；
5. 失败方式、适用边界与替代方案；6. 会改变的架构、选型、风险、成本或验收判断；
7. 实际应用、系统责任和客户价值；8. 验证方法、证据和下一步行动。

**术语出现不等于已经讲清**。理论和公式只在会改变真实判断时保留；先解释直观机制，
再考虑数学表达。

## 四、吸收 PPT 与外部材料（首次生成的方法论）

- **首次生成用「覆盖与缺口矩阵」**：逐项登记来源材料的——原始位置、知识点、关系、案例、
  问题、风险、动态事实、网页处理方式、PPT 处理方式、目标模块、来源关系；每项判定六种
  处置之一：**吸收并重组／独立扩展／合并／重画／拆分／有理由省略**。矩阵存目标模块
  `raw-data/`，命名 `YYYY-MM-DD-网页版覆盖与缺口矩阵.md`，是"借鉴 PPT 但不局限"的可执行形态。
- **通读讲义用 `scripts/kb_deck_text.py`**（按放映序导出每页文字），不要现写提取脚本——
  它规避了「文件号 ≠ 放映序」这个坑：插过页的册子两者会大面积错位（实测 Agent 册 122 页
  里 106 处不一致），按文件号读会把页码对错。
- **证据地位分自产与外来（裁决定版）**：本库自产讲义的事实经 MANIFEST 时效表与核实笔记
  背书，**直接可用**；外部材料（他人 PPT、白皮书、二次资料）只当**覆盖线索和学习材料**，
  不自动当公开事实证据——其中的数字与现状必须按工作流第 2 步独立联网核实后才能进网页。

## 五、信息架构

- **首页只承担**全局入口、搜索、知识地图和学习路径，不堆模块正文——现有顶层门户与
  `_prep/` 三件套是它的起点。
- 每个正式模块有稳定、可分享的独立地址；长模块具有**稳定章节 ID**（复用 MANIFEST 的
  稳定章节 ID 作锚点，不另造一套）、完整目录、当前位置指示、移动端等价入口和可复制深链接。
- 全站维护一份公开**来源清单**；正文通过稳定来源 ID 回链，来源须说明支持什么、不能支持什么。
- **同一概念只有一个主要归属模块**（判定用第 1 步「覆盖核对」的落位表），其他模块做足够的
  局部解释 + "相关模块"链接。
- 搜索尽量直达具体模块、术语、章节、问题和练习。
- 内容变化时同步复核：问答、证据、相关模块、学习任务和来源——这五项进 B 类收尾清单。

## 五·五、生成器由技能收尾跑，用户永远不手跑（2026-07-21 定版）

网页的知识地图/侧栏/关系网/保鲜看板/模块页脚都由构建脚本从 MANIFEST 生成。**跑它是技能
收尾的固定动作**（任何触碰 MANIFEST 或网页内容的任务，收尾必跑生成器再跑 `--check`），
产物随提交走——**用户拿到的页面永远是已生成好的**。
取证：用户看到页面占位与交接文档里的「请运行 build.py」后问"为什么还需要我自己去生成侧边栏"
——把维护命令写在读者面前，等于把机务转嫁给读者。这类文案按 `writing-rules.md`
「内部术语 → 成品说法」处理：占位一律写「本节自动生成」，页脚说「内容更新时自动重建」，
不出现命令与脚本名。

## 六、增补流程（B 类两面落地）

模块有增补时：**一次联网核实、一份大纲 → PPT 面落地 + 网页面落地 → 一次收尾两面对账**。
MANIFEST「最后更新」同时记两面落地情况；哪面欠着就显式登记待办报告用户，不许静默跳过
（同 A 类收尾「口径回刷」的纪律）。网页面的具体回刷项见第五节末条。

## 六·五、两面循环的边界（2026-07-20 定版，用户共同裁决）

两面构成一个双保险：**PPT 先把知识点整理清楚 → 网页版据此发挥并加深 → 反哺 PPT 查漏补缺**。
但循环必须可终止，否则「还能不能更好」这个命题永远为真，会一直转下去。

**先定完备标准的归属**（这是终止的前提）：**PPT 按讲义自己的标准就是完备的，网页版在它之上
更细**——不是 PPT 更粗、网页更全。所以「网页有而 PPT 没有」**不构成 PPT 的缺口**；只有
「PPT 按讲义标准本就该有却没有」才算。这一条定不下来，网页每细一层都会反推出一个假缺口。

**只有三类允许回流 PPT，其余一律留在网页版：**

| 闸门 | 判据 |
| --- | --- |
| **事实级** | 网页化时联网核实，发现 PPT 的数字/版本/现状错了或过期。这是纠错，必须回流 |
| **缺口级** | 深度模型八项里 PPT 真缺一项，且这一项**会改变读者的判断**（如「何时不该用 X」） |
| **结构级** | 归属错误或重复覆盖——同一概念两个模块都在讲主线 |

**明确不回流**：呈现形态（图 vs 表）、信息节奏、交互件、措辞打磨——本文件第一节
「不共享目录、页序、信息节奏」就是这条的挡板。

**单向收口**：PPT → 网页 → 回流 PPT → **停**。不许因为这次回流又回头改网页，那是第二圈。

**再启动只能由外部事件触发**，不能由「我又想到一点」触发：时效性事实到建议复查日、巡检报警、
用户新输入、上游模块结构变化导致串联边失效。

**收敛判据（可观测）**：某轮回流项里**事实级 + 缺口级都是 0**，说明该模块已收敛，封版。

**但封版只封内容，不封呈现**（2026-07-22 补，血的教训）：上面这条判据量的是**内容缺口**，
呈现规格（图/表/卡/段长）从来没被量过。结果是八批扩建里规格**按批次单调退化**——
样板册 5 图 17 卡 → 第二三批 1 图 4~11 卡 → 第四批以后 **0 图 0 卡**，而 19 册全部报
「已封版」，八道门禁一道都查不出。用户翻页时一句「都是文字罗列，没有交互、没有图片、
没有 flowchart」才把这根轴掀开。

所以**每批收尾要跑一次呈现体检**（`scripts/check_presentation.py`）：报每章的承载构成、
超长段清单、**并列项挤成一句**、每册图数。**它只报不判**——「非散文承载」的数量按内容
需求来，不设配额（这是「不设配额」铁律从问答轴到呈现轴的扩展，见 §七）。

**散文墙有两种形态，长的那种只是其中之一**（2026-07-22 再补）：第一轮体检只量了「一段
有多长」，于是 93 处超长段清零、报表全绿，用户翻到 Agent「拆开 agent 的三大件：规划是
大脑、记忆是笔记本、工具是双手」又说「没有突出重点」——那一段只有 80 字，长度轴永远
抓不到它。**一句里塞三个并列的知识对象，和一段里塞三个，是同一个病**，区别只在字数。
体检因此加了第二根轴：一段里 ≥3 个粗体并列项、段长却没超阈值。首跑全库 34 处（引言块
11、正文 23）。判据仍不是「见到就拆」——排除问答区那种粗体是日期的基线声明，也排除本来
就是行文的段落；**拆的是「N 件事」被压成一句**的那种。

**换承载的对号入座**（只换承载、一字不改事实，文字原样搬运）：

| 内容性质 | 换成 |
| --- | --- |
| 有序过程（N 段管线、N 步流程） | 流程图 |
| 取舍 / 对比（A vs B、两难） | 对比表 |
| 并列要点（三要点、四问、五件套） | 卡组；**引言块里**用 `.keypoints` 要点行（词 + 一行短注） |
| 有序层级（N 层架构、自上而下的栈） | 层叠图——**别排成并列列，那会把「层」读成「项」** |
| N 路产品对比且带一列数字（单价、时延） | 表——数字散在括号里读不出倍差，立成一列才一眼可比 |
| 有量级差的数字（带宽阶梯、token 账、时延拆账） | 等比长条——**量级差用长度表达，比读数字快一个数量级** |
| 有公式可算 | 接进交互件 |

**画图三条硬约束**：①颜色只用令牌（`--ink-900/500/400`、`--blue`、`--line`、`--card`、
`--ok`、`--warn`、`--red`），写死色值的那天就是深色模式坏掉的那天；②`viewBox` 内不许超界，
收尾用脚本量 rect/text 坐标；③**柱长必须与坐标轴一致**——2026-07-22 第一版画过一张
50% 的柱只有 25% 宽、坐标轴却标着 0/50/100 的图，属图表硬错误。

**回流是欠账，要在账本上挂着**：网页面补上一处缺口级内容时，MANIFEST「最后更新」必须显式
写明"这是缺口级、讲义侧尚未落地——待办"，直到讲义补完才销账。否则会出现"网页有、讲义没有、
谁也没记得"的状态——那正是双保险失效的样子（2026-07-20 MCP 册成本轴即按此登记）。

**判停清单**——四条齐即本轮结束，不再问「还能不能更好」：① 深度模型八项逐项有落点（能改变
判断，不是「提到了」）；② 时效性事实都带核实日期与不可外推边界；③ 问答过了第七节立题判据；
④ 门禁全绿（含 `web-design-system.md` §四 分层门禁）。

## 六·六、网页版优先做「纸面做不到的事」

同样的工时，先做 PPT 结构上不可能做的，再做锦上添花的。判据是**去掉网页这个介质它还成不成立**：

- **跨模块聚合**——19 册讲义各印各的，纸面无法互相指。关系网（讲这块还该带上哪几块）、
  全库术语表、跨册问答库都属此类。**窄屏下这类宽表要合并列**：出边与入边对读者是同一个用途，
  三列硬撑到手机上单行会高到 400px（2026-07-20 实测 403px，合并后 173px）。
- **时间序视图**——时效性事实分散登记在各模块账本里，纸面排不出「哪条该复查了」。
  **但首页只放"该管的"**（已过期 + 一个月内到期），长尾另开一页——首页是学习入口，
  不是维护看板；把全量看板摊在首页，来学东西的人先看到一屏"别的模块该复查了"。
- **可检索**——纸面靠目录和记忆，网页可以直达章节、事实、问题。**搜索索引必须含问答**：
  客户问的是一句话，读者搜的就该是那句话。只索引模块/章节/事实的搜索，搜「备案」「串答」
  这类问句里的词会一无所获（2026-07-22 实测，补入问答后索引从 398 条涨到 569 条）。
- **跨册问答库**——各册的「现场高频追问」分散在各自页面里，纸面与单页都无法互查。
  汇成一页时守一条：**它是索引不是副本**——只登记题目、出处与深链，答案留在原处。
  复制答案就等于给同一段话开第二个维护点，正是 §一「账本唯一」要防的。
  问答要有**稳定锚点**才谈得上分享：锚点用位序（`q-<模块ID>-<n>`）而不是文本哈希——
  问答是追加式的，位序稳定；哈希会在改一个错别字时把已分享的链接静默变成死链。
- **可追证**——每个结论一键回链到它的依据与来源，纸面只能给页码。
- **按需展开**——同一页对不同深度的读者给不同信息量。

**反面**：把讲义的图表原样搬过来、把小结页译成 HTML——这些换个介质也成立，优先级最低。
（取证：2026-07-20 首批网页版把 174 条串联出边、212 条时效性事实晾在注册表里没用，
页面却在重排讲义已有的表格；补上这两块之后，「网页版凭什么存在」才有了答案。）

## 六·七、新增标注

两件事，别混：

- **模块页给一行「本册最近改动 YYYY-MM-DD」**（页脚或页顶），由构建脚本从 MANIFEST 注入。
  页面内容的改动到此为止——**不给小节、不给段落逐个盖时间戳**，那只会让页面变成日志。
- **问答区新增一道题时**，条目旁给 `<span class="added">新增 YYYY-MM-DD</span>`；
  存量用问答区顶部一行基线声明覆盖，不逐条补。

理由、写法与"为什么这不算派生账"见 `shared/library-rules.md`「新增标注」（唯一出处）。

## 七、问答与证据纪律

- **立题判据**：问题只有在会改变方案、风险判断、验收方法、客户沟通或下一步行动时才独立
  存在；相近问题合并，不逐章机械配题，不同模块不追求相同题数（「不设配额」铁律的问答版）。
  **判据是写作纪律，不是页面元素**——不要把它做成分类标签贴在每道题旁边（2026-07-20 试过：
  10 道题分 5 类，「沟通」「下一步」各只有 1 题，**分类维度比样本还细，标签不承载信息**，
  已撤除）。判据的价值在于**立题时逼自己问一句"这题改变什么"**，答不上来就不立；
  读者要的是答案本身。题量大到几十道以上再考虑分组，那时分组才有统计意义。

- **网页面的问答区可以引一句说明判据**（如"只收会改变某个决定的问题——会改变技术方案、
  风险判断、验收方法、对客沟通或下一步行动"），让读者知道这里为什么没有面面俱到的题海。
- 每道重要问题按需要包含：结论短答、机制或权衡、下一问或验证动作、本题依据。
- **定量事实绑定五要素**：数据集或工作负载、模型或产品版本、条件、日期、**不可外推边界**——
  比 PPT 面「定量锚点」（ppt-rules）更严，网页面先行执行，后续视情回灌 PPT 面。
  **第五项最容易只落在样板册上**：2026-07-22 实测某库 84 条证据注里 51 条只有「核实 + 来源」，
  「不可外推边界」仅样板册 5 处、另一册 1 处，其余 17 册为 0。它恰是这五项里对客最值钱的一项
  ——客户拿你的数字去别的场景翻车，算你的。**写法**：证据注末尾另起
  `<b>不可外推边界</b>：…`，一句话说清「这个数字在什么条件下不成立」。
  **来源纪律同正文**：边界必须从页面与账本已记录的条件里读出来（是相对降幅不是绝对提升／
  厂商自报口径／论文基准环境／当期快照／不同维度的第一合不成总冠军），写不出来就说明
  这个数字的条件本来就没记全——该回去补条件，不是编一句边界。
  自查一行：`grep -c '不可外推边界' <网页根>/*/index.html`，某册为 0 即欠账。
- 动态事实优先核验一手来源，并登记复查周期（并入 MANIFEST「建议复查日」机制，不另建一套）。

## 八、语言（指针为主）

- 反 AI 腔写作纪律、术语中英对照与缩写展开英文全称（含边界判定）、语序铁律：**唯一出处在
  `writing-rules.md`（两面共用）**，网页文字全部适用，此处不复写。
- 网页特有增量：高频缩写在**当前阅读位置**给全称和简短说明，不把唯一解释藏在 hover 里；
  标题使用客观知识表达，不用"先确定""我们来看"这类编辑指令句。

## 九、交互纪律（每模块交互件进方案）

- **增益判据**：搜索、筛选、展开、切换、快捷键，必须帮助定位、比较、理解、追证或决策
  之一，否则不做。界面承诺与实际能力一致。
- **反模式**：假搜索、伪快捷键、装饰性切换、无信息增益的动画、只靠 hover 才能理解的内容。
- **交互件与模块结合**：第 3 步方案里为每个模块单列一条「本模块交互件」（如推理册的显存
  计算器、模型格局册的选型筛选器、协议册的报文逐字段注释），由内容性质决定形态，通用底座
  （全文搜索/对练抽题/术语总表/知识地图）放共享组件层。
- **形态就三类**，按内容性质挑：**计算器**（有公式可算，如显存账、成本账、遥测量）、
  **选型/分诊器**（有决策树可走，如场景分诊、形态定档、协议选型）、**自查器**（有清单可勾，
  如风险与合规自查）。判定标准是"讲义里已有的那套判断能不能被交互地走一遍"——
  **交互件不产生新事实，只是让既有判据可操作**（这条同时是覆盖矩阵里"网页版独立扩展
  不算缺口"的依据）。
- 组件集、公共渲染件、写法与验证纪律：见 `web-design-system.md` §三·六（唯一出处）。
- 可访问性与无 JS 阅读路径是门禁项，见 `web-design-system.md`。

## 九·五、每批网页版的收尾清单（2026-07-22 定版，八批实践收敛）

八批扩建下来这套动作一直靠执行者记忆，换人或隔月必漏步——固定成清单，**每批按序走完**
（对应 ppt-rules 的「一致性回刷清单」）：

1. **覆盖矩阵**：每册一份，存目标模块 `raw-data/`，命名 `YYYY-MM-DD-网页版覆盖与缺口矩阵.md`；
   逐章处置 + 独立扩展 + 缺口级判定（事实级／缺口级各给数字）。
2. **写页面**：从固定组件集搭；每模块一个交互件；问答标加入日；来源表给「支持什么／
   不能支持什么」。
3. **注册**：`build.py` 的 `WEB_DIRS` **只此一处**（键=模块 ID、值=目录名，页面路径与页脚
   注入路径都由它派生。此前是两份字典各写一遍，因值格式不同导致批量注册脚本直接产出
   语法错误、返工一轮——2026-07-22 已合一）。
4. **跑生成器**：`build.py` 再 `build.py --check`（第五·五节：用户永远不手跑）。
5. **过门禁**：`check_kb_layout` / `check_html_links` / `check_page_ledger` /
   `check_ebook_ledger` / `check_css_classes` / `check_prep_coverage` / `build.py --check` /
   `check_skill_package`（八道）。
   **脚本引用也要带版本戳**：`site.js` / `data.js` 与 `kb.css` 同戳，`bump_style_version.py`
   一并推进、`check_css_classes` 一并校验。此前只有 CSS 带戳，改完 JS 浏览器照吃旧缓存
   ——「改了但看不见」这个坑之前只堵了一半（2026-07-22 调本页查找时实测踩到）。
6. **验交互件**：脚手架多场景 + 已知答案对照（`web-design-system.md` §三·六）。
7. **记账**：每册 MANIFEST「最后更新」写网页版落地 + 矩阵位置 + 交互件一句话 + 事实级／
   缺口级结论；有欠账显式挂待办。
8. **里程碑回刷**：覆盖数或阶段一变，**门户三处表述会语义过期**，门禁查不出——
   逐个核对：网页版首页「做到哪儿了」、根 README 知识地图导语、`开始使用.html` 的网页版
   入口说明。取证：19/19 达成后这三处还写着 MCP 试点期的「其余模块还只有 PPT、定稿前不做
   全库转换」，七道门禁全绿照样没查出来（与 v3.9「页数账回刷清单」同族：那次漂的是页数，
   这次是覆盖状态）。
9. **提交推送**：门禁全绿再提交（`commit-and-push` 站着的授权）。

**清单之外的一条判断**：门禁报错先分清是**契约漂移**还是**条件分支未触发**——
19/19 达成那一刻 `check_css_classes` 报 `.ppt` 失联，根因是生成器里「仅 PPT」的 else 分支
不再触发，不是类名漂移；这类要按 `check_css_classes.py` 白名单的**第三类口径**登记
（写清触发条件与何时会回来），而不是删掉 CSS 定义。
