# 网页版设计系统（Web 面：视觉与门禁）

网页版生成与验收时读这份；内容与结构规则在 `web-rules.md`。与 PPT 面的
`ppt-design-system.md` 同分工：rules 管讲什么，design-system 管长什么样、怎么验收。

## 一、Portable 三铁律（裁决定版，网页面第一门禁）

整库（PPT + 网页 + 电子书）是一个**拷走即用**的分享单元。网页产物必须满足：

1. **纯静态、零运行时依赖**：无服务端、无 CDN 外链、无网络字体——断网/内网双击即开；
2. **file:// 协议可用**：浏览器在本地文件协议下会拦截 `fetch()` 和 ES module 加载——
   检索索引、题库、注册表等数据一律打成 `data.js` 用 `<script>` 引入；页面脚本不用
   浏览器端 module 化加载。**构建期工具链不受限**（Node/Python 构建脚本与 `_maintenance/`
   门禁脚本同地位），受限的是产物；
3. **纯相对链接**：面内互链、指向 PPT 与电子书的链接全用相对路径（`check_html_links.py`
   兜底检查）。**且必须链到具体文件，禁止目录链接**（如 `./mcp/`）——Web 服务器会自动打开
   目录下的 index.html，**file:// 不会**，浏览器只显示文件列表；2026-07-21 用户点 MCP 卡片
   "点不进去"即此因，而旧坏链判据只看"路径存在"、目录存在照样放行。`check_html_links.py`
   已把目录链接判坏。

## 二、可读性与知识密度

- **知识密度来自结构、关系、判断、案例和证据**，不来自缩小字号、堆卡片或塞表格。
- 首个笔记本视口呈现：主题、定义、内容概览、阅读入口和真实知识起点——**不是 PPT 式封面**。
- **分层阅读**：先给结论、机制和边界，再展开案例、权衡、诊断和证据。
- **字号底线**：中文连续正文、问答、机制解释 ≥14px；表格正文与次级解释 ≥13px
  （2026-07-21 用户两次下调定档——其惯用密度即 100% 缩放下的当前字阶，原 16px 档偏大两档）。
  **密度一致比字号本身更重要**：三页计算字号完全相同时，用户仍感到"网页版和 PPT 总览
  突然变大"——首页大部分可见文字是卡片里的 14–15.5px，内容页一进去是 30px 标题加 16px
  段落，同一套规则、观感差一档。字阶要**以密度最高的常用页为锚**整站对齐（本库以首页为锚：
  h1 收到 clamp 19–24、h2 17.5、表格 14），不要每类页面各配各的档。
  **分清"解释文字"与"标记"**：标签、徽章、面包屑、目录项、代码块不属于解释文字，可以小于
  14px——它们是导航与标记，不承载需要连读的内容。踩线的往往是**看着像标注、实则要连读**的
  那类（如每题下面的「依据 …」行、页脚说明段），2026-07-20 复查时抬了 6 处 13.5px→14px。
- 保护数字量词、缩写解释和专有名词的完整换行（不许折断）。
- 卡片随内容自然生长，只有真实比较才使用等高；图、卡、表、案例、章节、练习、证据和问答
  **都不设固定配额**（与配图「数量由内容深度驱动」同一条铁律，见 illustration-rules）。
- **任意数量布局**：组件必须支持 0、1、单数、双数和未来增减——不能为对称而删除、复制或
  捏造内容。

## 三、知识关系决定版面

「内容决定版式，设计系统负责统一」（ppt-rules 同名原则的网页版）。按关系选表达：

| 知识关系 | 表达方式 |
| --- | --- |
| 顺序、数据流、责任传递 | 流程图 |
| 反馈、重试、恢复 | 循环或状态机 |
| 层级、归属、边界 | 分层图或树 |
| 共同维度下的方案 | 比较矩阵 |
| 攻击、失败、诊断 | 路径或决策树 |
| 生命周期与发布门 | 阶段时间线 |
| 架构和系统边界 | 组件图或数据流图 |
| 没有关键关系的单一概念 | 正文、列表或简单提示 |

共享设计系统**只统一**字体、间距、证据样式、边界提示、可访问性和响应式底线；不同主题
必须允许不同核心视图，**不能全部退化成同一种卡片墙**。

**但「同一设计语言」的判据是组件级的，不是色板级的（2026-07-21 定版）**：任何两页并排看
必须像同一个人做的。只共享色板远远不够——用户并排打开四页评价"像是不同人设计的"，当时
四页确实共用一份样式表与令牌，**差在没共用组件**。统一手段是**全站固定组件集**：
页头带（.top）开场、区块标题（左蓝条 h2）、卡片（描边+浅投影）、表格进 .tw 容器、
色相字标、**交互件组件集（见 §三·六）**。新建任何页面从这套组件搭，不从空白 HTML 写起。

**导航模型：全站一条顶栏，不许有第二套（2026-07-21 定版）**。每页同一条 `.topbar`
（品牌回首页 + 网页版 / PPT 总览 / 实战包 / 学习路径 / 全库一页纸 / 开始使用，当前页高亮）。
教训：此前三套导航并存——网页版一套侧栏（窄屏塌成一堆药丸、与正文区块重复）、其它页一根
孤零零的返回链接、首页什么都没有；用户逐页指出"图一是重复，图三这里是没有"。两条推论：
- **侧栏锚点是反模式**：给"内容就在下面"的区块做侧栏锚点，等于把目录复制成第二份正文，
  窄屏塌陷后尤其像重复内容。长页内导航用模块页那种局部 `.toc`，站级导航只走顶栏。
- **顶栏上过的链接不再另做返回链/面包屑链**；面包屑只留不可点的位置说明（如「协议层 ·
  网页版样板」）。

**吸顶顶栏欠三笔账，一笔都不能漏（2026-07-22 定版，用户报的一个 bug 牵出三处）**：顶栏
是 `position: sticky; top: 0`，凡是「滚到某处」的行为都必须先把它的高度让出来。三处必须
挂在同一个值 `--topbar-h`（由 `site.js` 实测写入，`load` 后再量一次自我校正）：

| 谁 | 怎么让 | 漏了会怎样 |
| --- | --- | --- |
| 锚点落点 | `scroll-margin-top: var(--anchor-top)`，加在 `.sec` / `.layer` / `details[id]` 等一切可跳转元素上 | 点目录跳过去，标题被顶栏盖掉半行 |
| 目录吸顶位 | `top: calc(var(--topbar-h) + .9rem)` | 滚动时目录滑进顶栏底下 |
| 目录高亮的阅读线 | `bar.offsetHeight + 28`，**必须比锚点落点更低** | 跳转后当前节顶边还没越线，高亮停在上一节 |

第三行是最阴的：修完前两处后，若把阅读线设得比落点高，等于**把同一个 bug 换个数字重演**。
落点 16px、线 28px，这一对数要一起改。

**目录高亮不要用 IntersectionObserver 逐条 entry 设值**：观察带里同时压着两节时谁后触发
谁赢，表现就是「点了第 7 节、往下滚一点跳回第 6 节」。改成直接判定——**取最后一个顶边
已越过阅读线的节**，与滚动方向无关，`scroll` 事件 + `requestAnimationFrame` 节流即可。

**两栏要与页头留白**：`.cols { margin-top: 1.5rem }`。贴着页头分隔线开始时，目录卡片看着
像页头的一部分（用户 2026-07-22 指出）。

**布局居中（2026-07-21 两轮复核后定版）**：`.wrap` 与顶栏 `margin: 0 auto` + max-width。
曾因"品牌字每页位置不同"误改左锚定，后确认**位置漂移与字号差的共同真因是标签页缩放不一致**
（用户总纲页标签被 ⌘- 缩过、其余 100%；地址栏放大镜图标是铁证）——缩放归零后居中不漂移，
宽屏下居中更好看，按用户裁决恢复。**排查用户端"每页不一样"的顺序（修订版）**：
①先看地址栏有无缩放指示器（放大镜），让用户逐页 ⌘0 → ②看顶栏构建号是否一致 →
③同一标签页内用顶栏切换再比 → ④关掉重开标签页。**缩放差异在最前**：它能同时伪造
字号差与位置差，且任何 CSS 修改都追不平——2026-07-21 在它身上空转了三轮 CSS 级修补
（字阶收档方向本身没错，但"看不到变化"与"位置漂移"两条全是缩放层的）。

**同一内容全站只铺开一处（2026-07-21 定版）**：知识地图只住首页；网页版首页是「全库视图」
（搜索/关系网/保鲜看板），不复刻地图与每日取用。跨页到达靠顶栏，不靠把内容再贴一遍。
带注解的内容卡（如首页「每日取用」带说明的清单）与顶栏裸链接不算重复——前者是内容，
后者是导航。

## 三·五、设计方法：怎么做得好看（2026-07-20 并入）

> 本节方法取自 Anthropic 的 `frontend-design` 技能（Apache-2.0），按本库约束改写与增补，
> 不是原文转录。并入原因：网页版每次都要外挂一个设计技能太啰嗦，而它的方法与本节的门禁
> 是同一件事的两端——门禁管"不许难看"，方法管"怎么好看"。

**先定方向，再写代码。** 动手前回答四问：这个页面解决谁的什么问题；**语气取哪一个极端**
（克制的极简、编辑部/杂志、工业/实用、档案/资料库、粗野主义……）；有哪些技术约束；
读者会记住的那**一个**东西是什么。**关键不是强度而是意图**——大胆的繁复与克制的极简都成立，
半途而废的折中不成立。选定之后每个细节都服从它。

反 AI slop 的具体形态（比 writing-rules 的文字纪律更靠视觉侧）：

- **排版**：避开通用字体（Inter、Roboto、Arial、纯系统默认）。**本库的硬约束是 Portable 禁外链
  字体**，中文字体又大到不可能内嵌——所以性格不能靠"下载一个好字体"，只能靠**配对与处理**：
  衬线/无衬线搭配（如宋体标题 + 黑体正文，中文出版物的经典组合）、字重差、字距、小号大写
  英文题眉。这是本库对原方法最重要的一处改写。
- **色彩**：**主导色 + 锐利点缀** 胜过温吞平均分配的调色板。用 CSS 变量统一。
- **动效**：只用 CSS。**一次编排良好的载入错序显现，胜过满页零散的微交互**。必须尊重
  `prefers-reduced-motion`。
- **空间构成**：允许不对称、重叠、打破栅格、留白慷慨或密度可控——但**不得越过本文件 §二的
  可读性底线与"页面绝不横向滚"**：风格让位于这两条。
- **背景与细节**：营造氛围与层次，不要平涂色块。**图片与外部资源被 Portable 禁掉**，所以只能
  用纯 CSS 手段：渐变、极淡纹理、发丝线、阴影、装饰性边框。
- **复杂度匹配愿景**：繁复的方向需要繁复的实现；克制的方向需要克制、精确、在间距与字号上下功夫。

**与原方法的一处明确分歧（本库口径优先）**：原技能强调"每次生成都要不一样，绝不收敛到同一套
选择"——那是给一次性作品的建议。**本库是同一套长期资产的多个页面，跨模块一致性优先于每页
求新**：方向在样板册定稿时选定一次，后续各册沿用同一套令牌与版式语言，只让**内容**决定
版面差异（§三的知识关系映射表）。想换方向要整面重刷，不是逐册各选各的。

### 令牌与版式语言

**令牌表的唯一出处是 `shared/library-rules.md`「全库 HTML 视觉令牌」**——库内所有 HTML
（根总纲、面总览、模块 README、`_prep/` 取用页、网页版）共用同一套，定义在 `_assets/kb.css`
的 `:root`，本文件不复制第二份。方向＝**清爽导航站 / 工作台**（2026-07-21 换向，
经手教训见 library-rules 同节：**方向要跟读法走**——门户是快速取用，不是慢读杂志）。

版式语言：白底卡片矩阵（描边+浅投影+hover 抬起）、左栏分类导航（纯锚点、无 JS 可用）、
模块卡＝**色块字标（favicon 位）**+名称+标签+两行简介+元信息、层级七色相（字标/层标点/侧栏
色点同色）、页头统计条（JS 增强，数字从注册表算、不写死）；空栅格位不着色（§二「任意数量
布局」）。**导航站好看的来源是彩色小图标与节奏，不是排版技巧**——纯文字卡片一片灰白，
2026-07-21 第二轮被否即此因。

**教训**：本节曾自带一份令牌表，与 library-rules 的门户色板并存——结果一个库长出三套配色
（门户蓝、编辑部墨金、实战包的六色分类），用户一眼看出「三个 README 都是不同风格」。
**规则写「不得引入第三种」没用，样式分散在 28 个文件里就一定会分叉**；收进一个文件才治本。

## 三·六、交互件组件集与验证纪律（2026-07-22 定版，八批全库扩建后收敛）

每模块一个交互件是 `web-rules.md` §九的要求，本节管它**长什么样、怎么写、怎么验**。

### 组件集（属 §三 全站固定组件集的一员，定义在 `_assets/kb.css`）

| 类 | 角色 | 设计意图 |
| --- | --- | --- |
| `.tool` | 交互件外壳 | **无 JS 时整块隐藏**（靠 `.js-on .tool{display:block}`）——所以静态兜底必须留在正文，不能只活在交互件里 |
| `.controls` / `.ctl` / `.ctl.chk` | 输入区／单个控件／勾选型控件 | 标签与控件同属一个 `<label>`，点标签即聚焦 |
| `.hitcards` / `.hitcard` / `.tagline2` | 结果卡片栅格／单卡／卡内标签 | **DOM 顺序：标题 `<b>` 在前、`.tagline2` 作副标题、`<p>` 正文在后**——读者第一眼要看到结论不是分类；kb.css 里 `.tagline2` 的上紧下松边距就是按这个顺序调的 |
| `.kpis` / `.kpi` / `.kpi.warn2` | 指标块／单指标／越界告警态 | 计算类交互件的数字出口；`warn2` 只在越过规则给的建议区间时挂 |

**公共渲染件唯一出处：`Web-version/assets/site.js` 的 `window.KBUI`（`card` / `kpi`）**。
页面内联脚本只留数据与判定逻辑，不再自带渲染函数。
取证：这两个函数曾在 11 个交互件里各复制一份，到第八批 `card()` 已漂成**三个变体**——
参数顺序、DOM 顺序、`href` 是否可选各不相同（其中一种把标签放到了标题前面，与 CSS 边距
意图相反）。这是**组件契约漂移**，与 §四 第 3 层的类名契约漂移同族：分散的副本必然分叉，
收进一个文件才治本。2026-07-22 已把 20 份副本收敛成 1 份，73 处实参顺序一并对齐。

### 写法纪律（三条，都是踩出来的）

1. **类名一律写完整字面量**，禁止字符串拼接。
   `'<div class="kpi' + (warn ? ' warn2' : '') + '">'` 这种写法让 `check_css_classes.py`
   的静态扫描看不见 `warn2`，误报失联；正解是两条完整字面量二选一（`KBUI.kpi` 已按此实现）。
2. **数值钳位必须与输入同单位**。
   `Math.min(0.9, 用户输入的 20)` ——先钳位后除百，20% 被压成 0.9%；同单位写法是
   `Math.min(90, x) / 100`。这个 bug 七道门禁全绿，只有手算对照能抓出来（下条）。
3. **单价、系数、默认值一律可编辑，且注明「量级示例，非报价」**。
   会过期的数字不硬编码进交互件——这是「时效性事实带核实日期」在交互件里的等价要求。

### 验证纪律：已知答案对照 + 脚手架

**每个交互件上线前必须用一组手算可验的输入做对照**——这是决策器/计算器类交互件的
「评估集」。对照口径要落到能写进提交说明的一句话（如「默认 64 卡场景自建 1294 万、
云租 1060 万、盈亏 43%，恰落在讲义 40–50% 口径内」）。
判定器类没有算式，则对照**边界场景**：全不选、全选、互斥项——看输出条数与命中项是否符合规则。

**验证有两条腿，都要用**：

- **浏览器窗格**：验交互体感（点得动、焦点可见、窄屏不横滚）。
- **脚手架**（多场景数值验证，2026-07-21 第七批起为标准法）：把真页复制成同目录
  `_test.html`（相对路径的 `data.js`/`kb.css` 全部保真），尾部注入场景脚本——设值、
  `dispatchEvent(new Event("change"))`、把结果写进 `<pre id="test-out">`，再用无头浏览器
  `--virtual-time-budget` dump DOM，正则取回结果，**验完即删**。
  它不依赖窗格状态、可一次跑多场景、能与手算答案逐项对照。
  取证：窗格在长会话里会失稳（标签页被回收、navigate 报成功实未跳转、`location.href`
  跳转直接杀掉标签页），第七、八两批的六个交互件全靠脚手架完成验证。
  **重构公共件这类横向改动，用同一套脚手架取「改动前基线」再比对**——2026-07-22 收敛
  `KBUI` 时即用它证明 18 页文本输出零差异（只有 DOM 顺序按设计意图统一）。

## 四、分层质量门禁（Web 面的具体层；模型与使用纪律见 core-rules.md §三）

分层验收模型的唯一出处在 `core-rules.md` §三——**上层门禁全绿不等于真的能用**；
HTTP 200、构建成功、自动测试通过、题目数量、Git push 都不能单独作为完成证明。Web 面逐层：

1. **内容评审**：完整性、机制、边界、决策价值、主题针对性（对照 web-rules 深度模型八项）；
2. **证据评审**：来源可解析、动态事实已核验、支持范围与外推边界准确；
3. **自动检查**：构建、链接（`check_html_links.py`）、路由、来源时效、任意数量布局的
   回归用例；**外加扫一遍 HTML 里的 Markdown 残留**（`**粗体**`、`- 列表`、`## 标题`）——
   写惯了 Markdown 的手在写 HTML 时会漏带标记，浏览器原样显示星号，肉眼扫长页容易跳过
   （2026-07-20 补成本节时漏了一处 `**不是**`）；**再加样式契约对账**
   （`check_css_classes.py`）：CSS 定义的类与页面用的类必须互相认识——换肤把深色底从
   `pre` 挪到新类 `.wire-code`，页面标记没同批改，浅字浮在浅底上可读性归零而所有旧门禁
   全绿（2026-07-21 报文注释器事故）。**改类名契约＝改所有用它的页面，同一批落地**；
   可读性攸关的属性（深色底、对比色）优先挂在内容元素自身，别挂在可丢失的包装类上；
4. **真实视觉检查**：笔记本优先，同时查平板和手机；清单——长标题、长术语、展开后的问答、
   复杂表格、奇数/偶数卡片、无结果态、可见焦点、返回路径、**深色面板组件（代码块／
   报文注释器——全站唯一深底浅字的地方，换肤最容易漏检，2026-07-21 连续两轮换肤都没
   目检它）**；**换肤/整站改版的目检按组件类型过一遍，不是按页面抽查**；**外加一条方向自查**：页面是否
   服从 §三·五 选定的方向，有没有退化成同一种卡片墙、有没有该成图的地方还堆着表格；
   **并核对本轮判停清单**（web-rules §六·五）：八项覆盖、事实带日期与外推边界、
   问答过立题判据、门禁全绿——四条齐就收工，不再问「还能不能更好」；
5. **冷浏览器检查**：未预热的新标签用 **file:// 直达**首页和代表模块，检查标题、正文、
   交互、完成状态、控制台报错和字体/网络行为（任何外网请求都算违反 Portable 铁律）；
   **并导一次 PDF**——打印上下文不执行动画，而 `animation-fill-mode: both` 会把元素停在
   起始帧：2026-07-20 页头 `opacity: 0` 的起始帧导致**导出 PDF 时整个标题与导语消失**，
   屏幕截图完全看不出来。凡用了载入动效，必须给 `@media print` 强制复位（动画关掉、
   opacity/transform 归位），顺带给打印一套朴素配色（深底反白在纸上既费墨又难读）；
6. **发布检查**（若有推送/托管）：确认远端与公开站点使用同一精确提交。

可访问性底线（并入第 3、4 层检查）：键盘可用、触控可用、可见焦点、尊重减少动态效果，
**无 JavaScript 时保留基本阅读路径**（交互增强失效不阻断阅读）。

## 五、错误转化为系统能力（原则见 core-rules.md §五）

五件套纪律的唯一出处在 `core-rules.md` §五。Web 面的"单点特判"形态——**禁止**用
单模块 CSS 特判、固定数量分支或特定句子断言掩盖系统问题。
