# 初始化与迁移规则（第 0 步读不到 KB-CONFIG.md 时）

本技能是"引擎"，知识库是"数据"——包括个性配置 `KB-CONFIG.md` 也存在用户自己的库里。
这样一份技能可以分享给任何人，各人的库互不相干。第 0 步读不到配置时进入本流程，
先判断眼前是哪种情况：

## 情况一：全新目录（无模块痕迹）→ 开箱访谈

目录里没有 README.html、没有模块文件夹 → 这是新用户。一次性问完以下问题，**每题都给
默认值，用户全部"按默认来"也能得到完整可用的库**（默认值即下方模板中的值）：

1. **库放哪？** 默认当前工作目录，要写成绝对路径。
2. **这个库学什么领域？** 默认"AI 技术（大模型应用）"。选了非默认领域，就多问一句
   "想分几层来装知识"，帮用户定制书架（见下"层设计原则"）。
3. **你是什么角色、什么基础？** 决定讲义讲法和选书口味。默认"售前技术：有 Python/API
   基础、重概念与取舍、不啃重理论"。
4. **谁会考你？** 每章结尾有一页"对练页"，把最可能被追问的问题当面试题准备。
   默认"客户"；工程师常选"面试官"，内部分享型选"评审/同事"。
5. **要不要挂钩工作场景？** 默认"上云落地"（技术环节 → 云服务对照）；可换"自家产品对照"，
   或"无"（纯学习库，讲义里不出现落地节）。
6. **讲义拿来怎么看？** 决定 PPT 字号档位。默认"屏幕自读"（紧凑档：字小页密，
   自己在电脑上翻着舒服）；要投屏授课/汇报选"投屏演示"（大字档：正文 ≥16pt）。
7. **口味套装**：偏实践选书（理论 ≤ 2 本）、中文为主 + 术语中英对照、巡检阈值 90 天、
   季度巡检。默认全要，想改哪条单说。

访谈完成后依次生成：`KB-CONFIG.md`（模板见下）→ 顶层 `README.html` 骨架（知识地图用
访谈定的层，模块列表为空、留预留扩充位）→ `_maintenance/`（含一行说明的 README.txt，
并把技能包 `scripts/audit_pptx.py`、`scripts/check_html_links.py` 复制进来让门禁跟库走）。
最后提醒一句：**"要不要建个定期巡检的定时任务？"**（prompt 模板见 patrol-rules.md，
路径写死绝对路径）。库级产物（`_prep/` 实战包/一页纸等，见 prep-rules.md）是后期件，
初始化时**不建**，模块攒到 5 个以上再提议。

## 情况二：老库没配置（有模块痕迹）→ 静默迁移

目录里已有 README.html / 模块文件夹 / MANIFEST.md，只是缺 KB-CONFIG.md（多半是旧版技能
时期建的库）。**不要重新访谈**——答案都在库里：从现有 README、书单口吻、讲义风格、对练页
标题推断各字段值，生成一份配置草案给用户过目，用户点头后落盘。
原则：**迁移后的行为必须和迁移前一字不差**，迁移只是把隐含的个性显式化。

## KB-CONFIG.md 模板

字段名不可改——各规则文件按字段名取值。"值"列按访谈结果填：

```markdown
# KB-CONFIG · 知识库配置

> 这个库的"个性"都在这里：技能每次任务第 0 步读它，产出跟着它走。
> 想改口味直接改"值"列，下次任务生效；字段名不要动。

| 字段 | 值 |
| --- | --- |
| 配置版本 | 1 |
| 知识库根目录 | <绝对路径> |
| 知识领域 | AI 技术（大模型应用） |
| 读者画像 | 售前技术：有 Python/API 基础，重底层概念与取舍，不啃重理论 |
| 对练页提问者 | 客户（页标题：客户会怎么问（面试题）） |
| 工作场景挂钩 | 上云落地：技术环节 → 主流云服务对照，有真实关联才写 |
| 选书口味 | 偏实践少理论；理论类全库最多 1–2 本，标"可选读" |
| 语言策略 | 中文为主；缩写型术语首次出现展开英文全称（缩写，英文全称），译名视语境 |
| PPT 面根目录 | PPT-version |
| Web 面根目录 | Web-version |
| 知识库显示名 | AI 知识库（讲义封面眉题等成品对外署名，全库逐字统一） |
| 阅读场景 | 屏幕自读（紧凑档字号；投屏演示改"投屏"用大字档） |
| 巡检阈值 | 90 天 |
| 巡检节奏 | 季度 |

> 迁移老库时若缺「知识库显示名」「阅读场景」字段：显示名取现存讲义眉题的**多数派**写法
> （避免为统一字段反而改动多数成品），阅读场景按默认"自读"补写；补字段本身不算升配置版本。

## 知识地图层定义
| 层 | 放什么 |
| --- | --- |
| 应用模式层 | 用模型解决问题的模式 |
| 协议层 | 生态互操作标准 |
| 工程保障层 | 让系统可靠的工程手段 |
| 基础层 | 模型与提示词原理 |
```

## 层设计原则（非 AI 领域时帮用户定书架）

层是"稳定的认知维度"，不是话题清单：好的层五年不用改，模块在层内增删。3–5 层为宜。
检验标准：任何一个未来可能出现的新模块，都能不犹豫地放进**唯一**一层；如果经常犹豫，
说明层切错了维度。拿不准就从"基础原理 / 核心方法 / 工程实践"三层起步，库长大了再加层。

## v4.0 布局迁移（v3 旧布局 → 一源两面布局）

`scripts/check_kb_layout.py` 报退出码 2（模块在库根、无 PPT-version/）时走本节。这是一次
**呈现层手术**：只挪目录、改路径引用，不动任何内容——MANIFEST 内容态与 README「更新日期」
维持原样。迁移动手前先把顶层 README.html 与各模块书单复制一份进
`_maintenance/history/<日期>-迁移前留底/` 留底（迁移是批量改链接，坏了要能回退）。
**留底必须放在 `history/` 子目录下**：`check_html_links.py` 的排除目录含 `history` 却不含
`_maintenance`，而留底副本里全是旧路径——放 `_maintenance/` 根下会被门禁 2 当坏链报
（2026-07-20 首迁即报 41 条，全部来自留底自身）；放 `history/` 下也与库内
`raw-data/history/` 存旧版是同一个约定。步骤：

1. **建骨架**：库根建 `PPT-version/`、`Web-version/`（空目录即可）、`_reference/`；
2. **挪模块**：全部模块目录整体移入 `PPT-version/`（`mv`，目录内部结构不动）；
3. **撤除电子书文件**（v4.3 改向：旧库是"下载落地"，新契约是"只列链接"）：动手删之前
   **先把每份文件的出处落到书单对应条目上**——来源优先取 `ebooks/README.txt` 账本里的
   `来源:` 行，arXiv 类可由文件名编号反推 `https://arxiv.org/abs/<编号>`，都取不到的
   逐份联网查官方页面，**查不到就单独报给用户，不许静默删**。出处补齐后，把撤除当时的
   「文件名 → 出处」快照写进 `_maintenance/<日期>-原电子书馆藏出处存档.md`，再删
   `ebooks/` 目录（模块级与库级都删）。顺序反了就是不可逆的信息丢失。
4. **改路径引用**：①各模块 `电子书书单.md` 的 `./ebooks/…` 本地路径 → 换成第 3 步补齐的官方链接；
   ②顶层 README.html 模块链接 `./<模块>/…` → `./PPT-version/<模块>/…`；③`_prep/` 三件套
   与学习路径里的模块相对链接同步；④模块 README.html 里**只改指向库根的链接**
   （`../README.html`、`../_prep/…` → `../../…`）——**兄弟模块链接 `../<模块>/…` 不要动**，
   两端一起挪进 `PPT-version/`、相对关系不变，照"`../` 一律改 `../../`"执行反而会把它们改坏
   （2026-07-20 首迁实测：该改的 19 条、不该动的 96 条）。判据：**迁移中一条相对链接要不要改，
   看的是"两端是否一起挪"，不是链接长什么样**——这条判据对以后任何目录级手术都适用；
5. **KB-CONFIG 补两字段**：`PPT 面根目录 | PPT-version`、`Web 面根目录 | Web-version`
   （补字段不升配置版本，同显示名先例）；v4.3 起不再有「电子书馆藏」字段，旧库里有则删掉；
6. **四门禁收尾，全绿才算完**：`check_kb_layout.py` 退出码 0 → `check_html_links.py` 坏链
   0 → `check_page_ledger.py` 四账一致 → `check_ebook_ledger.py` 账实一致（后三个脚本
   已双布局兼容）；再抽 2–3 个模块人工点开门户链路（顶层 → 模块 README → 讲义/书单）确认；
7. **登记**：`_maintenance/` 存迁移记录（日期、挪了什么、门禁结果）；各模块 MANIFEST
   不必逐册记这次迁移（它不改模块内容），库级记录一份即可。

迁移与内容更新**不混批**：迁移批次里发现内容问题只登记不顺手改，改内容另开任务。

## v4.1 索引拆分（根总纲 / 面总览）

把库根 `README.html` 拆成**面中立的根总纲** + **PPT 面总览**（判据与两页各装什么见
`shared/library-rules.md`「索引两层」）。适用两种场合：从 v3 一路迁上来的库，在上面第 6 步
门禁之前做；已经在 v4.0 的库，单独按本节升级即可。改索引前先按 `library-rules` 落档旧
`README.html`（`_maintenance/history/<日期>-README.html`）。步骤：

1. **拆内容**：原 `README.html` 里，`<h2>模块列表</h2>` 表格 + 预留扩充位 + 本面目录约定
   → 新建的 `PPT-version/README.html`；知识地图、每日取用、库目录约定留在根总纲。
   `<style>` 原样复制给两页共用，**不要重画样式**——这是呈现层手术，视觉不该有可见变化。
2. **补两面入口**：根总纲加「两面入口」区，PPT 面卡链 `./PPT-version/README.html`，
   Web 面卡链 `./Web-version/`；Web 面未建时用虚线/待建样式，别给个死链就完事。
3. **改链接**：面总览表格里的模块链接去掉一级（`./PPT-version/<模块>/` → `./<模块>/`），
   加返回根总纲的 `../README.html`。**模块 README 的返回链不动**（仍 `../../README.html`
   指根总纲）；`_prep/` 三件套不动（它们链模块和根总纲，两者都没挪）。
4. **同步门禁脚本**：`check_page_ledger.py` 的模块表账改从 `PPT-version/README.html` 取
   （回落库根，兼容老库）；`check_kb_layout.py` 加 `PPT-version/README.html` 期望项。
   **这一步不能省**——账面搬家而脚本没跟上，它匹配不到就静默跳过，四账悄悄变三账。
5. **四门禁复跑 + 目检两页**：门禁全绿之外，两页都要实际渲染看一眼（新拆的页最容易出
   样式塌陷或表格串行）。
6. **顺手核对索引里描述布局的文字**：迁移改了结构，索引里"每个模块含 `ebooks/`"这类
   目录约定描述往往还停在旧布局——它描述的正是被改掉的东西，属本次范围，不算混批。
