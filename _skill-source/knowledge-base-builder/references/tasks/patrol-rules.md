# 保鲜巡检规则（C 类任务）

C 类不生产内容，只回答一个问题：**库里哪些"事实"可能已经过期了？** AI 领域的时效性事实
（版本号、"当前主流做法"、安全通告）半衰期常常只有几个月；讲义写完不管，一年后就是给读者挖坑。
巡检产出报告、由用户拍板刷新——巡检本身**永远不改任何成品**。

## 触发方式

- 用户一句话："巡检 / 体检 / 保鲜 / 查查哪些过期了 / 知识库维护"。
- 定时任务（节奏按 KB-CONFIG，默认季度一次）。定时任务的 prompt 是全新会话、没有上下文，
  必须自包含，模板（路径从 KB-CONFIG 抄**绝对路径**，别用相对路径）：

  > 对 <知识库根目录绝对路径> 执行保鲜巡检（C 类）：读各模块 MANIFEST.md，
  > 检查时效性事实是否过期，产出巡检报告到 _maintenance/。不要修改任何成品。

- 另有零成本兜底：A/B 类任务第 1 步会顺手扫各模块核实日期（见 SKILL.md），两者互为保险。

## 两级检查（先便宜后贵）

1. **离线筛选（不联网，秒级）**：读顶层 README 知识地图 → 逐模块读 `MANIFEST.md` 的
   "时效性事实"表 → 挑出满足**任一条件**的条目，按模块归组：
   - 核实日期距今**超过巡检阈值**（KB-CONFIG，默认 90 天）；
   - **今天 ≥ 该条的「建议复查日」**（登记时就知道的翻篇日——协议发版、优惠到期——
     不受 90 天阈值保护，到日即查；列缺省或为 — 的条目只按阈值筛）。
   一条都没命中 → 报告直接写"全库新鲜"，结束，不进第 2 级。
2. **联网抽查（只查被挑中的）**：对每条超期事实联网核实现状，逐条给结论：
   - `仍然成立` —— 回写 MANIFEST 核实日期即可；
   - `已过时` —— 写明现状与出处，给出"建议刷新"；
   - `拿不准` —— 写明疑点，留给用户判断。

### 附加呈现轴（离线，顺带跑，2026-07-14 增设）

时效两级检查之外，巡检顺带给全库讲义做一次**呈现契约自检**：跑库内
`_maintenance/audit_pptx.py`（缺失则用技能包 `scripts/audit_pptx.py`，并顺手复制一份进库；
批量跑遇坏文件/锁文件不会中断），报告附一节「呈现体检」：各册 PASS/FAIL 一览 +
系统性色板漂移/章数不一致/每章元素缺失等告警汇总。
只登记不动手——FAIL 项走 B 类修复，历史留痕照旧。人工已裁决的固定噪音（如 v3.8 保留的
补充型同题页对）登记在库内 `_maintenance/audit-exemptions.txt`，audit 输出 [豁免] 行、
不再计告警（v3.11 起；清单是库级文件，随库不随技能包）。呈现专项报告（如
`YYYY-MM-DD-呈现审查报告.md`）与巡检报告同存 `_maintenance/`，同样只增不改。

**门户坏链检查**（呈现轴的一部分）：跑库内 `_maintenance/check_html_links.py`（缺失则用
技能包副本并复制进库），检查全库 HTML（顶层/模块 README、`_prep/` 取用页）的相对链接
是否都指向真实存在的文件——模块改名、文件挪位、深潜指针失效都靠它兜底。坏链列进
「呈现体检」一节。

**页数账实核对**（呈现轴的一部分，v3.9 增设；v3.10 起有固定脚本）：跑
`_maintenance/check_page_ledger.py`（缺失则用技能包 `scripts/check_page_ledger.py` 兜底，
并复制一份进库），自动比对**四面派生页数账**——模块 README「主力成品 NN 页」/ 顶层 README
模块表 `✅（NN 页）`/（`_prep/`）一页纸 `<span class="pg">`/（模块）`MANIFEST.md` 讲义页数——
与讲义**放映序实测**（`ppt/slides/slide[0-9]+.xml` 计数，与 `audit_pptx.py` 报告页数同口径）
是否一致。配图/删页改了页数却漏回刷派生账是系统性漏洞（v3.9 前普遍存在，一册常同时挂三四个
不同页数）。**v4.2 起同一脚本兼查第二根轴「全库总页数」**——各册实测求和 vs 库根 `README.html`／
`README.md`、`_prep/全库一页纸.html`（标题+页脚）、`_prep/MANIFEST.md` 的总数声明；
此前这根轴无覆盖，单模块四账全绿而总数停在旧值是查不出来的。
对不上就登进「呈现体检」，建议动作＝按 `illustration-rules.md` 第八节四面回刷 + 回刷总数轴。
纯离线、秒级。

账实类检查（页数账/书单账/坏链）的账目全集与真源定义见 `library-rules.md`
「派生账登记表」（v3.13）——巡检是登记表的**事后兜底面**，账要在各任务收尾当场回刷。

### 附加书单轴（离线为主，顺带跑，v3.5 增设；v3.10 起有固定脚本）

跑 `_maintenance/check_ebook_ledger.py`（缺失则用技能包 `scripts/check_ebook_ledger.py`
兜底，并复制一份进库），自动做**账实核对**：`电子书书单.md` 里标"✅ 已下载"的行 ↔
`ebooks/` 实际文件清单，两边对不上（书单说有文件没有 / 文件在账上无记录）就登记。这步
纯离线、秒级。外链探活（对书单 🔗 行发请求确认未失效）**可选**——条目多时只抽查
"阅读建议"点名的高优先条目；失效只登记建议替换渠道，不当场改书单。结果附报告
「书单体检」一节。

## 巡检报告

存 `_maintenance/YYYY-MM-DD-巡检报告.md`，只增不改（和 raw-data 一个脾气：旧报告永不回改，
新巡检出新报告）。固定结构：

```markdown
# 知识库巡检报告 · YYYY-MM-DD

## 总览
| 模块 | 登记事实数 | 超期数 | 本次结论 |
| --- | --- | --- | --- |
| RAG | 5 | 2 | 1 条已过时，建议刷新 rag-hybrid |

## 明细（仅列超期条目）
| 事实 | 章节 ID | 原核实日期 | 现状结论 | 建议 |
| --- | --- | --- | --- | --- |
| …… | rag-hybrid | 2026-07-08 | 已过时：…（出处） | 刷新该章 |

## 建议动作
1. 刷新 RAG 第 7 章（rag-hybrid）：……（预计动到哪些页）
2. 其余"仍然成立"条目已回写核实日期，无需动作。
```

## 巡检之后

- 用户圈定要刷新的范围 → 逐模块走 **B 类更新**（旧版自动进 `raw-data/history/`，
  改完回写 MANIFEST）。
- "仍然成立"的条目直接回写 MANIFEST 的核实日期——这不算改成品，巡检可以自己做。
- 巡检发现的结构问题（如某模块缺 MANIFEST、命名不一致）也写进报告"建议动作"，但同样只建议不动手。

## 节奏建议

**默认季度全量巡检 + 日常顺手体检**（节奏与阈值在 KB-CONFIG 可调），别轻易做成月度：
个人库的变化频率撑不起月度巡检，
噪音会淹没信号，用户会开始忽略报告——巡检制度死于"狼来了"。
