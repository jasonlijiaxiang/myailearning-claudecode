# 对外文字通用纪律（两面共用）

本技能产出的一切对外文字——讲义正文与对练问答、模块/顶层 README、`_prep/` 三件套、
书单简介、网页版页面文字——都按本文件执行；本文件是这些纪律的**唯一出处**（v4.0 从
ppt-rules 抽出，PPT 面与 Web 面共用）。两面规则文件只写各自面特有的落点与呈现形态。
**加载时机**：动笔写任何对外文字之前——流程第 3、4 步、Web 面任务、D 类库级产物、
文案打磨类任务都读。

## 反 AI 腔写作纪律（v3.15 定立）

**适用范围**：本技能产出的一切对外文字——讲义正文与对练问答、模块/顶层 README、`_prep/`
三件套、书单简介、网页版页面文字——**生成时就按本节执行**，收尾再按同一判据快速自查
（见节末）。prep-rules / ebook-rules 与两面规则文件（ppt-rules / web-rules）不复写本节，只挂引用。

**"AI 生成感"的判定口径**：表达过度抽象、概念堆叠、句式高度规律、语气刻意完整、反复总结、
喜欢制造对立、使用看似专业但不帮助理解的词语。

产出时执行：

- 每句话承担明确的信息作用；没有新增内容的铺垫、重复和总结直接删。
- 用准确、具体、自然的中文，让读者直接理解实际对象、动作和关系。
- 术语去留由技术必要性决定，必要时给简明解释，不简单删除或降级——与下文「语言策略」的缩写展开全称规则**并行不悖**：全称规则是术语纪律，不属于本节反对的"固定词表
  机械替换"。
- 句式和节奏按内容自然安排，全篇不呈现统一模板；不刻意口语化、不追求表面"人味"，而是像
  真正理解内容的人思考后写出来的。
- 不增加事实之外的判断或承诺——与既有铁律「实测不估算」（见 ppt-rules「定量锚点」）、「内部工作
  用语不进成品」（见 SKILL.md 风格底线）同族，此处引用即可，不复写成第二份。
- 结合主题、读者画像（KB-CONFIG）与使用场景自行判断，不按固定词表机械替换。
- **用词贴用户的日常语感，回避行政腔／公文腔**。已知锚点样例（用户两次明确反馈）：「台账」
  他从来不用——行文一律改用「清单／记录／账」（如"吸收台账"在正文里称"吸收清单"）；给
  用户日常取用的东西起名用生活词（先例：错题本，不叫复习台账）。这些锚点是**语感校准，不是
  固定词表**——遇到同样"看着正式但用户不会这么说"的词，按同一语感判断处理。既有内部
  文件名（如 `_reference/吸收台账.md`）按 2026-07-12 的裁决维持不动、只改行文称呼；用户
  想连文件名一起改时，单独问一句再动。

**收尾自查（流程第 5 步挂钩）**：定稿前把本次新增/改动的对外文字按判定口径过一遍，发现即改。
自查是快速通读，不是重写循环。

**存量口径**：既有成品不因本节自动回刷，本节对新增与今后的改写生效；若认为某册/某份材料
特别值得单独立项打磨，列清单交用户拍板，不自行动手。

**打磨类任务动手前必做「征兆密度体检」（v3.16）。** 用户点名要对存量做文案打磨（无论范围
多大）时，**先量后改**：①按判定口径扫出全库/目标册的征兆命中，给出**密度**（命中数 /
总页数或总字数）；②抽一册**通读**，把命中逐条放回上下文判真伪——判定口径里的"制造对立"
指的是**虚假二分**，不含承载真实区分的对照句式，后者是讲义最有效的表达之一，**误伤即质量
倒退**；③拿这两份数据向用户复述范围建议，再动手。**密度低就如实说"不值得整体重写"并给
定点清单，即使用户已经选了大范围**——用户选范围时手里没有密度数据，给出数据、说明理由
才是本技能该做的事，照单盲刷不是。（取证：2026-07-20 实测全库征兆仅 47 处/1732 页，其中
29 处"不是…而是"逐条判定全是实打实的区分，真该改的只有 11 处踩语感锚点的词。）

**兼作"文案打磨"类 B 任务的执行标准**：用户点名"给某册/某份材料做文案打磨"时，按判定
口径**整体改写**——先理解内容再改，不逐词替换——交付改写稿 + 主要问题与修改方向说明。

## 语言策略：术语中英对照 + 缩写展开英文全称（以 KB-CONFIG 为准）

按默认策略：重点技术名词**首次出现时给中英对照**，格式 `中文（English）`，如 `重排序（Reranking）`、
`向量检索（Vector Retrieval）`、`嵌入（Embedding）`。原因：售前读者查英文资料、和客户/原厂对话时
都会碰到英文原词，中英对照能让他一眼对上号、不至于"知道中文却听不懂英文"。**正文仍以中文为主**，
英文只是括注，不要整段英文；同一术语后续再出现直接用中文即可，不必反复标注。

**缩写型术语首次出现必须展开英文全称（2026-07-20 定版）**。格式「缩写，英文全称」，全角逗号分隔：
- 术语条目/对照页写法：`RNN，Recurrent Neural Network`；
- 嵌入正文句中时放括号内：`……位置编码（RoPE，Rotary Position Embedding）……`。

中文译名**视语境**，两条判据：①正文已在用中文讲这个概念（同页或前文已有该概念的中文说法）→
不再附译名，避免同一信息三写；②生僻术语（读者大概率没有现成中文对应词）→ 补第三段，如
`RNN，Recurrent Neural Network，循环神经网络`。同一册内后续出现只用缩写即可；API、GPU、HTTP
这类日常词汇不过度标注。**不合格样例**：只写缩写（通篇 `RoPE` 却始终不给全称）；只做
"缩写↔中文"对照而不给英文全称（`RNN（循环神经网络）`——读者仍不知道 RNN 缩的是什么）。
原因：缩写是客户与原厂口中最高频的词形，展开全称读者才"知其所指"，也是他查英文资料的入口。
**图内豁免**：信息图/图表内的缩写不强制展开（图版排版空间受限），由正文文字页和开篇
"重点术语对照页"兜底。

**边界判定（v3.14）——什么算"缩写型术语"**：指业内以缩写形态流通的**技术概念**（RAG、SFT、
TTFT、MoE 这类）。以下**不属于**，不强制展开：①**专名**——组织/机构（NIST、CNCF）、产品/
框架/模型（vLLM、CUDA、BM25、GPT-4）、基准/数据集（MMLU、SWE-bench）、硬件型号（H100、
NVL72）——它们是名字不是概念，展开无助理解；②**单位与规格记号**（TFLOPS、FP8、GB/s）；
③**日常词**（API、GPU、HTTP、SLA 这类读者画像下的常用词）；④**模块名交叉引用**——知识
地图/书架表/串联页里出现的本库模块名（"MCP 模块""RAG · Agent"）是引用不是术语使用，豁免。
拿不准时问一句"读者需要知道它缩的是什么才能理解吗"——需要才展开。面特有落点见文末。

**中英对照不只适用于正文**：图示节点、流程步骤、表头、图例里的关键术语同样要给英文
（节点内中文为主、英文小一号），如流程节点「切分 Chunking」「重排 Reranking」。原因：图是
讲义里被单独截图、单独讲解最多的部分，图里的术语比正文更需要让读者对上英文原词。

**语序铁律（2026-07-13 修订）：凡「翻译对」一律中文在前、英文在后**——正文括注、标题、
封面章节条、导览、页眉页脚、图示节点、表头皆同（「切分 Chunking」对，「Chunking 切分」错）。
四类**例外**保留英文领头：①缩写（MCP、DRA、MIG(多实例 GPU)——缩写+中文全称注释是惯例）；
②协议/规范对象名（Task、Tools、Agent Card——规范里英文就是本名）；③页面主旨即教学英文
口径本身的术语（如「open weight ≠ open source」「deflection vs resolution」——此时英文是被
讨论的对象）；④括号内非翻译性备注（如「token（而非按请求）」）。**同页/同表的兄弟条目必须
同一语序**。背景：2026-07-12 全库排查曾清出 33 处翻转，根因就是此前没写死这条。

**面特有落点与形态**：PPT 面的展开**落点纪律**与开篇"重点术语对照页"口径见
`ppt-rules.md`「语言策略 · PPT 面落点」；Web 面"高频缩写在当前阅读位置给全称、
不把唯一解释藏在 hover"见 `web-rules.md` 第八节。
