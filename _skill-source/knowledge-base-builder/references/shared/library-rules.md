# 库结构与账本（library-rules）· 布局 · 模块模板 · 命名约定

流程第 1 步（挂载/定位）和第 5 步（收尾）用这份。核心目标：**整个知识库一个模子刻出来**，
新模块加进去毫不违和，任何一块都能被单独更新而不牵动其它。

## 知识库定位

知识库由多个"主题模块"组成，一个一个往里加。它有两个不变量：

- **一个根总纲 + 每面一个面总览**（v4.1 拆分，见下）。根 `README.html` 是面中立的总入口，
  面总览列出本面所有产出、并明确"预留扩充位"——让人一眼看出这是个会长大的库。
- **每个模块结构统一、内部章节相互独立**。相互独立是关键：更新 MCP 的某一节，不该逼你重读 RAG。

## 库目录布局（v4.0 定版：一源两面；v4.1 拆总纲）与模块统一模板

**v4.0 布局**——模块目录收进 `PPT-version/`（内容真源所在地），网页派生站点在
`Web-version/`。**书单只列正规渠道链接，库内不存放电子书文件**（v4.3 起，见 `ebook-rules.md`）。
**v4.1 补一刀：索引按"面中立 vs 面特有"拆两层**——根 `README.html` 只留面中立的东西
（知识地图、每日取用、两面入口、库目录约定），各面的产出清单下沉到该面自己的
`README.html`。理由与 v4.0 把共享规则抽进 `shared/` 是同一个判据：Web 面一起来，
"模块讲义 19 册各多少页"这种东西显然只属于 PPT 面，把它放在总入口就是让一面的索引冒充全库门面：

```
<知识库根目录>/
├── KB-CONFIG.md                  # 个人配置（第 0 步读它，见 init-rules.md；v4.0 增三字段：
│                                 #   PPT 面根目录 / Web 面根目录 / 电子书馆藏）
├── README.html                   # 【根总纲·面中立】知识地图 + 每日取用 + 两面入口 + 库目录约定
│                                 #   （v4.1：模块清单不再住这里，见下「索引两层」）
├── PPT-version/                  # PPT 面 = 内容真源所在地：模块目录整体在此
│   ├── README.html               # 【PPT 面总览】模块列表表格（页数/书单/更新日期）+ 预留扩充位
│   └── <模块名>/                  # 一个主题一个文件夹，如 RAG/、MCP/
│       ├── README.html           # 模块内索引：本模块章节列表 + 状态（HTML，给人看）
│       ├── MANIFEST.md           # 模块清单：章节稳定 ID、时效性事实、串联出边——两面共同账本（见下）
│       ├── 电子书书单.md          # ① 精选电子书书单（索引页；文件路径指向馆藏，见 ebook-rules.md）
│       ├── raw-data/             # 原始素材留痕：历次输入与相关数据，按日期存 .md（见下）
│       │   └── history/          # 成品历史版本：每次覆盖/改写成品前，旧版先存这里（见下）
│       └── <模块名>-讲义.pptx     # ② 讲义式 PPT（PPT 面主力成品，见 ppt-rules.md）
├── Web-version/                  # Web 面 = 派生站点根（web-rules / web-design-system；试点前可为空）
├── _reference/                   # 外部参考输入（书单只列链接，不存电子书文件）
├── _prep/                        # 库级产物三件套（见 prep-rules.md）
└── _maintenance/                 # 库级维护：巡检报告、门禁与工具脚本（见 patrol-rules.md）
```

**布局就位检查**：Web 面任务、迁移收尾、巡检呈现轴都跑 `scripts/check_kb_layout.py`——
退出码 0=就位、1=残缺（含迁移半途）、2=v3 旧布局待迁移（老库迁移步骤见 init-rules
「v4.0 布局迁移」；老库在迁移前仍按旧布局工作，账实脚本双布局兼容）。

以 RAG 为例：`PPT-version/RAG/` 下直接放 `README.html` 和 `RAG-讲义.pptx`，用户历次的
原始输入放在 `PPT-version/RAG/raw-data/`；RAG 书单里的资料只给官方链接，不落地文件。

### 索引两层：根总纲 vs 面总览（v4.1）

**判据一句话：一块内容如果换到另一面就不成立，它就是面特有的，不该住根总纲。**

| | 根 `README.html`（总纲） | `<面>/README.html`（面总览） |
| --- | --- | --- |
| 定位 | 面中立的总入口 | 本面产出的清单页 |
| 装什么 | 知识地图（模块 × 层，两面共享的真源结构）、每日取用（`_prep/` 三件套）、两面入口卡（含各面状态）、库目录约定 | 本面全部产出的表格（PPT 面：模块 / 一句话 / 电子书 / 讲义页数 / 更新日期）、预留扩充位、本面目录约定 |
| 不装什么 | 任何一面的产出清单、页数、册数明细 | 知识地图、每日取用（面中立，只在总纲维护一份） |

三条纪律：

- **模块表账面住面总览**。`check_page_ledger.py` 的"模块表账"从 `PPT-version/README.html`
  取（找不到则回落库根 `README.html`，兼容 v4.0 及更早的库）。移动索引时**必须同步这个脚本**
  ——账面位置变了而脚本没跟上，它匹配不到就跳过，四账会悄悄变三账（v4.0 首迁即踩，见 CHANGELOG v4.1）。
- **知识地图只维护一份，且由生成器注入，不手写**（2026-07-21 起）：根总纲与网页版首页的
  知识地图都由构建脚本从各模块 MANIFEST 生成（同一渲染器、只差链接前缀），手维护版本已废止
  ——包括 2026-07-11 定的「六行展示口径」及其"勿修正"裁决，**该裁决随手维护地图一并作废**：
  它当年防的是人手改乱，生成之后展示口径与机器口径天然一致，无需再分两套。
- **模块 README 的返回链指根总纲**（`../../README.html`，措辞"返回知识库首页"）：
  用户要回的是库的家，不是某一面的清单页。

## raw-data/ 原始素材留痕（每次任务必做）

这个模块是"讲义/书单"的**原材料仓库**。用户每次的输入——一个概念、一个名词、一段和大模型的
对话、或一份资料——都是宝贵的一手素材，值得原样留存：一来可追溯"这页讲义/这条书单是从哪来的"，
二来日后回看/复用/再加工都靠它。所以**每次任务（无论 A 还是 B），在动讲义/书单之前，先把本次输入
存一份**到 `<模块名>/raw-data/`：

- 文件名带日期，便于追溯：`YYYY-MM-DD-<简短主题>.md`，如 `2026-07-07-reranking提问.md`。
- 内容：原样粘贴用户这次给的东西（对话就存对话、概念就存那段描述、资料就存链接+摘录），
  顶部加一两行元信息（日期、这次据此做了什么：新增/更新了哪节）。
- 相关数据（示例代码、截图说明、外部片段等）也可一并放进 `raw-data/`。
- 只增不改：历史输入是流水账，别去改写旧的，新的另存一份。

## raw-data/history/ 成品历史版本留痕（改成品前必做）

`raw-data/` 存的是"输入"，`raw-data/history/` 存的是"旧的输出"。知识库会被反复更新——
技能升级后重刷旧内容、往已有章节里增补新料，都会改写讲义 PPT 这类成品。旧版不是垃圾：
它记录了"当时讲到了哪一步"，万一新版改坏了还能回滚、对照。所以：

- **凡是本次要改写/重新生成的成品文件**（讲义 PPT、`电子书书单.md` 等），动手之前先原样
  复制一份到 `<模块名>/raw-data/history/`（没有这个文件夹就建一个）。
- 存档命名：`YYYY-MM-DD-<原文件名>`，日期是"被替换的那天"，如 `2026-07-08-RAG-讲义.pptx`。
  同一天替换多次就加序号或字母后缀顺延：`2026-07-08-2-RAG-讲义.pptx` / `2026-07-08b-RAG-讲义.pptx`。
- **重名保护（2026-07-13 修订）**：写入存档前**必须先检查目标文件名是否已存在**，存在就顺延
  后缀，**禁止用会静默覆盖的复制命令直接写入**（脚本里应"存在即报错"）。教训：2026-07-12
  曾因 cp 静默覆盖丢失一份同日中间版存档。
- **顺延取"已有最高之后的下一位"，不回填中间空档（2026-07-20 修订）**：早先丢档会在同日
  序列里留下空档位（如 MCP 的无后缀档位就是被 cp 覆盖丢掉的），按"第一个空位"顺延会把新档
  填进墓碑档位，文件名排序上反而排到旧档前面——同日四册踩中。`kb_archive.py` 已按此实现，
  手工落档同理。
- 与 raw-data 一样**只增不改**：history/ 里的旧版永远不再改动，要看历史按日期翻即可。
- 本次输入的存档 .md 元信息里顺手记一句"旧版已存档为 `history/xxx`"，这样
  输入 → 旧版 → 新版三者相互可追溯。
- `README.html` 等索引页只是状态页、随做随更，不必存档。

## MANIFEST.md 模块清单（机器读它，人读 README.html）

每个模块一份"户口本"：收尾（流程第 5 步）时更新它，C 类巡检读它。它解决两个具体问题：

- **章节编号会漂，ID 不会。** RAG 已经从 5 章长到 9 章过一次；跨模块引用若写"见第 8 章"，
  一次增补编号就全错。所以每章分配一个**稳定章节 ID**（`<模块小写>-<语义词>`，如 `rag-hybrid`），
  跨模块引用一律写 `模块#章节ID`（如 `agent#agent-context`），"第 N 章"只作展示。
- **巡检要便宜，靠日期字段。** 讲义/书单里"会过期的事实"登记成表、每条带核实日期，
  巡检先离线筛日期、再联网抽查超期的，不必每次重读全部成品。

固定格式（四段，全是表格，便于逐段解析）：

```markdown
# <模块名> · MANIFEST（模块清单）

## 模块信息
| 字段 | 值 |
| --- | --- |
| 模块 ID | rag |
| 所在层 | 应用模式层 |
| 建立日期 | 2026-07-07 |
| 最后更新 | 2026-07-08 |
| 产出 skill 版本 | v2.0 |

## 章节清单
| 章节 ID | 现编号 | 标题 | 状态 | 最后核实 |
| --- | --- | --- | --- | --- |
| rag-what-why | 第 1 章 | 是什么/为什么 | ✅ | 2026-07-07 |

## 时效性事实（巡检盘查对象）
| 事实 | 章节 ID | 核实日期 | 来源 | 建议复查日 |
| --- | --- | --- | --- | --- |
| 混合检索是生产最低标配，RRF k≈60 | rag-hybrid | 2026-07-08 | ES/Qdrant 官方文档 | — |
| 新版规范 2026-07-28 发布（RC 已锁定） | rag-hybrid | 2026-07-12 | 官方博客 | 2026-07-28 |

## 串联出边
| 本模块章节 | 指向 | 关系 |
| --- | --- | --- |
| rag-agentic | agent#agent-context | Agentic RAG ↔ 上下文工程互为参照 |
```

登记"时效性事实"的口径：版本号、"当前现状/主流做法"、安全事件与 CVE、产品格局、
带日期的承诺——一切"半年后可能不再成立"的表述都算；纯原理（如"RRF 是倒数排名融合"）
不会过期，不用登记。指向尚不存在的模块的串联，在"指向"列标"（候选）模块名"即可。

**「建议复查日」列（可选，没有就填 —）**：登记时就知道翻篇日期的事实——协议新版发布日、
优惠到期日、大版本迭代窗口——把日期显式写进这列。为什么：巡检默认按"核实日期距今超阈值"
筛选，一条 07-12 核实、07-28 就翻篇的事实要等到 10 月才被盘查；写了复查日，巡检一级筛选和
A/B 类顺手扫都会在"今天 ≥ 建议复查日"时点名它（见 patrol-rules 与 SKILL.md 第 1 步）。
只知道大概窗口就写窗口首日（如"8 月上旬"写 2026-08-01）。老 MANIFEST 缺这一列不算违规，
下次更新该模块时顺手补上。

## 新增标注：问答题记下它是哪天加进来的（2026-07-20 起，只对今后生效）

**只标问答题**（讲义对练页的题、网页问答区的条目），章节与小节不标。

**为什么范围这么窄**（用户 2026-07-20 收窄，原方案含章/节）：问答题是会累积的**离散条目**，
一册几十道混在一起，"哪几道是这次新加的、我还没练过"完全看不出来——这正是标注要解决的问题。
章节不一样：**它本来就有覆盖**（MANIFEST「最后更新」记着增章流水、模块 README 有更新日期），
而且章少、体量大，加没加读者一眼看得出。**给已经看得出的东西再加一层标注，只会稀释标注的信号。**

**怎么写：**

| 位置 | 写法 |
| --- | --- |
| 讲义对练页 | 题目后缀「新增 YYYY-MM-DD」 |
| 网页问答条目 | `<span class="added">新增 YYYY-MM-DD</span>` |

**字段语义是「加入本题库的日期」，不是「原创日」**——这一条先定死，否则会像
2026-07-20 那次一样把自己绕进去（当时以为给存量盖今天是写假日期，其实是把语义想窄了）。

**存量按两级取值：**

1. **能追出真实日期就用真实的**——这是最好的情况。追法：题目出自讲义哪一章，就查该模块
   MANIFEST 的建立日期或该章「最后核实」；讲义里本来就有的题，用讲义那一版的日期。
2. **追不出来就用登记日**（打标那天）。有个日期比没有强，语义上也站得住——它确实是
   "这道题进入本题库被记录下来"的那天。

不要为存量硬考据到分毫，也不要因为怕不准就一条都不标。

**取证（2026-07-20 MCP 册）**：10 道题里 8 道来自讲义首版对练页 → 标模块建立日 `2026-07-08`；
2 道是网页版新立的（讲义全文检索确认没有）→ 标 `2026-07-20`。**比统一盖今天的信息量大得多**：
一眼看出哪两道是网页版加深出来的。存量标注前值得花几分钟做这个区分。

**模块级「最近改动」另算**：每个模块页在页脚（或页顶）给一行「本册最近改动 YYYY-MM-DD」，
**从 MANIFEST「最后更新」取、由构建脚本注入，不手写**——手写就是派生账，必漂。
注意「最后更新」是长散文，里面混着改动日与内容里提到的日期（如"2026-07-28 新版规范"），
**取"不晚于构建日的最大值"**：取第一个会拿到最早那次改动，取最大值会拿到未来的内容日期
（2026-07-20 实测：19 个模块曾全部显示成 2026-07-12，就是取了第一个）。

**为什么这不违反「真源唯一、派生只引用」**：新增日期是**历史不变量**，写下那一刻就永远不会变。
派生账危险的根源是"真源会变而副本不跟着变"（页数、章数、模块数都是这样）；不变量不会打架，
所以直接写在成品里即可，**不建账本、不建回刷机制**。

**将来若要"最近新增的题"聚合视图**：让构建脚本扫页面里的 `.added` 标记汇总即可，
不要为此新建一份题库账本——题目本来就只住在成品里，建账本等于凭空造一个会漂的派生源。

## 派生账登记表（v3.13 · 全库唯一出处，各回刷清单引用它）

库里有一批"同一事实写在多处"的**派生账**：一处是真源，其余全是派生。历史上最高发的失败类
就是"动了真源、漏刷派生"（实战包停 18 模块、封面停旧章数、四面页数账打架、_skill-source
停旧版……至少 6 起独立事故）。各任务分支的回刷清单（ppt-rules 增补回刷、illustration-rules
§八、prep-rules 口径回刷、patrol-rules 账实轴）都是下表的**任务视角切片**——账目全集以本表
为准；发现本表缺账先补表，再改分支清单（同一账目散写多处必漂移）。

| 派生账 | 真源（唯一） | 派生位置 | 哪些任务会动真源 | 核对手段 |
| --- | --- | --- | --- | --- |
| 讲义页数 | 放映序实测（`ppt/slides/slide[0-9]+.xml` 计数） | 模块 MANIFEST／模块 README／顶层 README／`_prep` 一页纸+全库总数 | 增章、配图、删页 | `check_page_ledger.py` |
| 章数口径 | 正文眉题「第 N 章」最大值 | 封面章节条/分组条、导览「X 章一条主线」、总收束行数、速查章编号 | 增章、并章 | audit 检查项 10/13 |
| 全库口径（模块数/层数/题数） | 顶层 README 知识地图 | `_prep` 三件套正文与 `_prep/MANIFEST.md` | A 类新增模块、新设层 | prep-rules 回刷清单（人工） |
| 电子书账 | 各条资料的正规渠道链接 | `电子书书单.md` 的「获取方式」列、**讲义正文里指向资料的说法**（「已入库…」「见 ebooks/…」「书单 #N」） | 书单增删、链接失效、**契约变更（撤馆藏/换源）** | `check_ebook_ledger.py`（轴 1–4 查书单、轴 5 扫讲义 XML 与库内 HTML） |
| 时效性事实 | 各模块 MANIFEST 时效表（含核实笔记） | `_prep` 实战包题目/一页纸看板、讲义数字弹药页/速查页 | B 类刷新、巡检回写 | `_prep` 引用映射（巡检） |
| 视觉令牌 | `_assets/kb.css` 的 `:root` | 各页面的 class 用法（**页面里不写颜色字面量**） | 改配色、新建页面 | 全库 grep 颜色字面量与 `<style>`／行内 `style=` |
| 样式契约 | `kb.css` 的类定义与页面 class 用法**互为契约** | 全部页面／生成器／site.js 的 class；每页 `kb.css?v=` 缓存戳与可见构建号 | 换肤、组件重构、改类名 | `check_css_classes.py`（孤儿类＋全站版本戳一致；运行期动态类记在脚本白名单里，新增时同步） |
| 内部链接 | 实际文件路径 | 全库 HTML 相对链接、MANIFEST 串联边 | 改名、挪位、删模块 | `check_html_links.py` |
| skill 版本 | SKILL.md frontmatter | 各模块 MANIFEST「产出 skill 版本」、CHANGELOG、.skill 包、安装目录 | 技能升级 | 打包后三处 diff/md5 + `check_skill_package.py`；**md5 一致只证明搬运没错，能不能装要真装一次** |

用法两条：**收尾时**，本次任务动了哪行真源，就把该行派生位置全部回刷（操作细节看对应分支
切片）；**巡检时**，账实类脚本逐行兜底，兜出不一致按分支切片修。

## 为什么 README 用 HTML

README（顶层索引和每个模块索引）**一律用 HTML 写，不用 Markdown**。原因：HTML 能承载更丰富的
排版——彩色状态标签、卡片、可点击的内部链接、内嵌小图/表格样式，比纯 Markdown 表格信息量更大、
更好看，也更适合当"门户页"长期维护。

**样式一律外链库级共享样式表 `_assets/kb.css`，禁止各页内联 `<style>`（2026-07-20 定版）。**
早先的规则写的是"保持轻量：内联 `<style>`，不依赖外部 CSS"——**那句话把"无外部依赖"和
"不共享文件"混为一谈了**。`file://` 下相对路径的样式表是能加载的，浏览器拦的只有 `fetch()`
与 ES module（Portable 铁律的准确含义见 `web/web-design-system.md` §一）。照那句话执行的结果：
**52152 字符的 CSS 散在 28 个页面里各自演化**，加上写死在 markup 里的行内颜色，一个库长出了
三套配色，用户一眼就看出来了。改法与代价：
- 新建任何页面都写 `<link rel="stylesheet" href="<相对路径>/_assets/kb.css">`，不写 `<style>`；
- **markup 里不许出现写死颜色的 `style=` 属性**——它比内联 `<style>` 更难发现，扫描时容易漏；
- 代价是页面不再"单文件自带样式"：单独拷走一个 HTML 会掉样式。这是有意的取舍——
  本库的分享单元是**整个库目录**，不是单页（Portable 铁律同样是按整库定义的）。

### 全库 HTML 视觉令牌（唯一出处，2026-07-21 换向定版）

**库内所有 HTML——根总纲、面总览、模块 README、`_prep/` 取用页、网页版——共用同一套令牌**，
定义在 `_assets/kb.css` 顶部的 `:root`。方向＝**清爽导航站 / 工作台**（参照用户给的
shentoushi.top 类工具导航）：白底卡片矩阵、高信息密度、蓝色点缀、系统黑体、卡片描边+浅投影。
**上一版「技术档案/编辑部」（宋体+墨金+纸感）已废弃**——用户实测不喜欢：知识库门户的读法是
"快速取用"，不是"慢读杂志"，方向要跟读法走。

| 角色 | 令牌 | 值 |
| --- | --- | --- |
| 页面底 / 卡片 | `--bg` `--card` | `#F4F6F9` `#FFFFFF` |
| 描边 | `--line` `--line-2` | `#E2E8F0` `#EEF2F7` |
| 文字 | `--ink-900/700/500/400` | `#0F172A` `#334155` `#5B6B7F` `#8494A7` |

**样式表引用必须带版本参数**（2026-07-21 入规）：所有页面写
`href=".../_assets/kb.css?v=YYYYMMDD[x]"`，**每次改 kb.css 同批把 28 页的版本号一起换**
（一条 sed 的事）。原因：Chrome 对 `file://` 的 CSS 有缓存，用户开着的旧标签页在样式更新后
仍吃旧 CSS——2026-07-21 字阶收档后用户回看三个旧标签页，反馈"没任何变化"，磁盘上明明已经
是新版。**改了样式但用户看不到，等于没改**；版本参数让每次变更强制生效，不依赖用户会不会硬刷新。
**并且顶栏右端放一个可见构建号**（`.ver`，如 `v0721c`，与版本参数同步换）——哪个标签页在
渲染旧样式，肉眼可判，不用再对着截图猜（2026-07-21 靠比对截图里"标题:顶栏"字号比例才确认
三个标签页跑着两套 CSS，排查了两轮；有构建号一眼的事）。排查用户端样式不一致的顺序（2026-07-21 二次修订）：
**①先查缩放**——地址栏有放大镜图标 = 该标签页缩放 ≠100%，让用户逐页 ⌘0（缩放差能同时
伪造字号差与位置差，CSS 怎么改都追不平）→ ②看构建号是否一致 → ③同一标签页内用顶栏
切换再比 → ④关掉重开标签页。

三条纪律不变：**只有 `_assets/kb.css` 能定义令牌**，页面里不写颜色字面量；**讲义 PPT 是另一套
识别色**（`ppt/ppt-design-system.md`，主强调青 `#128199`），两套互不串门；**不得出现第三套**
（规则挡不住结构，令牌收进一个文件才治本——2026-07-20 教训）。

**换向的操作教训（2026-07-21）**：整站换肤之所以一次收口，全靠此前把 26 个页面收进一份样式表
——换向＝重写一个文件+重跑生成器；若样式仍散在各页，这次换向就是 26 处手术。**方向可以换，
"单一样式表"这个结构不许回退。**另有一处连带修复：页面行内引用了从未存在的令牌
`var(--bg-2)`（首版 site.css 遗物，实际渲染为透明底）——**行内 style 引令牌同样是漂移源**，
一律改类。

## 分享与交接（2026-07-21 入规：portable 的最后一公里）

Portable 铁律定义了"库可分享"，但分享还差两件：**怎么打包**、**接收者拿到后怎么接着建**。

**打包 = `scripts/make_share.py`**（随库复制进 `_maintenance/`）。分享单元是整库目录，默认瘦身：
带讲义/书单/网页版/`_assets`/`_prep`/`_maintenance` 工具/KB-CONFIG/raw-data 输入与核实笔记
（事实溯源随库走）/**当前技能包 `.skill`**（接收者装它接着建自己的库）；不带 `.git`、
`raw-data/history/`（作者的回滚保险，实测占库 49M/76M）、`_skill-source/history/` 与解包源、
`_maintenance/history/`、`_reference/` 内容。两条实现要点：

- **`_reference/` 排除内容但保留空目录**——布局契约要求它存在（`check_kb_layout` 会验），
  接收者自己的参考资料以后放这。首跑就是被这条卡住的：瘦身包解压后布局检查判残缺。
- **打完必解压自检**（在解压出的库里跑布局+坏链门禁）——打包链的"真装一次"：zip 建成功
  只证明压缩没错，不证明拆开还是一个能用的库（技能包 v4.8 同款教训，这里直接做成了自动层）。

**交接 = 库根 `开始使用.html`**（成品，挂根总纲入口卡）。三段固定内容：每天怎么看（三个
常用入口）、整库怎么分享（分享单元=整个文件夹 + 打包命令）、**接收者三步把它变成自己的库**
（装 `.skill` → 改 KB-CONFIG → 直接对 Claude 说话），外加一张"库的约定"表（账本/门禁/留痕/
生成器各在哪）。接收者的库与原库各自演化，互不影响——这正是 v3.0 把个性外置进 KB-CONFIG
的设计初衷，交接页是它面向接收者的说明书。

## 技能包打包与验收（2026-07-20 入规）

分发用的 `.skill` 就是一个 zip，但**结构错了装不上**：

- **`SKILL.md` 必须在压缩包根层**，不能把整个技能目录多套一层。正确做法是**进到技能目录内**
  打包：`cd <技能目录> && zip -qr ../<名>.skill . -x '*.DS_Store'`；
  在父目录执行 `zip -r <名>.skill <技能目录>` 会多套一层，包就废了。
- `references/`、`scripts/` 同样在根层；不带 `.DS_Store` / `__MACOSX` 杂物。
- `description` ≤1024 字符（claude.ai 上传硬门槛，超限直接拒收）。

**验收分两层，缺一不可**（core-rules §三）：

1. `scripts/check_skill_package.py <包>` 查结构；
2. **真装一次**。这一层无法自动化，交用户抽验。

**教训（2026-07-20）**：此前每次升级都报告"已重打包、四处副本 md5 一致"，用户拿去装才发现
"这个包用不了"。**md5 一致只证明文件搬运没出错，完全不证明产物可用**——打包这条链上一直
缺最后一层防线，而报告的措辞让它听起来已经验过了。凡是"产出一个供别处消费的产物"的动作，
都要问一句：我验的是搬运，还是验的是能用？

## 命名约定

- 模块文件夹用主题的通用英文简称（`RAG`、`MCP`、`Agent`），全库大小写风格保持一致。
- 模块内文件名沿用模板里的名字（`README.html`、`MANIFEST.md`、`电子书书单.md`、`raw-data/`、
  `<模块名>-讲义.pptx`），新模块照抄，不要自创。**库内任何层级都不放电子书文件**——
  v4.3 起书单一律只列正规渠道链接。
- B 类更新时，**先看该模块现有命名和结构，照它来**；不要引入与其它模块不一致的新写法。

## A 类 vs B 类 的差别（一句话）

- **A 新增模块**：先在知识地图上落位，再从零建上面整套骨架（含 HTML README 与 MANIFEST.md），
  挂进顶层索引。
- **B 更新已有模块**：不建新骨架，进到已有模块里，严格沿用其结构与风格，只改用户指定的章节；
  改写任何成品前先把旧版存档到 `raw-data/history/`（见上）；改完更新该模块 `README.html`
  的章节状态与 `MANIFEST.md`（章节、核实日期、串联），以及顶层 `README.html` 里该模块的"更新日期"。
