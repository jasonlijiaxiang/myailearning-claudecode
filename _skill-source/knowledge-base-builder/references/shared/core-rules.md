# 核心原则（共享层：横切一切任务与两面）

本文件只收**在多份规则文件里反复出现的横切原则**——每条在此定义一次，其它文件只做
上下文应用与指针（DRY：一条规则只写一处）。SKILL.md 的工作流与风格底线不在此复写。
收录门槛：确实有 ≥2 处独立出现的原则才进来，防止本文件变成空谈。

## 一、真源唯一，派生只引用

任何事实、账目、规则只登记一处"真源"，其它出现处都是引用。原因：同一信息写两处，
刷新时必漏一处，之后两处口径打架、读者不知道信哪个——派生漂移是本技能历史上最高发的
失败类（实战包停 18 模块、封面停旧章数、四面页数账漂移、_skill-source 停旧版，≥6 起
独立事故）。落点：

- 内容事实与章节结构的真源 = 模块 `MANIFEST.md`（PPT/Web 两面共同账本，网页注册表由它生成）；
- 全库派生账的登记 = `library-rules.md`「派生账登记表」（账目→真源→派生位置→触发任务→核对手段）；
- 库级产物不自创事实 = `prep-rules.md` 引用映射纪律；
- 规则本身同理：本技能各文件间只指针不复写，第 6 步自省固定查一遍重复。

## 二、内容决定形式，不设配额

版式、数量、节奏由内容决定，不由模板、配额或对称性决定；为配平而增删内容 = 削足适履。
应用面（各处写的是同一条原则的上下文应用）：

- PPT 版式与每章页数：`ppt-rules.md`「内容决定版式」「反模板化硬指标」；
- 配图张数：`illustration-rules.md` §二（**取证出处**——用户两次纠正"齐刷刷 4 张/5 张
  都是变相配额"）；
- 对练/问答题数：`ppt-rules.md` 对练页、`web-rules.md` 问答立题判据（相近合并、不逐章配题）；
- 网页布局：`web-design-system.md`「任意数量布局」（0/1/单双数都不塌，不为对称删/复制/捏造）。

## 三、分层验收（防线模型）

历次"门禁全绿仍出事"的事故结构完全相同：**上一层防线全绿、下一层拒收**。所以触碰成品的
收尾按层验收——每层只能抓它那类病、**互不替代**，跳层 = 给对应事故留门。两条使用纪律：

1. 收尾按层依次跑，最后一层（真实用户环境亲测：双击打开 / 冷浏览器直达）无法自动化，交用户抽验；
2. 出新事故先问"**是哪一层防线缺失或没跑**"，再决定是给某层加检查项、还是补一层新手段——
   不要条件反射地"加一个检查项"了事：历史上第 1 层（结构门禁）原理上抓不住的病类约占一半。

各面的具体层与手段：PPT 面五层见 `ppt-design-system.md` §10 表；Web 面六层见
`web-design-system.md` §四。

## 四、工具中立

引擎随环境换，**规范和验收标准永远不换**——按"能力→环境映射"选工具（PPT 生成、网页
构建、渲染、检查皆同；无现成工具就直接写脚本）。构建工具链不受限，受门禁约束的是**产物**
（Web 面的 Portable 铁律即此原则的产物侧表述）。PPT 面的能力→工具映射表见
`ppt-design-system.md` §9。

## 五、错误转化为系统能力

发现系统性问题不许只修当前实例，五件套同做：①修复当前实例；②更新共享规则或组件；
③可机器判断的部分加回归检查；④无法自动判断的部分加入人工验收清单；⑤复核其他已发布
成品有无同类问题。**禁止用单点特判掩盖系统问题**（各面的特判形态见各自 design-system）。
成品/页面层问题按本条走，规则层问题走 SKILL.md 第 6 步技能自省——同一进化机制的两层。
